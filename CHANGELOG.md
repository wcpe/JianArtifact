## 未发布版本

### 新增
- 离线 blob store 多线程并发枚举（FR-133，增强 FR-37）：上万 blob 的离线 Nexus blob store 枚举由单线程串行遍历改为**多线程并发**——`content/` 下各子目录（vol/chap 分片）作独立并行单元，用 `std::thread::scope` + `mpsc` 把「逐目录读 + 解析 .properties」分发到 `min(目录数, available_parallelism())` 个工作线程（获取失败回退 4、单核 / 单目录退化为串行、不无界开线程），主线程锁外汇总。`enumerate_blob_store` 与 `enumerate_blob_entries` 内部均改走并发解析，**公开签名 / 返回类型不变**；结果经 `BTreeMap` 归并 + 按 `(repo, blob_name)` 排序保证**与单线程串行等价**（采集顺序无关、去重不变），容错（损坏 / 软删 / 缺字段跳过）语义不变。**纯 std、无新依赖**。单测加「并发结果集 == 串行预期」等价断言 + 退化目录（空 content / 无 chap）不 panic。**大库真机加速属真机维度、待真机验收**
- 发布产物压缩包 + 自更新解压（FR-138，增强 FR-86/89）：CI 改为每目标同时上传裸可执行（过渡保留）与压缩包（`jianartifact-{version}-{target}.zip`，三平台统一 zip，规避静态 musl ELF 杀软误报），各附 sha256 / md5 校验和文件；`update` 模块 apply 优先选压缩包（zip + .sha256 均存在时下载 zip → 校验 sha256 → 解压取出二进制 → 原子替换），资产列表无压缩包时回落裸 exe（向后兼容旧版发布）；sha256 安全门不变，利用已有 zip 依赖、无新依赖。**CI 打包产物与真机更新一次属待 CI / 待真机维度**
- 在线更新异步化 + 进度页 + 日志（FR-126，增强 FR-85/87，复用 FR-83 异步 job 基建）：修复在线更新（FR-85）同步流程在反向代理后的 **504 / 卡 95% / 不跳转 / 检查结果丢失**。① **检查异步 + 留存**——`POST /api/v1/update/check` 起后台联网检查 job（立即返回 `202 + job_id`），结果**留存**到数据目录状态文件；`GET /api/v1/update/check` 改为**只读留存、不联网**（页面打开即见上次结果，不必每次重新检查、不触发 GitHub 限流）。② **应用 / 回滚异步**——`POST /api/v1/update/apply`、`/update/rollback` 立即返回 `202 + job_id`，下载 / 校验 / 替换在后台执行；保留 FR-85 单飞互斥（409「更新进行中」）+ 安全门（仅 sha256 校验通过才原子替换、出站默认关闭、token 不入日志）。apply **不支持取消**（半截替换会坏二进制）。③ **进度按阶段 + 重启后可续看**——新增 `GET /api/v1/update/jobs/{id}`（单任务进度，阶段：检查 / 下载 / 校验 / 替换 / 重启，**不做假百分比**，根治旧「卡 95%」客户端假进度）与 `GET /api/v1/update/jobs`（重启后回填上次 apply 终态供续看）；启动早期 `backfill_last_apply` 从数据目录状态文件回填。终态留存到 `{data_dir}/update-state.json` 是对 ADR-0019「迁移 job 不落库、重启即丢」的**有意例外**（更新替换二进制后无法靠重跑恢复、必须留存供查看），token / 凭据绝不写入该文件。④ **全过程写 FR-107 中文分级日志**（后台可 tail 看更新进度）。前端系统页把「客户端模拟进度卡 95%」替换为轮询真实 job 进度 + 触发后进度视图 + 留存检查结果展示。后端单测（check 异步 + 留存、GET/check 只读、apply 返 202、终态写文件且重启后回填可读、阶段进度、日志）+ 前端 vitest 覆盖。**反代后触发不 504 / 真重启续看 / 真下载替换属真机维度、待真机验收**
- 出站代理连通性测试（FR-128，增强 FR-84/87，仅 Admin）：设置页代理节新增「测试」按钮。填入目标 URL（仅支持 http/https），点测试后后端经当前生效出站代理对该 URL 发 GET（10s 超时），返回连通性结果（`ok/status/elapsed_ms/error`）。前端展示成功状态码 + 耗时（绿色）或失败原因（红色）。非法 URL / 非 http/https 返回 400；仅 Admin 可调用；仅访问用户给定 URL，不外发使用数据（符合 ADR-0009）。**回流修（UX 改造 + MSW handler 缺口修复）**：① MSW mock handlers 补 `POST /settings/proxy-test` 处理器，修复 mock 模式下点击测试无反应（请求缺 handler 静默失败）；② UX 改为三代理各一个独立「测试」按钮，点击弹出 Mantine `Modal`，模态框内预填当前代理 URL、可修改后点「测试」、展示状态码 / 耗时 / 失败原因。**真机经代理测试属真机维度、待真机验收**
- 离线 Nexus proxy/hosted 搬运异步化（FR-125，增强 FR-38/39，复用 FR-83/91 异步 job 基建）：继 FR-124 预览异步化后，把离线搬运也异步化——`POST /migrate/nexus/proxy/migrate` 与 `/migrate/nexus/hosted/migrate` 由**同步搬运返回完整报告**改为**异步任务**：同步阶段只做在线 REST 枚举源仓库配置（失败即 400/502、不开任务），随后**立即返回 `202 + job_id`**，实际搬运（建仓 + 从离线 blob store 流式 ingest）在后台任务执行，进度经 `GET /migrate/jobs/{id}` 轮询（`phase` / `total_assets` / `done_assets` / `migrated` / `skipped` / `current_repo` / `current_path` / `repos` / `skipped_repos`）。**复用 FR-91 取消 / 暂停**：每个 blob 与每个仓库边界响应控制信号——取消即停止后续搬运（已搬运保留、不算失败、标 `cancelled`），暂停即在锁外挂起等待继续。`proxy.rs` / `hosted.rs` 抽出 `migrate_*_repositories_with_progress` 变体（无进度的便捷入口委托之、签名不变），复用 `online` 的 `await_control` / `mark_cancelled` 边界逻辑、不引入新依赖；「blob 先落盘并校验 sha256 再写索引、失败回滚不留孤儿、单制品失败 / 不可覆盖不中断整批、幂等可重入」等不变量不变。前端迁移页离线搬运改为「发起任务 → 轮询进度 → 终态展示报告」，与在线拉取共用同一进度面板与重连存档。单元测试（proxy/hosted 各加进度上报 + 预先取消标终态）+ 前端 vitest（离线 proxy 搬运异步轮询至 done 渲染报告）覆盖。**真实大库经反代不再 504、可取消属真机维度、待真机验收**
- 离线 Nexus blob store 预览异步化（FR-124，增强 FR-37，复用 FR-83 异步 job 基建）：修复「枚举大型离线 blob store（上万 blob / 数 GB+）经反向代理（如 1Panel/OpenResty）访问时同步遍历超时返回 **504**（后端无错误日志，因仍在跑）」。预览端点 `POST /migrate/nexus/offline/preview` 由**同步遍历返回完整列表**改为**异步任务**：立即返回 `202 + job_id`，后台 `spawn_blocking` 执行 `enumerate_blob_store`，枚举结果经 `GET /migrate/jobs/{id}` 进度的新增字段 `offline_preview` 轮询取回（复用在线拉取 FR-82/83 同一 job 注册表与轮询端点，进程内不落库）；路径为空 / 不存在等可同步判定的错误仍即时返回 400（廉价 `is_dir` 预校验，不起任务）。前端迁移页离线预览相应改为「发起任务 → 轮询至终态 → 渲染 `offline_preview`」，超时 / 失败有中文提示。后端集成测试（提交得 job_id → 轮询 done 拿按 repo 枚举结果；路径不存在 400）+ 前端 vitest（异步轮询渲染）覆盖。**真实大库经反代不再 504 属真机维度、待真机验收**；离线 proxy/hosted 搬运异步化见后续 FR-125
- Web 上传页 Maven 适配：坐标自动识别 + 可选 pom + 快照提示 + 多文件（FR-123，增强 FR-74）：让通用上传页（FR-74）产出合规 Maven 制品，复用已落地的 FR-120/121/122 后端能力、无新依赖。① **坐标自动回填**——Maven 上传 `groupId`/`artifactId`/`version` 可留空，服务端从 jar 内嵌 pom（FR-120）自动识别（表单为主、缺项 jar 补齐、都无则 400 提示）；后端上传路由由 `resolve_maven_coords`（表单可选 + jar 兜底）统一解析坐标，`build_upload_path` 取代旧 `resolve_upload_path`/`maybe_mint_snapshot_path`（含快照时间戳铸造 FR-122）。② **可选 pom（用户上传层）**——表单可附带 `pom` 文件，落主构件旁同基名 `.pom` + 四校验和 sidecar，作为 FR-121 pom 三级兜底的「用户上传」层（先于服务端兜底写入，client-priority 不被覆盖）。③ **快照提示**——前端版本含 `-SNAPSHOT` 时提示服务端将生成时间戳唯一版本。④ **多文件**——单次上传 jar（主 `file`）+ 可选 `pom`。前端上传页 Maven 坐标输入改为可选（恒可提交、附自动识别提示）+ 新增可选 pom `FileInput` + 快照提示；文案进 zh-CN/upload.ts。后端集成测试（坐标留空自动识别 / 无坐标无内嵌 pom 400 / 用户 pom 不被覆盖）+ 前端 vitest（坐标留空可上传且 FormData 省略坐标 / 快照提示显示）覆盖。**真 mvn 解析与上传页真机交互属真机维度、待真机验收**
- 完整 Maven 快照规则：时间戳唯一版本 + 快照级 maven-metadata.xml（FR-122，ADR-0037 快照级扩展，增强 FR-14/61）：在 FR-121 服务端权威元数据之上补齐 SNAPSHOT 完整规则。① **唯一时间戳版本**——SNAPSHOT 主构件经 **Web 上传**时服务端按 `{a}-{base}-{yyyyMMdd.HHmmss}-{buildNumber}.{ext}` 铸造唯一时间戳版本落库（版本目录仍为 `{base}-SNAPSHOT`），时间戳取真实 now（`SystemTime` + 自写 Hinnant `civil_from_days` 历法换算，**无新依赖**）、构建号据目录现有最大值 +1。② **快照级 maven-metadata.xml**——写入快照时间戳构件后按目录内时间戳构建聚合生成 `{base}-SNAPSHOT/maven-metadata.xml`（`snapshot/timestamp/buildNumber`、`snapshotVersions/snapshotVersion(extension/value/updated)`、`lastUpdated`）+ 四校验和 sidecar，供 `mvn` 解析「最新快照」。③ **client-priority**——`mvn deploy` 自带的时间戳构件不被改写（不兜底 pom），服务端据目录扫描权威重生成快照级 metadata（与客户端计算等价/超集）；release 不可覆盖、snapshot 时间戳唯一可拉最新（FR-61 覆盖语义不变）。pom 兜底（FR-121）对快照落同时间戳唯一名（pom `<version>` 仍为 `{base}-SNAPSHOT`）；artifact 级 metadata 把 `{base}-SNAPSHOT` 列为一个版本。纯协议逻辑（路径 / 解析 / 聚合 / XML / 历法）下沉 `MavenFormat`，铸造编排在 `api/maven_publish.rs`。单元 + 集成测试覆盖（快照判定 / 时间戳构件解析 / epoch→时间戳向量 / 构建聚合与构建号递增 / 快照 metadata 结构 / Web 传 SNAPSHOT 生成时间戳版本+metadata / 连传构建号递增 / mvn deploy 时间戳构件服务端重生成）。**真 mvn 快照互通（`mvn deploy` SNAPSHOT、`mvn dependency:get` 拉最新快照）属真机维度、待真机验收**
- Maven 服务端权威 maven-metadata.xml + pom 三级兜底（FR-121，ADR-0037，增强 FR-14/61）：服务端成为 Maven 元数据权威维护者，**取代** p1-maven「服务端不重写 metadata、客户端自管」旧决策。① **artifact 级 `maven-metadata.xml` 写时权威生成**——任一 Maven 主版本文件（jar/war/pom 等，能反解 GAV、非 sidecar、非 metadata）写入 hosted 后，按 SQLite 索引（唯一真源）聚合该 `{groupId}/{artifactId}` 全部版本，重生成 `maven-metadata.xml`（`versions` / `latest` / `release` / `lastUpdated`）+ 四校验和 sidecar；`lastUpdated` 取该 artifact 版本文件 `created_at` 最大值去非数字截 14 位为 `yyyyMMddHHmmss`，**不新增时间库依赖**；`latest` 取部署序末位、`release` 取末位非 SNAPSHOT 版本（无则省略）。② **pom 三级兜底**——保证主构件 GAV 下存在 `{a}-{v}.pom`：客户端已上传则不覆盖（client-priority）→ jar 内嵌 `META-INF/maven/.../pom.xml` 原样提取（复用 FR-120）→ 按 GAV 生成最小 pom（`modelVersion=4.0.0` + GAV + 据扩展名推断 `packaging`），附四校验和 sidecar。③ **client-priority**——`mvn deploy` 自带 pom 不被改写（deploy 路径不兜底 pom、仅维护聚合 metadata）；Web 上传缺 pom 时服务端兜底，使网页上传的 release 也具备完整 pom + metadata 供 `mvn dependency:get` 解析。生成逻辑为纯函数（路径 / 聚合 / XML 生成）下沉 `MavenFormat`，写时编排在 `api/maven_publish.rs`（接 mvn deploy PUT 与 Web 上传两条路径），复用既有 `put_hosted`（blob 先落盘校验再写索引）+ sidecar 机理、锁外执行。单元 + 集成测试覆盖（版本聚合排序去重 / metadata latest-release-lastUpdated / 内嵌 pom 提取 / 最小 pom / 多版本聚合 / Web 上传生成 pom+metadata / client pom 不被覆盖）。**真 mvn 互通（`mvn deploy` / `mvn dependency:get`）属真机维度、待真机验收**

- 动态配置入库与覆盖（FR-106，ADR-0028，扩展 ADR-0022）：除启动必需项与凭据外的**非密钥**配置可经面板在线编辑、持久化到 SQLite 并重启不丢，按「黄金组合」生效——文件兜底 + DB 覆盖 + 内存槽缓存，优先级 **env 显式 > DB > 文件默认**。新增迁移 `0012_app_settings.sql`（KV 表 `app_settings(key, value_json, updated_at)`）与 `meta` 三方法（`load_settings` / `upsert_setting` / `delete_setting`，DB 只经 meta）。新增**装配层**覆盖模块 `config_overlay`（`config` 模块零 DB 依赖、依赖方向不变）：启动序列在 meta 就绪后读 `app_settings` 作 DB 覆盖，经**纯函数** `merge_effective_config` 合并出生效配置再填充各热替换槽 / 后台任务参数；合并 / 解析失败只 WARN + 回落文件默认，不阻断启动。**白名单 + 默认拒绝**（守凭据红线）：仅 `limits` / `protection` / `observability.{audit,usage,metrics,metrics_timeseries}` / `vuln` / `auth` 三个可调标量 / `update` 非密钥字段可入库；代理账密 / update token / OIDC·LDAP 密钥 / JWT 密钥与 bootstrap 项（`server.*` / `data.*`）**永不入库**（`auth` / `update` 经专用非密钥视图序列化，结构上不可能带出凭据）。`PATCH /api/v1/protection/config`（FR-79）与 `PATCH /api/v1/settings`（FR-88，仅 update 非密钥字段）的改动同时落库 `app_settings`、重启仍生效；代理凭据与 token 继续只入内存槽不落库。env 显式给值的节仍以 env 为准、不被 DB 覆盖。新 Dynamic 节（limits / observability / vuln / auth 非密钥项）的在线编辑表单已落地：新增**仅 Admin** 端点 `GET` / `PATCH /api/v1/settings/dynamic`（`src/api/dynamic_config.rs`）——GET 以启动期生效配置为基线叠加当前 DB 覆盖回显「当前 + 待生效」值，PATCH 整体校验各节边界（采样间隔 / 刷新周期 / 下载超时 / 会话 TTL / 锁定时长须 > 0）后按节落库 `app_settings`；`auth` 经 `AuthTunables` 非密钥视图序列化（OIDC / LDAP 密钥绝不入库），端点只写固定白名单键、bootstrap 项不经此路径。这些节多在启动期装载、无热替换槽，改动**保存后重启生效**（黄金组合「变更=改 DB、下次装载生效」，不为每个后台任务强造热替换槽）。控制台「设置」页二级导航新增「系统配置」tab，分组（限制与配额 / 可观测性 / 漏洞库 / 安全 · 会话）编辑上述非密钥项，独立保存按钮并显著标注「保存后重启生效」（区别于代理 / 更新 / 防护的即时生效）。真机「面板改 → 重启 → 仍生效」端到端持久化待真机验证
- 主机 / 系统监控采集（FR-98，ADR-0023）：新增**仅 Admin** 端点 `GET /api/v1/monitor/host`，经新增依赖 `sysinfo`（裁 features 仅 `system` + `disk`）跨平台采集**这台主机**的基础资源画像——CPU 全局使用率 + 逻辑核数、内存 / 交换的已用 / 总量、磁盘逐盘挂载点 / 总量 / 可用及汇总、系统 uptime。**按请求单次采样**（共享 `sysinfo::System` 经 `Mutex` 串行刷新、磁盘按请求刷新），不后台轮询、不落库、不留历史时序；纯本机内部数据、**绝不外发、不向外部遥测 phone-home**（守 ADR-0009 / 0015 基调），GET 读取类不入审计。新增 `monitor` 模块承载采集（「sysinfo 读数 → DTO」纯映射抽为无副作用纯函数）；`cpu.usage_percent` 首样可能为 0（CPU 使用率需两次采样间隔）属已知取舍
- 设置可编辑与运行时热替换（FR-88，ADR-0022，取代 ADR-0020 的「代理只读 / 运行时不热替换」取向）：网络代理 `[network.proxy]` 与在线更新可调字段（`enabled` / `repo` / `api_base_url` / `restart_mode` / `token`）改为经控制台「设置」页或 `PATCH /api/v1/settings`（仅 Admin）在线编辑、**即时生效、无须重启**。新增随 `AppState` 共享的出站网络热替换槽 `config::NetworkState`（std `RwLock<Arc<NetworkSnapshot>>`，快照含代理配置 + 据其构造的 `reqwest::Client`）：5 处出站点（proxy 回源 / Nexus 迁移 / 漏洞库镜像 / OIDC / 在线更新）不再持启动期 client，改持 `Arc<NetworkState>`、每次出站取当前 client；PATCH 改代理后锁外重建 client、原子换槽，下个出站请求即用新代理。设置页从只读改可编辑（代理 http/https/no_proxy 表单 + 在线更新 enabled/repo/api_base_url/restart_mode/token），保存调 `PATCH`。校验失败 400 且不改现有生效值；代理凭据与 token 只入内存槽、**不写回 TOML / 不入 DB / 不进日志 / 不回显**，重启回落文件 + env 配置
- 审计日志全面补齐（FR-97，增强 FR-31，扩展 ADR-0015）：审计中间件覆盖从"精选事件"扩为"**全量非读**"——**所有变更类请求（POST/PUT/PATCH/DELETE）一律产一条审计**，已知端点归精确语义 `action`（用户 / Token / 仓库 / ACL / 组管理、制品上传 / 删除，及新覆盖的设置 PATCH `settings.update`、防护配置 PATCH `protection.config.update`、迁移控制 `migrate.job.control` / 迁移搬运 `migrate.run`、在线更新 `update.apply`、登出 / 刷新 `auth.logout` / `auth.refresh`、组 / 组成员 / 组 ACL 增删等），未显式归类的非读路径走兜底 `change.{method}`（新增端点不漏记）；**读取类（GET/HEAD，含下载 / 浏览 / 搜索 / 详情）一律不入审计**（交使用分析计数，避免刷屏与性能损耗）；登录仍由登录 handler 显式发 `login`。保留既有异步有界 channel 投递 + 批量落库 + 保留期轮转；主路径只做一次非阻塞 enqueue，采集 / 写入失败仅 WARN、不影响业务。密码 / Token / JWT / 上游 / 代理凭据绝不进审计（`actor` 只记用户名，中间件不读请求体与 `Authorization` 值本体）
- Nexus 迁移任务控制（FR-91，增强 FR-83）：在线拉取异步任务新增取消 + 暂停 / 继续——新增 `POST /api/v1/migrate/jobs/{id}/cancel`、`/pause`、`/resume`（均仅 Admin）；后台逐 asset 循环在 asset 边界响应信号：取消即停止后续搬运、任务标 `cancelled`（不算失败、已搬运保留），暂停即挂起不推进、继续即恢复；进度快照新增 `paused` 布尔、阶段枚举新增 `paused` / `cancelled`；对已结束任务的控制为幂等空操作（200），未知 id 返 404。用进程内 `AtomicBool` + `tokio::sync::Notify` 实现（不引入新依赖）。Web 迁移页进度面板新增「取消 / 暂停 / 继续」按钮，按任务态启停

- 交互前台自更新后进程脱离终端、看不到日志（增强 FR-85，ADR-0032）：`restart_mode=self` 原以 `Command::spawn()` 把新版本拉起为**子进程**、旧进程随即 `exit(0)`——新进程变孤儿（reparent 到 init）、脱离 shell / tmux 作业控制：提示符回来、`Ctrl-C` 管不到、日志虽仍写 tty 但与提示符混杂，在 tmux 前台运行时表现为「自更新后进程不知去向、看不到日志」。修复：**Unix 改用 `exec` 原地替换进程映像**（同 PID / 终端 / 前台作业 / 会话，进程不脱离、日志连续可见；端口已在优雅停机时释放、新映像启动再绑），**Windows 无 `exec` 语义保持 `spawn`+`exit`** 不变。端到端「exec 后进程留在 tmux 前台、日志连续」依赖真机验证
- 自更新备份文件 compound 堆积（增强 FR-85/104，ADR-0032）：备份 / 暂存名由 `sibling_with_suffix` 把后缀追加到 `current_exe` 文件名上，当用户（多因上面的进程丢失而手动跑备份文件去恢复）运行的二进制名本身已带 `.bak` / `.rollback.bak` 后缀、再触发更新 → 叠成 `.bak.bak` / `.bak.rollback.bak`，目录里堆出一串备份。修复：派生 `.new` / `.bak` / `.old` / `.rollback.bak` 前先**剥离 exe 名末尾已叠的更新管理后缀**（`strip_managed_suffixes`，正常运行恒等、仅异常路径收敛）+ 在 `.bak` / `.rollback.bak` 复制处**守自拷贝**（剥离后规范名可能恰等于当前 exe，跳过避免毁源）；启动清理增删「规范名 + ≥2 层管理后缀」的 **compound 残留**（保留单层 `.bak`〔事务兜底〕与 `.rollback.bak`〔持久回滚源〕两份不动）。另澄清：原地替换**保留原文件名**（更新后文件名仍为旧版串如 `…-dev.8-…`）为**设计行为**——保路径稳定给 service manager / 脚本 / `.rollback.bak` 派生引用，`--version` 自报真实版本，不重命名
- 自更新回滚跨迁移把数据库锁死（增强 FR-104，ADR-0031）：回滚只还原二进制、不还原 DB schema——回滚到旧二进制（如内嵌迁移只到 `0010` 的 dev.8）后，DB 里已被新版应用的 `0011_metric_samples`/`0012_app_settings` 不在旧二进制内嵌集里，`sqlx::migrate!().run()` 默认报 `migration 11 was previously applied but is missing in the resolved migrations` → DB 打不开、回滚后起不来。修复：`MetaStore::open` 跑迁移设 `set_ignore_missing(true)`——旧二进制忽略 DB 里它不认识的更高已应用迁移、只跑自身待应用项；并定「迁移只增不改不删（向前兼容）」硬约束（ADR-0031），破坏性变更须先走 ADR。**遗留**：已发布的旧二进制无本修复，恢复办法为跑任一含相应迁移或带本修复的二进制
- 网页配置的网络代理重启即丢（增强 FR-84/106，ADR-0030）：代理（`[network.proxy]`）经 `PATCH /api/v1/settings` 配置后原本只入进程内热替换槽、不落库，FR-106 又把代理整体作「含凭据项」排除在动态配置入库之外——导致网页配的代理一旦重启（尤其在线更新自更新会重启进程）即全部丢失、出站直接断连。改为**加密落库持久化**：代理落 `app_settings` 单键 `network.proxy`，每代理拆为 `{url（脱敏 host，明文）, username（标识，明文）, password_enc（密文）}` + `no_proxy`（明文）——**密码只以密文落库**（XChaCha20-Poly1305 AEAD，192 位随机 nonce，密钥经 `JwtSigner::derive_key` 域分隔派生、真源 `.jwt_secret`，明文密码与密钥绝不入库 / 进日志，守凭据红线）；URL 与用户名属非密钥明文落库（沿用 ADR-0024）。启动序列在 JWT 就绪后、装配可编辑设置槽前从 `app_settings` 读回该键、解密重建代理覆盖文件默认，代理（含密码）重启不丢；解密失败（如 `.jwt_secret` 轮换）该代理降级为无密码 + WARN，不阻断启动。新增纯 Rust 依赖 `chacha20poly1305`（RustCrypto 生态，零原生 C 依赖）。`GET` / `PATCH /api/v1/settings` 对外契约不变（仅后端多了持久化）
- 控制台外壳页眉横跨顶部占据侧边栏顶部（增强 FR-92）：`AppShell` 改 `layout="alt"`——侧边栏（navbar）占满整列高度、从最顶端起（logo 置于侧栏最上），页眉（header）只压在右侧内容区上方、不再横跨顶部，对齐参考版式
- 相对时间显示偏一个时区（增强 FR-108/77）：后端时间戳为裸 UTC 串（`"2026-06-27 16:10:29"`，无时区标识），前端 `formatRelativeTime` 直接 `Date.parse` 会按本地时区解析——在 UTC+8 等环境下「刚刚」被显示成「8 小时前」（仪表盘近期活动等处可见）。新增 `parseUtcMs` 统一按 UTC 解析（裸串补 `T`/`Z`、已带时区标识则原样），`formatRelativeTime` 改用之；纯前端、不改后端时间戳契约
- 在线更新版本号永不收敛（增强 FR-85/86/89）：`current_version` 原恒取 `CARGO_PKG_VERSION`（基线 `0.4.0`），而 prerelease 的 `dev.N.sha` 只存在于 Release 名 / 资产名、从未编进二进制——导致 prerelease 通道自更新后「当前版本」纹丝不动（恒 `0.4.0`），与 `latest=0.4.0-dev.N.sha` 永远不等而一直显示「有可用更新」、永不收敛。新增 `src/version.rs` 的 `build_version()`：CI 发布时经环境变量 `JIANARTIFACT_BUILD_VERSION` 注入完整版本串（prerelease=`{cargo}-dev.{run}.{sha}`、tag=版本），编译期由 `option_env!` 读入、优先取用，回退 `CARGO_PKG_VERSION`；全部「当前版本」展示点（在线更新检查 / 应用、`/health`、设置页、clap `--version`）统一改经它取值。`release.yml` 在 release 构建步骤注入该 env（`ci.yml` 测试任务不注入，本地 / 测试仍为基线版本，行为不变）。修复后 CI 出的 dev 二进制自报 `0.4.0-dev.N.sha`，自更新到同版即 `当前==最新`→无更新（收敛）
- 控制台设置页「保存」按钮漂移（FR-96）：保存动作条改为 sticky 底部固定条（`position: sticky; bottom: 0`），始终贴在滚动视口底部、不随内容 / 窗口缩放漂移、滚动时常驻可见；配顶部描边 + 背景 + 内边距与正文分隔、不遮挡内容。仅定位呈现，沿用 FR-88/89 既有保存 / 检查 / 应用逻辑与 `GET` / `PATCH /api/v1/settings` 契约
- 控制台侧栏导航高亮串台：导航 active 判定由前缀匹配改为按路径段精确匹配，修复进入「防护监控」（`/protection-monitor`）时「防护配置」（`/protection`，前者前缀）被一并高亮的问题；其它有前缀关系的路由同样不再串台，仓库等子路径仍正确高亮
- 发布流水线 dev 预发布资产堆积（FR-86）：滚动 `dev` 开发版快照的资产名内嵌 `{run_number}+{sha}`、各次不同，原先 `action-gh-release` 只追加不删旧资产致其在 `dev` 预发布里无限堆积。发布前显式删除旧 `dev` 预发布及其 tag（仅 prerelease 渠道，正式 `v*` 不删），使每次推 master 的 `dev` 预发布只含当前快照资产
- 在线更新 prerelease 通道拉不到预发布版（FR-89，增强 FR-85）：修复开启在线更新 + `channel=prerelease` 时拉不到 dev 预发布的两处缺陷——① 发布流水线 dev 版本串由 `{cargo版本}-dev.{run_number}+{shortsha}`（含 `+`）改为 `.{shortsha}`（点分），避免 GitHub 上传 Release 资产时把资产名里的 `+` 改写成 `.`、致自更新按含 `+` 的版本重构的期望资产名匹配不到；② prerelease 通道的版本判定由 `major.minor.patch` 三段比改为按完整版本串判定（目标与当前不同即视为可更新 / 可切换），修复 `0.4.0` 当前版对 `0.4.0-dev.N.<sha>`（核心版本相等）被误判「无更新」。stable 通道仍维持 SemVer 严格更高语义不变。另修 prerelease 滚动发布的 `tag_name` 为固定标签 `dev`（非版本串）时版本解析失败的问题——版本回退取 release 标题 `name`（内嵌完整 dev 版本串）
- Linux 发布二进制 glibc 版本锁致跨发行版跑不起来（FR-86，增强 FR-85）：Linux 目标由 `x86_64-unknown-linux-gnu`（动态链接构建机 glibc，CI 在 ubuntu-latest=glibc 2.39 上编、旧发行版报 `GLIBC_2.39 not found`）改为 `x86_64-unknown-linux-musl`（**musl 静态链接、不依赖 glibc**，老新发行版皆可跑）；CI 仅 Linux musl 目标装 `musl-tools` 编 C 依赖（bundled sqlite / ring）。自更新 target 推导同步改 musl，资产名随之为 `…-x86_64-unknown-linux-musl`（`unknown` 是 Rust target triple 的厂商字段、属正常命名，非异常）
- 监控页字节型指标渲染原始浮点（增强 FR-99/105/108）：监控页 KPI「存储用量」显示 `121.33333333333333 B` 这类未取整的原始浮点字节——根因在前端 `formatBytes` 的「小于 1KB」分支直出 `${bytes} B` 未取整，而字节型 gauge（去重存储字节）经 FR-105 降采样按桶平均后变为带小数浮点。修为该分支对字节取整（`Math.round`）；≥1KB 分支本就保留两位小数不受影响。纯展示层修复，KPI 卡 / 时序卡当前值 / 折线 hover 浮层均经此函数、一并修正；后端采样仍存整数字节、未改
