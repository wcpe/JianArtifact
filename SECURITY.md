# 安全说明：JianArtifact

> 信任模型、敏感数据处理、外部输入校验与漏洞报告渠道。安全模型变化时更新。

## 1. 信任模型

JianArtifact 面向多用户制品库场景，按身份划分信任级别，并以仓库可见性与每仓库 ACL 作为信任边界。

身份与信任级别：

- 匿名访客（Anonymous）：未认证，最低信任。仅能读取 public 仓库；对 private 仓库的任何访问一律拒绝（返回 401/404，不泄露存在性）。
- 注册用户（User）：经认证后，按全局角色与每仓库读/写 ACL 访问；可自助管理自己的 API Token。其权限范围由授权名单约束，不可越权访问未授权仓库。
- 管理员（Admin）：最高信任，可管理用户、仓库、ACL 与全局配置。
- CLI/包管理器客户端（系统角色）：mvn / npm / docker / curl 等通过 Bearer Token 或 Basic Auth 鉴权后，以对应注册用户身份访问格式端点。

信任边界：

- 仓库可见性边界（public / private）：private 仓库与 public 仓库严格隔离；私有仓库对未授权（含匿名）一律拒绝，这是核心安全不变量。
- 权限判定边界：由鉴权中间件按「目标仓库 + 操作（读/写）」综合判定 public/private、全局角色、每仓库 ACL 三者，任一不满足即拒绝；写操作额外校验对应仓库写权限。
- 数据真源边界：SQLite 是元数据唯一真源；上游凭据等敏感真值在配置/env，DB 仅存引用，不存明文。

攻击面概览：

- 管理 API（`/api/v1/*`）：认证、用户、仓库、ACL、Token 等 REST + JSON 端点，是越权与认证绕过的主要面。
- 格式 API（Maven / npm / Docker registry v2 / Raw 等原生协议端点）：接收外部包名、路径、上传内容，是注入与路径穿越的主要面。
- proxy 仓库的上游交互：拉取并缓存上游制品，涉及上游不可信内容与上游凭据使用。
- Web 控制台（`/` 提供 React 应用，`/assets/*` 静态资源）：登录与管理操作的入口。

后端以 `#![forbid(unsafe_code)]` 编译，从语言层面消除内存安全类攻击面。建议将服务置于反向代理之后做 TLS 终结。

防护边界（分层防御）：

- 应用层（L7）防护由本程序承担：多维限流、并发/连接上限、慢速攻击超时、访问异常检测与自动封禁、IP 黑白名单、CC 挑战、可配置 WAF 规则引擎与监控告警。该套件为 P2 能力，详见 ADR-0008。
- L3/L4 体积型 DDoS 不在本程序职责内，须由前置反向代理 / CDN / WAF 承担——这是部署侧的必备一环，不可只依赖应用层防护。
- 遥测与隐私：访问/下载等使用分析数据为本机内部数据，默认不主动外发、不向外部平台 phone-home（详见 ADR-0009）。

身份供给与登录防护：

- 用户由管理员创建，系统**不开放公开自助注册**；首个管理员经引导创建（首启从环境变量 `JIANARTIFACT_ADMIN_*` 或随机口令，详见 ADR-0010），仅在空库时触发。
- 登录接口具备暴力破解防护：按账户 / IP 统计失败次数，超阈值在时间窗内临时锁定或限流（argon2 仅防存储侧泄露，不能替代登录侧防护）。
- 私有仓库错误语义：对匿名 / 无读权限者一律返回 `404`（隐藏存在性），不用 `401` / `403`（见 API.md §2）。
- 漏洞数据隐私（P2）：漏洞匹配采用本地漏洞库离线镜像 + 制品坐标本地匹配，不把“你托管了哪些包”逐包外发到外部漏洞服务（见 ADR-0012）。

## 2. 密钥与敏感数据

- 用户密码：使用 `argon2` 哈希后存储于 `users.password_hash`，绝不存明文，绝不可逆。
- API Token：以哈希形式存储于 `tokens.token_hash`，不存明文；明文仅在签发时一次性返回给调用方，此后无法找回，只能吊销重发。
- 上游凭据：proxy 仓库访问上游所需的凭据不入 DB 明文——数据库仅存引用（`repositories.upstream_auth_ref`），真值走配置文件（TOML）或环境变量（前缀 `JIANARTIFACT_`），env 覆盖优先于文件。
- 严禁硬编码与入库：任何密钥、口令、令牌、凭据、证书均严禁硬编码在代码、注释、提交信息中，也严禁通过版本库提交。仅提供 `config.example.toml` 占位示例入库；真实 `config.toml` / `config.local.toml`、`.env` / `.env.local`、数据目录（如 `./data`）、`*.db` 及 `*.log` 等均不入库。
- 日志脱敏：日志与错误信息中不得输出密码、Token、上游凭据等敏感真值；涉及凭据的字段需脱敏处理。Token 的过期/吊销、Basic/Bearer 鉴权边界等敏感流程同样不得回显凭据原文。
- 哈希参数：密码 argon2 采用符合 OWASP 当期推荐的参数（内存 / 迭代 / 并行度），随安全基线演进；API Token 为高熵随机串，仅存其哈希。首启引导若生成随机管理员口令，须为高熵强口令、仅打印到启动日志一次，并要求尽快改密。
- 制品校验和：每个制品提供 sha256 / sha1 / md5 / sha512。md5 / sha1 已不适合安全用途，此处仅为兼容 Maven 等客户端 sidecar；制品完整性校验以 sha256 及以上为准。

示例配置占位（仅示意，请勿写入真实值）：

```toml
# config.example.toml
[upstream.example]
auth_ref = "REPLACE_WITH_AUTH_REF"   # DB 仅存此引用
# 真实凭据通过 env 提供，例如 JIANARTIFACT_UPSTREAM_EXAMPLE_TOKEN=REPLACE_ME
```

## 3. 外部输入

所有外部输入在使用前必须校验，防止注入、越权与路径穿越。

- 包名 / 制品路径校验：对格式端点传入的包名、版本、制品路径进行合法字符与格式校验，拒绝包含空字节（`\0`）的输入。
- 路径穿越防护：落盘与读取 blob 时，对外部提供的路径进行规范化与边界检查，禁止 `..` 等穿越序列逃逸出仓库的存储根目录。
- 上传大小限制：制品上传走流式处理，不整体载入内存；对上传大小施加限制，防止超大文件耗尽磁盘或内存。
- 各格式输入校验：Maven / npm / Docker / Raw 各格式按其原生协议校验元数据与校验和；写入流程先落 blob 并校验 sha256 通过后再写元数据，避免孤儿索引与被篡改内容。
- proxy 上游内容：从上游拉取的制品视为不可信内容，落盘前校验、缓存时记录 sha256，保证缓存一致性。
- 上游连接信任：proxy 拉取上游默认校验上游 HTTPS 证书；如需对接自签名上游，须经显式配置允许（默认不放行），避免中间人。
- 上传大小限制：制品上传受可配置的大小上限约束，超限返回 `413`，防止超大上传耗尽磁盘 / 内存。
- 鉴权前置：任何写操作均先经鉴权中间件判定目标仓库写权限，校验通过后方可处理输入。
- 搜索结果过滤：跨仓库搜索与制品详情必须按调用方身份过滤，只返回其有读权限的仓库制品（匿名仅 public）；绝不通过搜索 / 详情泄露无权私有仓库的存在或内容。

## 4. 漏洞报告

如发现安全漏洞，请通过项目维护者的私有渠道私下上报，**请勿**公开提交 issue 或在公共讨论区披露，以免在修复前被利用。

上报请尽量包含：受影响的版本、复现步骤、影响范围与可能的缓解措施。

- 上报渠道：请联系项目维护者（具体私有联系方式见项目仓库说明，此处不公开个人信息）。
- 响应预期：维护者将在确认后评估严重程度并安排修复；高危问题优先处理。
- 披露策略：遵循协调披露——在修复发布前请对漏洞细节保密，修复版本发布后再行公开。

> 注：本文档不包含任何真实联系邮箱或个人信息，请以项目仓库中维护者指定的私有渠道为准。
