# 运维手册：JianArtifact

> 部署、升级、备份恢复、回滚、排障的操作指南。运维方式变化时更新。

JianArtifact 以单一可执行二进制运行：前端构建产物已在编译期嵌入二进制，元数据存于嵌入式 SQLite，制品 blob 存于本地文件系统（数据目录）。运行期无需独立数据库或中间件，零外部运行时依赖。运维对象因此收敛为三件：可执行二进制、TOML 配置文件、数据目录。

## 1. 部署

### 前置依赖
- 无外部运行时依赖：无需安装数据库、中间件、缓存或独立静态资源服务器。
- 仅需目标平台的可执行二进制一份。支持 Linux / Windows / macOS，x86_64 与 arm64。
- 可选但推荐：在二进制前置一层反向代理 / CDN / WAF，承担 TLS 终结与 L3/L4 体积型 DDoS 防护（应用本身只做 L7 应用层防护，体积型攻击须由前置设施兜底；L7 防护套件为 P2 能力）。

### 配置项
- 完整配置项清单、默认取向与环境变量映射见 [`CONFIG.md`](CONFIG.md)。
- 单个 TOML 配置文件（示例文件随仓库提供为 `config.example.toml`，复制为 `config.toml` 后按需修改）。
- 关键配置：监听地址与端口、数据目录路径、blob 存储路径。
- 环境变量覆盖：以 `JIANARTIFACT_` 为前缀的环境变量优先级高于 TOML 文件中的同名项。敏感项（如上游仓库凭据）建议通过环境变量注入，不要写入入库的配置文件。
- 数据目录：运行需要一个数据目录，用于存放 SQLite 元数据文件与制品 blob。首次启动会在其中初始化 SQLite 文件。
- 审计日志保留期（P2 / FR-31）：审计事件落 SQLite 的 `audit_log` 表，由后台任务按保留期轮转。经 TOML 节 `[observability.audit]` 配置 `retention_days`（默认 90 天，超期行定期删除）与 `max_rows`（默认 100 万，行数兜底，超限删最旧行）。审计数据为本机内部数据、默认不外发；如需缩小数据库体量可调小这两个值。
- 漏洞库离线镜像（`[vuln]`，默认关闭）：启用需设 `JIANARTIFACT_VULN_ENABLED=true`（或 TOML `[vuln] enabled = true`），并在 `ecosystems` 列出要镜像的生态（如 `["Maven","npm"]`，对应 OSV 数据源的生态名）。`source_base_url` 默认指向 OSV 公开数据集，`refresh_interval_secs` 控制刷新周期（默认 24 小时）。镜像下载需进程可联网访问该数据源；下载的是公开数据集整体镜像，不外发本机制品坐标。临时镜像 zip 落在数据目录下的 `vuln-mirror` 子目录、刷新后清理。

### 启动步骤
1. 将二进制放入部署目录，赋予可执行权限。
2. 准备配置文件：复制 `config.example.toml` 为 `config.toml`，填写监听地址、数据目录等。敏感凭据用 `JIANARTIFACT_*` 环境变量提供，不写入配置文件。
3. 创建并确认数据目录存在且进程对其有读写权限（示例：`./data`）。
4. 启动二进制（指向配置文件，例如以参数或工作目录约定加载 `config.toml`）。
5. 首次启动会自动初始化 SQLite 元数据库；典型配置下启动时间应在 2 秒内。
6. 首个管理员引导（仅空库首次启动）：若已设置环境变量 `JIANARTIFACT_ADMIN_USERNAME` 与 `JIANARTIFACT_ADMIN_PASSWORD`，据此创建首个管理员；否则系统生成随机口令并打印到启动日志（仅此一次）——请从启动日志取得该口令登录，并尽快改密。系统不开放公开自助注册，后续用户由管理员在控制台创建。

### 健康检查
- 健康检查端点：`GET /health`。返回成功表示进程已就绪、可对外提供服务。
- 部署后即用该端点确认服务存活；置于反向代理或进程守护下时，将其配置为存活探针。

### 可选：S3 兼容对象存储后端（FR-30 / ADR-0014）

默认部署 blob 本体落本地文件系统，零外部运行时依赖、开箱即用。若需把 blob 改存 S3 兼容对象存储（应对制品体量超出单机磁盘、或需对象存储级别的持久性/容灾），可启用可选的 S3 后端。

- **启用即引入外部依赖**：启用 S3 后端后，本部署不再是「零外部运行时依赖」——blob 读写依赖外部对象存储（AWS S3 / MinIO 等）的可用性与延迟，这是运维的显式选择。本地文件系统仍是默认与开箱即用形态；不需要对象存储的部署无需任何额外操作。
- **需专用构建**：S3 后端经 Cargo 特性 `s3` 编译进二进制，默认构建**不含** S3 代码与依赖。须使用启用了 `s3` 特性的构建（`cargo build --release --features s3`）。配置 `backend = "s3"` 但二进制未启用该特性时，进程启动**直接报错退出**，不静默回退本地（避免误以为已用对象存储）。
- **配置**：在 `config.toml` 设 `[data.storage] backend = "s3"`，并在 `[data.storage.s3]` 填 `region` / `bucket`（必填）、`endpoint`（自建网关如 MinIO 必填，指向 AWS 可省略）、`prefix`（对象 key 前缀，默认空）、`path_style`（MinIO 等需 true，默认 true）。完整键见 [`CONFIG.md`](CONFIG.md)。
- **凭据**：S3 access key / secret key 的真源沿用 AWS SDK 标准环境变量（`AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`，如临时凭据再加 `AWS_SESSION_TOKEN`），**绝不写入入库配置、绝不进日志或 DB**。建议以环境变量或运行环境的实例角色注入。
- **写入语义与本地一致**：S3 后端先把上传内容写本地临时文件并校验 sha256，再以内容寻址 key 流式上传（大对象走 multipart），成功才写元数据索引；失败清理临时文件与未完成分片，不留孤儿对象。临时文件位于数据目录下 `s3-tmp/`，需保证数据目录有足够磁盘空间中转。
- **备份恢复差异**：启用 S3 后，blob 本体在对象存储而非数据目录；备份/恢复需把对象存储（按桶+前缀）与数据目录的 SQLite 元数据**作为一个整体**一并处理，仍须保证索引与本体一致。SQLite 仍是元数据唯一真源。

## 2. 升级

JianArtifact 的升级以「替换二进制」为核心，配置文件与数据目录原地保留。

### 升级流程
1. 备份：升级前先按第 3 节完成一次数据目录备份（含 SQLite 文件与 blob），并记下当前二进制版本。
2. 停止运行中的旧进程。
3. 用新版本二进制替换旧二进制（配置文件与数据目录不动）。
4. 启动新二进制；新版本若包含 SQLite 结构变更，会在启动时执行迁移。
5. 用 `GET /health` 确认就绪，再做一轮关键路径冒烟（登录、仓库列表、一次读取）。

### 版本兼容注意
- 二进制内注入了 `VERSION`，升级前后请记录版本号，便于回滚定位「上一个已知良好版本」。
- 配置项可能随版本演进：升级前对照新版 `config.example.toml` 检查是否有新增/变更的配置键，避免缺项导致启动失败。
- 跨多个版本的大跨度升级，优先逐版升级而非一次性跨越，降低迁移与配置不兼容风险。

### SQLite 迁移注意
- 元数据结构变更通过启动时迁移完成；迁移会改写 SQLite 文件，属不可逆操作。
- 升级（触发迁移）前务必已有可用备份；一旦迁移后发现问题，回滚需还原备份的 SQLite 文件（见第 4 节）。
- 迁移期间不要并发启动多个实例指向同一数据目录，避免对同一 SQLite 文件的写竞争。
- 迁移过程中如进程异常退出，启动新进程前先核对数据目录与备份的一致性。

## 3. 数据备份与恢复

### 备份内容
全部持久状态都在数据目录内，备份只需覆盖数据目录：
- SQLite 元数据文件（用户、Token、仓库、ACL、制品索引等元数据的唯一真源），以及其同名的 `-wal`、`-shm` 临时文件。
- 制品 blob 文件（本地文件系统中的实际制品本体）。
- 配置文件 `config.toml`（如含部署个性化配置；敏感凭据由环境变量提供，不在备份内）。

> 元数据（SQLite）与 blob（文件系统）必须作为一个整体一起备份与恢复，避免索引与本体不一致（孤儿索引或丢失本体）。

### 频率与保留
- 频率按写入活跃度确定：写入频繁建议每日；变化较少可按周。重要升级前必须临时全量备份一次。
- 保留策略示例：保留最近若干个每日备份与若干个每周备份，过期自动清理。具体周期按容量与恢复点目标自行设定。
- 备份产物应存放在与运行实例不同的存储介质/位置，避免单点同时损毁。

### 备份步骤
1. 为获得一致快照，优先在停止写入或停止进程后备份；若需热备，应使用一致性快照手段，避免在 SQLite 写入半途复制。
2. 完整复制整个数据目录（含 SQLite 文件与 blob 子目录）到备份位置。
3. 记录备份对应的二进制版本号，便于恢复时匹配兼容的二进制。

### 恢复步骤
1. 停止当前进程。
2. 用与备份匹配版本的二进制（避免新二进制对旧结构二次迁移引发偏差）。
3. 将备份的数据目录整体还原到目标位置，覆盖现有 SQLite 文件与 blob。
4. 启动进程，用 `GET /health` 确认就绪。
5. 抽样校验：登录、列出仓库、下载一个已知制品并核对其 sha256，确认元数据索引与 blob 本体一致。

### 恢复演练
- 定期（如每季度）在非生产环境用最近一次备份执行完整恢复，验证备份确实可用、流程无遗漏。
- 演练要点：恢复后服务能起、`GET /health` 通过、关键制品可下载且 sha256 校验一致。
- 记录每次演练的时间、所用备份、结论与发现的问题；据此修正备份频率或步骤。

## 4. 回滚

回滚分两层：二进制回滚与数据回滚，二者边界需分开判断。

### 二进制回滚
1. 停止当前（新版本）进程。
2. 用上一个已知良好版本的二进制替换当前二进制（版本号见升级前记录）。
3. 启动并用 `GET /health` 确认就绪，再做关键路径冒烟。

### 数据回滚边界
- 若本次升级未触发 SQLite 迁移：数据目录格式未变，通常只需回退二进制即可，无须动数据。
- 若本次升级已触发 SQLite 迁移：新结构对旧二进制不兼容，仅回退二进制不足以恢复，必须同时按第 3 节将数据目录还原到迁移前的备份。
- 数据回滚会丢失迁移后到回滚点之间产生的写入（新增用户、Token、仓库、上传的制品等）。回滚前需评估该数据丢失窗口是否可接受，并尽量在还原前导出/记录这段时间的关键变更。
- 制品 blob 与 SQLite 索引必须一并回滚到同一时间点，避免回滚后出现索引指向不存在 blob 或 blob 无索引的不一致。

## 5. 排障

排查通用顺序：先看 `GET /health` 是否通过 → 看进程日志 → 定位到对应模块（认证、鉴权、代理缓存、存储、元数据）。日志为应用进程输出，可按部署方式重定向到文件或交由进程守护采集。

| 现象 | 排查路径 | 处置 |
| --- | --- | --- |
| 进程启动失败、立即退出 | 看启动日志：配置加载报错？数据目录无权限？端口被占用？ | 修正 `config.toml` 缺项或类型错误；确保数据目录存在且可读写；释放或更换监听端口 |
| 升级后启动失败 | 是否卡在 SQLite 迁移；新版是否有新增必填配置键 | 对照新版 `config.example.toml` 补齐配置；迁移失败则按第 4 节还原备份并回退二进制 |
| `GET /health` 不通过 | 进程是否存活；监听地址/端口与反向代理是否一致 | 重启进程；核对绑定地址与代理转发目标；查看日志中的初始化错误 |
| 匿名能访问私有仓库 / 越权 | 复核目标仓库可见性（public/private）与该用户的每仓库读写 ACL、全局角色 | 修正仓库可见性或 ACL；私有仓库对未授权（含匿名）必须拒绝（401/404），如未拒绝按缺陷处理 |
| 包管理器客户端鉴权失败（mvn/npm/docker/curl） | 区分 Bearer Token 与 Basic Auth；Token 是否已吊销/过期；客户端是否携带正确凭据 | 重新签发 Token；核对客户端鉴权配置；查日志中鉴权中间件的拒绝原因（凭据已脱敏） |
| proxy 仓库拉取失败或缓存为空 | 上游地址是否可达；上游凭据（`upstream_auth_ref` 对应的 env 真值）是否正确；上游是否返回错误 | 修正上游地址与凭据环境变量；上游不可用时依回退策略处理；确认 cache-miss 拉取链路通畅 |
| 下载制品报校验/损坏 | 比对 SQLite 索引中的 sha256 与 blob 实际内容；是否存在孤儿索引或丢失本体 | 若元数据与本体不一致，从备份恢复该数据目录（索引+blob 一起回到同一时间点） |
| 大文件上传/下载中断或内存异常 | 确认走的是流式路径；磁盘空间是否充足；客户端连接是否中断 | 清理磁盘空间；重试上传（写入先落 blob 校验 sha256 再写元数据，失败不留孤儿索引） |
| 体积/启动异常变慢 | 是否误用非 release 档；数据目录是否过大、磁盘 IO 是否瓶颈 | 使用 release 构建产物；评估数据目录容量与磁盘性能 |
| 漏洞库镜像未更新/落库为空 | `[vuln] enabled` 是否为 true、`ecosystems` 是否非空；进程能否联网访问 `source_base_url`；看日志中"漏洞库离线镜像"相关 ERROR/WARN | 开启并配置生态；放通到数据源的出网；数据源不可达时单生态失败仅告警跳过、不影响其余，恢复网络后下轮自动重试 |

关键日志/指标位置：
- 日志：应用进程标准输出/错误，含请求 ID 与关键参数，凭据/密钥已脱敏。按部署方式重定向到日志文件或由守护进程采集。
- 健康状态：`GET /health` 端点。
- 数据状态：数据目录内的 SQLite 文件为元数据真源，blob 子目录为制品本体；二者一致性是排障核心抓手。
