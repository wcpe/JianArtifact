# 运维手册：JianArtifact

> 部署、升级、备份恢复、回滚、排障的操作指南。运维方式变化时更新。

JianArtifact 以单一可执行二进制运行：前端构建产物已在编译期嵌入二进制，元数据存于嵌入式 SQLite，制品 blob 存于本地文件系统（数据目录）。运行期无需独立数据库或中间件，零外部运行时依赖。运维对象因此收敛为三件：可执行二进制、TOML 配置文件、数据目录。

## 1. 部署

### 前置依赖
- 无外部运行时依赖：无需安装数据库、中间件、缓存或独立静态资源服务器。
- 仅需目标平台的可执行二进制一份。支持 Linux / Windows / macOS，x86_64 与 arm64。
- 可选但推荐：在二进制前置一层反向代理 / CDN / WAF，承担 TLS 终结与 L3/L4 体积型 DDoS 防护（应用本身只做 L7 应用层防护，体积型攻击须由前置设施兜底；L7 防护套件为 P2 能力）。

### 配置项
- 完整配置项清单、默认取向与环境变量映射见 [`CONFIG.md`](CONFIG.md)。
- 单个 TOML 配置文件（示例文件随仓库提供为 `config.example.toml`，复制为 `config.toml` 后按需修改）。
- 关键配置：监听地址与端口、数据目录路径、blob 存储路径。
- 环境变量覆盖：以 `JIANARTIFACT_` 为前缀的环境变量优先级高于 TOML 文件中的同名项。敏感项（如上游仓库凭据）建议通过环境变量注入，不要写入入库的配置文件。
- 数据目录：运行需要一个数据目录，用于存放 SQLite 元数据文件与制品 blob。首次启动会在其中初始化 SQLite 文件。
- 审计日志保留期（P2 / FR-31）：审计事件落 SQLite 的 `audit_log` 表，由后台任务按保留期轮转。经 TOML 节 `[observability.audit]` 配置 `retention_days`（默认 90 天，超期行定期删除）与 `max_rows`（默认 100 万，行数兜底，超限删最旧行）。审计数据为本机内部数据、默认不外发；如需缩小数据库体量可调小这两个值。
- 使用分析采集（P2 / FR-57）：制品访问 / 下载的聚合计数始终落 SQLite 的 `usage_stats` 表（异步采集、不阻塞主请求，采集失败不影响业务）。逐条明细（`usage_events`）默认关闭；如需明细，设 TOML 节 `[observability.usage]` 的 `detail_enabled = true`，并以 `max_detail_rows`（默认 100 万）兜底裁剪明细行数。统计数据为本机内部数据、**默认不主动外发、不向外部遥测 phone-home**；当前不提供任何外部导出 / 上报通道（本批不做导出）。数据面板展示（FR-58）尚未实现。
- Prometheus 指标端点（P2 / FR-32）：经 TOML 节 `[observability.metrics]` 配置。`enabled`（默认 true）控制是否暴露 `GET /metrics`（关闭则端点 404 且不安装 recorder）；`allow_anonymous`（默认 false）控制匿名抓取。指标为进程内自采的 pull 模型，仅在被抓取时渲染，不向任何外部端点 push / remote-write，数据本机内部、默认不外发。**安全提示**：默认仅 Admin 可抓取；若把 `allow_anonymous` 设为 `true`，端点将对任意匿名请求开放，**务必先把 `/metrics` 限定在内网 / 反向代理 / 防火墙之后**，否则服务的请求量、延迟分布、回源健康度等运行画像可能被外部探知。指标标签均为有界枚举（method / status_class / format 等），不含仓库名 / 路径 / 用户名，抓取本身不泄露制品坐标。Prometheus 抓取配置可用 Basic / Bearer 凭据访问受保护端点（默认模式）。**七层防护监控指标（P2 / FR-56）**也经本端点暴露，可用于在外部 Prometheus 侧自定义防护告警规则（见下「防护监控与告警」）：`jianartifact_rate_limit_rejected_total{dimension=ip|token|repo|concurrency}`（限流被拒）、`jianartifact_ban_triggered_total`（自动封禁触发）、`jianartifact_ban_active_ips`（当前封禁 IP 数 gauge）、`jianartifact_cc_challenge_issued_total` / `jianartifact_cc_challenge_failed_total`（CC 挑战下发 / 失败）、`jianartifact_waf_blocked_total`（WAF 阻断）、`jianartifact_slowloris_timeout_total`（慢速超时 / 截断拒绝）；这些指标同为低基数，不以 IP / 仓库名 / 规则模式串作标签。
- 多维速率限制与并发上限（P2 / FR-33 + FR-51 / ADR-0008，`[protection.rate_limit]`，默认关闭）：启用后，限流中间件按**连接来源 IP**、**已认证身份/用户**（用户及其所有 Token / 会话）与**目标仓库**（按格式路径首段仓库名）分别在固定时间窗内计数，任一维度单窗超阈值即返回 `429`（带 `Retry-After`）；并按 IP / 用户 / 仓库限制**在途并发请求数**，任一维度超并发上限同样返回 `429`，均不进入业务。经 TOML 节 `[protection.rate_limit]` 配置 `enabled`（默认 false）、`window_secs`（默认 60）、`ip_max_requests`（默认 1200）、`identity_max_requests`（默认 2400）、`repo_max_requests`（默认 0 不启用）、`ip_max_concurrent` / `user_max_concurrent` / `repo_max_concurrent`（默认 0 不限并发）。**默认阈值保守宽放、新增维度默认不启用**，不误杀正常包管理器批量并发拉取（如 CI 高频拉取）——调小阈值前请先评估正常峰值，避免误伤合法客户端。并发上限以在途计数实现，请求结束（含出错 / panic）可靠归还、不泄漏在途计数。**反向代理注意**：来源 IP 取连接级地址、**不采信 `X-Forwarded-For`**（防伪造来源绕过），故部署在反向代理之后时，所有请求的连接 IP 都是代理地址、IP 维度限流与并发会退化为全局阈值——此场景应在前置代理层做按真实客户端 IP 的限流，本应用的 IP 维度仅作兜底，身份 / 用户与仓库维度仍按真实主体 / 真实仓库生效。这是应用层（L7）多维限流与并发/连接上限；体积型（L3/L4）攻击仍由前置反向代理 / CDN / WAF 承担。
- 访问异常检测与自动封禁 + IP 黑/白名单（P2 / FR-53 / ADR-0008，`[protection.ip_list]` + `[protection.ban]`，默认关闭）：
  - **IP 黑/白名单**（`[protection.ip_list]`）：`allow` / `deny` 为 IP 或 CIDR 数组（IPv4 / IPv6 均可，如 `10.0.0.0/8`、`2001:db8::/32`）。命中 `deny` 的来源在进入业务前直接拒 `403`；命中 `allow` 的来源**豁免一切应用层防护**（限流 / 封禁 / 异常统计），**优先级高于黑名单**（同一 IP 同列两表时按白名单放行）。非法名单项启动时记 WARN 跳过、不阻断启动。两表默认空（不启用）。
  - **异常检测与自动封禁**（`[protection.ban]`）：启用后在 `window_secs`（默认 60）窗内按来源 IP 统计**异常信号**（响应 4xx，含 401/403 鉴权失败与限流 429；5xx 不计），单 IP 一窗内异常信号达 `threshold`（默认 100）即自动封禁 `duration_secs`（默认 900 秒），封禁期内该 IP 一律拒 `403`，到期自动解封。封禁状态**进程内内存维护、重启即清**（不落 DB）；故重启会清空所有自动封禁，黑名单（配置）则持久有效——需长期阻断的来源应入 `deny` 黑名单而非依赖自动封禁。
  - **反向代理注意**：黑白名单与封禁均取连接级来源 IP、**不采信 `X-Forwarded-For`**（防伪造绕过）。部署在反向代理之后时，所有请求连接 IP 都是代理地址——此时本应用的 IP 黑白名单 / 自动封禁会对"代理地址"生效（可能误伤全部流量或失去区分度），应在前置代理 / WAF 层按真实客户端 IP 做名单与封禁，本应用名单仅作直连兜底。
  - **误杀防范**：默认关闭且阈值保守宽放，正常包管理器偶发 404（探测制品存在性）/ 鉴权重试不应触顶；启用并调小阈值前请评估正常 4xx 峰值，必要时把可信来源（CI、内网）加入 `allow` 白名单豁免。仅应用层（L7）；体积型（L3/L4）攻击仍由前置设施承担。
- 慢速攻击超时与通用请求体大小限制（P2 / FR-52 / ADR-0008，`[protection.slowloris]`，默认关闭）：启用后，慢速防护中间件对**任意端点**的请求体设超时与通用大小上限。经 TOML 节 `[protection.slowloris]` 配置 `enabled`（默认 false）、`body_read_timeout_secs`（相邻数据块**空闲超时**，默认 30）、`header_timeout_secs`（等待首个数据块超时，默认 30）、`max_body_bytes`（请求体通用大小上限字节，默认 0 不启用）。**超时按「块间空闲」而非「整体时长」判定**：只要客户端持续有数据到达就不触发，故对正常大文件流式上传（mvn deploy 大 jar、docker push 大层）友好，只切断长时间不发数据的慢速 drip（slowloris、慢速 POST）连接——**调小超时档位前请评估正常网络抖动，避免误伤慢速合法客户端**。`max_body_bytes` **区别于** `limits.max_artifact_size`（后者仅约束制品上传体）：本项是对所有请求体的兜底上限，超限返回 `413`（带 `Content-Length` 时进入业务前即拒）；启用时应设得**高于预期最大制品体**，仅作异常超大体的兜底拦截，否则可能误杀正常大制品上传。这是应用层（L7）防护；体积型（L3/L4）攻击仍由前置反向代理 / CDN / WAF 承担——slowloris 等慢速攻击在应用层切断连接即可，但前置设施仍应配置连接数 / 超时等以补强。
- CC 挑战（工作量证明 PoW）（P2 / FR-54 / ADR-0008，`[protection.cc_challenge]`，默认关闭）：启用后，置于身份解析之内的 CC 挑战中间件对疑似 CC（HTTP 洪水）攻击的**匿名**来源下发工作量证明挑战——客户端须找到 `nonce` 使 `sha256(challenge_token + ":" + nonce)` 的二进制前导零位数达 `difficulty`，以请求头 `X-CC-Solution: <token>:<nonce>` 带证明重试方放行；无 / 错误证明返回 `429`（错误码 `cc_challenge_required`，响应体含挑战令牌 / 难度 / 过期）。经 TOML 节 `[protection.cc_challenge]` 配置 `enabled`（默认 false）、`difficulty`（PoW 前导零比特数，默认 20）、`ttl_secs`（挑战令牌有效期秒，默认 300）、`exempt_authenticated`（是否豁免已认证请求，默认 true）。⚠️ **对 CLI 的误伤风险**：正常包管理器 CLI（mvn / npm / docker / curl）**不会解 PoW**，启用后对匿名拉取无差别下发挑战会**直接打断正常匿名拉取**——故默认关闭、默认豁免已认证（带凭据）客户端，挑战只面向匿名可疑流量；**仅在确有 CC 攻击、且能接受匿名访问受影响时开启**，若部署允许匿名拉取公开仓库尤须谨慎评估。挑战令牌由服务端 HMAC 无状态签名（密钥复用 JWT 派生的域分隔子密钥、不落 DB、不存挑战态），绑定**连接级来源 IP**、**不采信 `X-Forwarded-For`**（换 IP 的证明不可复用，防绕过）；反代部署时来源 IP 是代理地址，匿名挑战会退化为按代理 IP 绑定，此场景的 CC 缓解更应交前置设施。未启用时直接放行、零开销。这是应用层（L7）防护；体积型（L3/L4）攻击仍由前置反向代理 / CDN / WAF 承担。
- 可配置 WAF 规则引擎（P2 / FR-55 / ADR-0008，`[protection.waf]`，默认关闭）：启用后，置于热路径前端的 WAF 中间件对**任意端点**按有序规则匹配请求属性。经 TOML 节 `[protection.waf]` 配置 `enabled`（默认 false）与有序 `[[protection.waf.rules]]` 数组，每条规则含 `field`（`method`/`path`/`query`/`header`，`header` 须配 `header_name`，大小写不敏感）、`pattern`、`match_type`（`literal` 子串包含 / `wildcard` 通配 `*`/`?` 整体匹配 / `regex` 正则子串搜索）、`action`（`block` 拒 403 / `allow` 放行并短路后续规则）。匹配**按声明顺序、首个命中生效**——可把对合法模式的 `allow` 规则排在前面给其豁免，再用 `block` 兜底。规则在**启动期编译一次**（正则预编译）；**非法规则（字段 / 匹配类型 / 动作非法或正则无法编译）启动时记 WARN 跳过、不阻断启动**——配置后请检查启动日志确认无规则被意外跳过。**误杀防范**：`literal` 走子串包含、`regex` 走子串搜索，过宽的 `pattern`（如对 `path` 写 `/`、对常见参数 / UA 写宽松串）会误伤正常请求；`block` 规则上线前建议在测试环境验证其只命中目标请求，必要时配合 `allow` 给可信来源开豁免。WAF 按请求属性匹配、**与来源 IP 无关、不采信 `X-Forwarded-For`**。默认空规则集 + 关闭，不影响正常包管理器请求。这是应用层（L7）防护；体积型（L3/L4）攻击仍由前置反向代理 / CDN / WAF 承担。
- 防护监控与告警（P2 / FR-56 / ADR-0017，`[protection.alerts]`，默认关闭）：把上述各 L7 防护的命中量收口为**可观测 + 可告警**。① **指标**：各防护被触发的计数随 `GET /metrics` 暴露（指标名见上「Prometheus 指标端点」），运维可在外部 Prometheus 侧基于这些指标自定义告警规则（pull 模型，本应用不主动外发）。② **进程内阈值告警**：启用后在 `window_secs`（默认 300）窗内按维度（限流 / 自动封禁 / CC 挑战失败 / WAF 阻断 / 慢速超时）累加防护事件计数，单维度窗内计数达对应阈值即记**中文分级日志**（WARN，远超阈值升级 ERROR）并把告警写入 SQLite 的 `protection_alerts` 表（异步、不阻塞主请求，失败不影响业务）；同一维度**窗内只告警一次**（去抖、不刷屏）。经 TOML 节 `[protection.alerts]` 配置 `enabled`（默认 false）、`window_secs`（默认 300）、各维度阈值 `rate_limit_warn_threshold`（默认 1000）/ `ban_warn_threshold`（默认 50）/ `cc_challenge_fail_warn_threshold`（默认 1000）/ `waf_block_warn_threshold`（默认 500）/ `slowloris_warn_threshold`（默认 200）、`max_rows`（告警表行数兜底，默认 10 万）。③ **状态端点**：`GET /api/v1/protection/status`（仅 Admin）返回防护健康快照（各维度窗内计数、当前封禁 IP 数、最近告警）；`GET /api/v1/protection/alerts`（仅 Admin）分页查询告警历史（支持 `dimension` 过滤）。**误报规避 / 阈值调优**：默认关闭且阈值默认保守宽放——正常高频访问 / CI 合法批量拉取偶发的少量防护事件不应触顶；启用后请按自身流量基线观察 `/metrics` 各防护计数的正常峰值，再据此设阈值（建议设在正常峰值的数倍以上），避免无差别刷告警。窗内去抖确保单维度一窗最多一条告警，不会因持续命中而刷屏。**数据不外发**：告警是本机内部数据，**只落本地 SQLite、经 `/metrics` 被动 pull 或状态端点本地查询，本应用不内置任何外发型通知（Webhook / 邮件 / IM 等）**；如需告警外发，请在外部 Prometheus + Alertmanager 侧基于本应用暴露的指标自行配置（由外部系统 pull 后外发，本应用始终不主动 phone-home）。这是应用层（L7）防护的监控收口；体积型（L3/L4）攻击的监控告警仍由前置反向代理 / CDN / WAF 承担。
- 漏洞库离线镜像（`[vuln]`，默认关闭）：启用需设 `JIANARTIFACT_VULN_ENABLED=true`（或 TOML `[vuln] enabled = true`），并在 `ecosystems` 列出要镜像的生态（如 `["Maven","npm"]`，对应 OSV 数据源的生态名）。`source_base_url` 默认指向 OSV 公开数据集，`refresh_interval_secs` 控制刷新周期（默认 24 小时）。镜像下载需进程可联网访问该数据源；下载的是公开数据集整体镜像，不外发本机制品坐标。临时镜像 zip 落在数据目录下的 `vuln-mirror` 子目录、刷新后清理。
- 统一出站网络代理（`[network.proxy]`，P2 / FR-84 / ADR-0020，运行时可编辑见 FR-88 / ADR-0022，默认不注入）：受限网络下令全部出站请求（proxy 回源 / Nexus 在线迁移 / 漏洞库镜像 / OIDC / 在线更新）经统一正向代理出站。经 TOML 节 `[network.proxy]` 配置 `http`（HTTP 代理 URL）、`https`（HTTPS 代理 URL）、`no_proxy`（逗号分隔的直连绕过主机 / 域 / CIDR），或对应环境变量 `JIANARTIFACT_NETWORK_PROXY_HTTP` / `_HTTPS` / `_NO_PROXY`。**env-vs-系统环境**：三键任一给值即以本配置为真源（注入即关闭 reqwest 的自动系统代理探测，压过进程的系统 `HTTP_PROXY` / `HTTPS_PROXY` / `NO_PROXY`）；三键全空时不注入、保持现状（仍 honor 上述系统环境变量）。**凭据安全**：代理 URL 可含 `user:pass@` 凭据，凭据不入库、不进日志 / 错误信息——含凭据的代理 URL 建议仅经环境变量提供，不写入入库 TOML；配置无效时日志只记不含凭据的「代理配置无效」类消息。当前为全局一套代理（不分出站点），不支持 SOCKS。**运行时可编辑（FR-88）**：Admin 可经控制台「设置」页或 `PATCH /api/v1/settings` 在线改代理，**即时生效、无须重启**（锁外重建出站 client、原子换槽）；运行时改动的代理（含凭据）只入内存、**不写回 TOML**，重启回落文件 + env 配置——需持久代理仍应写 `config.toml` / env。

### 启动步骤
1. 将二进制放入部署目录，赋予可执行权限。
2. 准备配置文件：**首启时若配置文件路径（默认 `./config.toml`，或 `--config` 指定）不存在，二进制会自动生成一份带中文注释的默认配置到该路径并记 INFO 日志**（FR-90，已存在则绝不覆盖）——直接启动即可拿到现成可编辑的 `config.toml`，按需改监听地址、数据目录等。也可手动复制 `config.example.toml` 为 `config.toml` 后再启动。敏感凭据用 `JIANARTIFACT_*` 环境变量提供，不写入配置文件。
3. 创建并确认数据目录存在且进程对其有读写权限（示例：`./data`）。
4. 启动二进制（指向配置文件，例如以参数或工作目录约定加载 `config.toml`）。
5. 首次启动会自动初始化 SQLite 元数据库；典型配置下启动时间应在 2 秒内。
6. 首个管理员引导（仅空库首次启动）：若已设置环境变量 `JIANARTIFACT_ADMIN_USERNAME` 与 `JIANARTIFACT_ADMIN_PASSWORD`，据此创建首个管理员；否则系统生成随机口令并打印到启动日志（仅此一次）——请从启动日志取得该口令登录，并尽快改密。系统不开放公开自助注册，后续用户由管理员在控制台创建。

### 健康检查
- 健康检查端点：`GET /health`。返回成功表示进程已就绪、可对外提供服务。
- 部署后即用该端点确认服务存活；置于反向代理或进程守护下时，将其配置为存活探针。

### 可选：S3 兼容对象存储后端（FR-30 / ADR-0014）

默认部署 blob 本体落本地文件系统，零外部运行时依赖、开箱即用。若需把 blob 改存 S3 兼容对象存储（应对制品体量超出单机磁盘、或需对象存储级别的持久性/容灾），可启用可选的 S3 后端。

- **启用即引入外部依赖**：启用 S3 后端后，本部署不再是「零外部运行时依赖」——blob 读写依赖外部对象存储（AWS S3 / MinIO 等）的可用性与延迟，这是运维的显式选择。本地文件系统仍是默认与开箱即用形态；不需要对象存储的部署无需任何额外操作。
- **需专用构建**：S3 后端经 Cargo 特性 `s3` 编译进二进制，默认构建**不含** S3 代码与依赖。须使用启用了 `s3` 特性的构建（`cargo build --release --features s3`）。配置 `backend = "s3"` 但二进制未启用该特性时，进程启动**直接报错退出**，不静默回退本地（避免误以为已用对象存储）。
- **配置**：在 `config.toml` 设 `[data.storage] backend = "s3"`，并在 `[data.storage.s3]` 填 `region` / `bucket`（必填）、`endpoint`（自建网关如 MinIO 必填，指向 AWS 可省略）、`prefix`（对象 key 前缀，默认空）、`path_style`（MinIO 等需 true，默认 true）。完整键见 [`CONFIG.md`](CONFIG.md)。
- **凭据**：S3 access key / secret key 的真源沿用 AWS SDK 标准环境变量（`AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`，如临时凭据再加 `AWS_SESSION_TOKEN`），**绝不写入入库配置、绝不进日志或 DB**。建议以环境变量或运行环境的实例角色注入。
- **写入语义与本地一致**：S3 后端先把上传内容写本地临时文件并校验 sha256，再以内容寻址 key 流式上传（大对象走 multipart），成功才写元数据索引；失败清理临时文件与未完成分片，不留孤儿对象。临时文件位于数据目录下 `s3-tmp/`，需保证数据目录有足够磁盘空间中转。
- **备份恢复差异**：启用 S3 后，blob 本体在对象存储而非数据目录；备份/恢复需把对象存储（按桶+前缀）与数据目录的 SQLite 元数据**作为一个整体**一并处理，仍须保证索引与本体一致。SQLite 仍是元数据唯一真源。

## 2. 升级

JianArtifact 的升级以「替换二进制」为核心，配置文件与数据目录原地保留。

### 升级流程
1. 备份：升级前先按第 3 节完成一次数据目录备份（含 SQLite 文件与 blob），并记下当前二进制版本。
2. 停止运行中的旧进程。
3. 用新版本二进制替换旧二进制（配置文件与数据目录不动）。
4. 启动新二进制；新版本若包含 SQLite 结构变更，会在启动时执行迁移。
5. 用 `GET /health` 确认就绪，再做一轮关键路径冒烟（登录、仓库列表、一次读取）。

### 版本兼容注意
- 二进制内注入了版本号（来源 `Cargo.toml`，可用 `--version` 查看），升级前后请记录，便于回滚定位「上一个已知良好版本」。
- 配置项可能随版本演进：升级前对照新版 `config.example.toml` 检查是否有新增/变更的配置键，避免缺项导致启动失败。
- 跨多个版本的大跨度升级，优先逐版升级而非一次性跨越，降低迁移与配置不兼容风险。

### SQLite 迁移注意
- 元数据结构变更通过启动时迁移完成；迁移会改写 SQLite 文件，属不可逆操作。
- 升级（触发迁移）前务必已有可用备份；一旦迁移后发现问题，回滚需还原备份的 SQLite 文件（见第 4 节）。
- 迁移期间不要并发启动多个实例指向同一数据目录，避免对同一 SQLite 文件的写竞争。
- 迁移过程中如进程异常退出，启动新进程前先核对数据目录与备份的一致性。

### 发布产物来源（FR-86）
升级所需的新版本二进制由 GitHub Actions 发布流水线（`.github/workflows/release.yml`）产出，无需本地手工编译：
- **正式版本**：在 GitHub 仓库的 Releases 页取对应 `vX.Y.Z` 的正式 Release 资产。
- **开发版快照**：每次推送 `master` 自动出一个滚动预发布（tag `dev`，标记为 prerelease），供尝鲜 / 验证；版本串形如 `0.3.0-dev.{run_number}+{shortsha}`，不建议用于生产。
- **资产命名**：每个发布目标提供一份二进制与一份 `.sha256` 校验文件，命名为 `jianartifact-{version}-{target}{ext}`（Windows 目标 `{ext}` 为 `.exe`，Linux / macOS 为空）。当前覆盖三目标：`x86_64-unknown-linux-gnu`、`x86_64-pc-windows-msvc`、`aarch64-apple-darwin`。
- **下载后校验**：替换二进制前先用配套 `.sha256` 校验下载完整性（如 `sha256sum -c` 或比对裸哈希），一致后再按上述升级流程替换。
- **在线自更新衔接**：管理员手动触发的在线自更新（FR-85，见下「在线自更新」）即以上述命名契约从最新稳定 Release 推导本机资产名、下载并校验 sha256 后原子替换；其依赖的发布产物由本流水线提供。

### 在线自更新（FR-85 / ADR-0021，默认关闭）
除上述手动「停进程 → 替换二进制 → 重启」外，管理员可经在线自更新让程序自行完成下载、校验、替换与重启，把手动三步收敛为一次触发。该能力**默认关闭**（出站开关），须显式开启。

- **触发**：管理员（仅 Admin）先经 `GET /api/v1/update/check` 查 GitHub 最新稳定 Release 与本机当前版本的差异，确认有更新后经 `POST /api/v1/update/apply` 触发；程序按本机平台推导资产名、流式下载并边下边算 sha256、与同名 `.sha256` 资产比对，一致才原子替换当前二进制并触发自动重启。**端到端「下载 → 替换 → 自重启 → 新版本运行」依赖真机**（Linux + Windows 各验一遍），生产启用前请在非生产环境实测。
- **配置 `[update]`**：`enabled`（默认 `false`，须显式置 `true` 才联网与允许更新，否则两端点一律拒绝、不联网）、`repo`（仓库源 `owner/repo`，默认 `wcpe/JianArtifact`）、`api_base_url`（GitHub API 基址，默认 `https://api.github.com`，可指向镜像）、`restart_mode`（`self` | `exit`，默认 `self`）、`download_timeout_secs`（资产下载整体超时秒，默认 300）。私有仓库可选访问 token 的真源是环境变量 `JIANARTIFACT_UPDATE_TOKEN`（**不写入入库配置、不进日志、不回显**；公开仓库免凭据）。出站经统一出站代理（`[network.proxy]`，见上）注入。**运行时可编辑（FR-88 / ADR-0022）**：`enabled` / `repo` / `api_base_url` / `restart_mode` / `token` 可经控制台「设置」页或 `PATCH /api/v1/settings` 在线改、即时生效、无须重启；运行时改动（含 token）只入内存槽、不写回配置文件，重启回落 `[update]` + env。
- **`restart_mode` 选择**：**裸进程部署用 `self`**——替换后程序经 graceful-shutdown 排空在途请求、释放端口，再自拉起新进程（带原启动参数）后退出。**置于进程管理器下（systemd / docker 等）用 `exit`**——程序排空后直接退出，由管理器按其重启策略拉起新实例；避免「程序自拉起 + 管理器又重启」叠加成双实例。选错会导致更新后服务不自动恢复（`self` 但实为容器内）或双实例（`exit` 但无管理器接管），部署形态变化时务必同步调整本项。
- **校验失败保护**：sha256 不一致即拒绝替换、删除临时文件、**保留旧二进制**，进程仍以旧版继续运行——替换是最后一步、仅校验通过才执行，校验失败永不触碰二进制。仅做 sha256 完整性校验，不做来源签名验签。
- **Windows `.old` 残留**：Windows 下替换会先把运行中的 `.exe` 改名为 `{exe}.old` 再落入新文件；该 `.old` 文件在**下次启动时自动清理**，无需手工删除。若清理前观察到该文件属正常现象；如改名因杀软 / 句柄锁定失败，程序会尽力还原并报错、进程续以旧版运行。
- **回滚**：在线自更新会在替换前留一份旧二进制副本（Unix 为 `{exe}.bak`），单步回退可用之；若新版触发了 SQLite 迁移，回退二进制仍须按第 4 节同步处理数据，边界不变。

## 3. 数据备份与恢复

### 备份内容
全部持久状态都在数据目录内，备份只需覆盖数据目录：
- SQLite 元数据文件（用户、Token、仓库、ACL、制品索引等元数据的唯一真源），以及其同名的 `-wal`、`-shm` 临时文件。
- 制品 blob 文件（本地文件系统中的实际制品本体）。
- 配置文件 `config.toml`（如含部署个性化配置；敏感凭据由环境变量提供，不在备份内）。

> 元数据（SQLite）与 blob（文件系统）必须作为一个整体一起备份与恢复，避免索引与本体不一致（孤儿索引或丢失本体）。

### 频率与保留
- 频率按写入活跃度确定：写入频繁建议每日；变化较少可按周。重要升级前必须临时全量备份一次。
- 保留策略示例：保留最近若干个每日备份与若干个每周备份，过期自动清理。具体周期按容量与恢复点目标自行设定。
- 备份产物应存放在与运行实例不同的存储介质/位置，避免单点同时损毁。

### 备份步骤
1. 为获得一致快照，优先在停止写入或停止进程后备份；若需热备，应使用一致性快照手段，避免在 SQLite 写入半途复制。
2. 完整复制整个数据目录（含 SQLite 文件与 blob 子目录）到备份位置。
3. 记录备份对应的二进制版本号，便于恢复时匹配兼容的二进制。

### 恢复步骤
1. 停止当前进程。
2. 用与备份匹配版本的二进制（避免新二进制对旧结构二次迁移引发偏差）。
3. 将备份的数据目录整体还原到目标位置，覆盖现有 SQLite 文件与 blob。
4. 启动进程，用 `GET /health` 确认就绪。
5. 抽样校验：登录、列出仓库、下载一个已知制品并核对其 sha256，确认元数据索引与 blob 本体一致。

### 恢复演练
- 定期（如每季度）在非生产环境用最近一次备份执行完整恢复，验证备份确实可用、流程无遗漏。
- 演练要点：恢复后服务能起、`GET /health` 通过、关键制品可下载且 sha256 校验一致。
- 记录每次演练的时间、所用备份、结论与发现的问题；据此修正备份频率或步骤。

## 4. 回滚

回滚分两层：二进制回滚与数据回滚，二者边界需分开判断。

### 二进制回滚
1. 停止当前（新版本）进程。
2. 用上一个已知良好版本的二进制替换当前二进制（版本号见升级前记录）。
3. 启动并用 `GET /health` 确认就绪，再做关键路径冒烟。

### 数据回滚边界
- 若本次升级未触发 SQLite 迁移：数据目录格式未变，通常只需回退二进制即可，无须动数据。
- 若本次升级已触发 SQLite 迁移：新结构对旧二进制不兼容，仅回退二进制不足以恢复，必须同时按第 3 节将数据目录还原到迁移前的备份。
- 数据回滚会丢失迁移后到回滚点之间产生的写入（新增用户、Token、仓库、上传的制品等）。回滚前需评估该数据丢失窗口是否可接受，并尽量在还原前导出/记录这段时间的关键变更。
- 制品 blob 与 SQLite 索引必须一并回滚到同一时间点，避免回滚后出现索引指向不存在 blob 或 blob 无索引的不一致。

## 5. 排障

排查通用顺序：先看 `GET /health` 是否通过 → 看进程日志 → 定位到对应模块（认证、鉴权、代理缓存、存储、元数据）。日志为应用进程输出，可按部署方式重定向到文件或交由进程守护采集。

| 现象 | 排查路径 | 处置 |
| --- | --- | --- |
| 进程启动失败、立即退出 | 看启动日志：配置加载报错？数据目录无权限？端口被占用？ | 修正 `config.toml` 缺项或类型错误；确保数据目录存在且可读写；释放或更换监听端口 |
| 升级后启动失败 | 是否卡在 SQLite 迁移；新版是否有新增必填配置键 | 对照新版 `config.example.toml` 补齐配置；迁移失败则按第 4 节还原备份并回退二进制 |
| `GET /health` 不通过 | 进程是否存活；监听地址/端口与反向代理是否一致 | 重启进程；核对绑定地址与代理转发目标；查看日志中的初始化错误 |
| 匿名能访问私有仓库 / 越权 | 复核目标仓库可见性（public/private）与该用户的每仓库读写 ACL、全局角色 | 修正仓库可见性或 ACL；私有仓库对未授权（含匿名）必须拒绝（401/404），如未拒绝按缺陷处理 |
| 包管理器客户端鉴权失败（mvn/npm/docker/curl） | 区分 Bearer Token 与 Basic Auth；Token 是否已吊销/过期；客户端是否携带正确凭据 | 重新签发 Token；核对客户端鉴权配置；查日志中鉴权中间件的拒绝原因（凭据已脱敏） |
| proxy 仓库拉取失败或缓存为空 | 上游地址是否可达；上游凭据（`upstream_auth_ref` 对应的 env 真值）是否正确；上游是否返回错误 | 修正上游地址与凭据环境变量；上游不可用时依回退策略处理；确认 cache-miss 拉取链路通畅 |
| 下载制品报校验/损坏 | 比对 SQLite 索引中的 sha256 与 blob 实际内容；是否存在孤儿索引或丢失本体 | 若元数据与本体不一致，从备份恢复该数据目录（索引+blob 一起回到同一时间点） |
| 大文件上传/下载中断或内存异常 | 确认走的是流式路径；磁盘空间是否充足；客户端连接是否中断 | 清理磁盘空间；重试上传（写入先落 blob 校验 sha256 再写元数据，失败不留孤儿索引） |
| 体积/启动异常变慢 | 是否误用非 release 档；数据目录是否过大、磁盘 IO 是否瓶颈 | 使用 release 构建产物；评估数据目录容量与磁盘性能 |
| 漏洞库镜像未更新/落库为空 | `[vuln] enabled` 是否为 true、`ecosystems` 是否非空；进程能否联网访问 `source_base_url`；看日志中"漏洞库离线镜像"相关 ERROR/WARN | 开启并配置生态；放通到数据源的出网；数据源不可达时单生态失败仅告警跳过、不影响其余，恢复网络后下轮自动重试 |

关键日志/指标位置：
- 日志：应用进程标准输出/错误，含请求 ID 与关键参数，凭据/密钥已脱敏。按部署方式重定向到日志文件或由守护进程采集。
- 健康状态：`GET /health` 端点。
- 运行指标：`GET /metrics`（默认仅 Admin）含 HTTP / 代理 / 审计与七层防护各维度计数。
- 防护状态：`GET /api/v1/protection/status`（仅 Admin）查看各防护维度窗内计数、当前封禁 IP 数与最近告警；防护阈值告警（启用时）记中文分级日志并落 SQLite 的 `protection_alerts` 表，可经 `GET /api/v1/protection/alerts`（仅 Admin）查询。均为本机内部数据、不外发。
- 数据状态：数据目录内的 SQLite 文件为元数据真源，blob 子目录为制品本体；二者一致性是排障核心抓手。
