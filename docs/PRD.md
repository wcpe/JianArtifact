# 产品需求文档（PRD）：JianArtifact

> 需求的单一真源（WHAT / WHY），也是产品的**需求登记册 + 路线图**——全生命周期都在记，不是一次性文档。每个需求在 §4 加一行 FR（带优先级/期 + 状态），交付即标版本；§4 FR 表是迭代中最常动的部分（🔥 高频），分期（§7）只是其中很小、很静的一页粗线条规划。单功能的详细规格放 `docs/specs/`，PRD 只保留"一行 FR + 期 + 状态"的索引级。

## 1. 背景与目标

团队在管理多种语言生态的构建产物时，往往要为每种包格式各自搭建一套制品库，或者部署体积庞大、依赖外部数据库与中间件的传统制品仓库，运维成本高、上手门槛高。JianArtifact 用 Rust 编写并打包为单一可执行二进制，原生支持 13 种主流包格式，内置多用户认证、全局角色与每仓库 ACL，支持公开/私有仓库隔离与匿名访客，零外部运行时依赖。

一句话价值主张：一个单文件、零外部依赖、开箱即用的轻量级多格式制品库管理器。

### 非目标

- 不引入任何外部数据库（PostgreSQL/MySQL/Redis 等）、消息队列、独立搜索引擎或容器编排等重型依赖；元数据只用嵌入式 SQLite，缓存只用进程内。
- 不在第一期实现 Cargo/PyPI/Go/NuGet/RubyGems/Terraform/Ansible/Pub/Conan 任一格式，也不为其预留镀金占位字段。
- 不在第一期提供 S3 兼容对象存储后端、OIDC/LDAP 认证、审计日志、指标端点、速率限制、Nexus OSS 迁移、group/virtual 聚合仓库、GC/保留策略、备份恢复；也不在第一期提供七层防护增强（并发/连接控制、慢速攻击防护、异常检测与自动封禁、CC 挑战、WAF 规则引擎、监控告警）、使用分析数据面板，以及权限增强（用户组/团队、读/写之外的细粒度权限动作）。
- 不在二进制内承担 L3/L4 体积型 DDoS 缓解：那属于前置反向代理 / CDN / WAF 的职责，本程序只做应用层（L7）防护。
- 使用分析与统计数据为本机内部数据，不主动外发、不向外部遥测平台 phone-home（导出需运维显式开启，默认关闭）。
- 不做服务端渲染前端；前端为 React 单页应用（Mantine 组件库），编译期嵌入二进制。

## 2. 角色

- **匿名访客（Anonymous）**：未认证。仅能读取 public 仓库；对 private 仓库一律拒绝（401/404）。
- **注册用户（User）**：认证后按角色与每仓库 ACL 访问；可自助管理自己的 API Token。
- **管理员（Admin）**：管理用户、仓库、ACL 与全局配置；并在 P2 管理用户组/团队、细粒度权限、七层防护策略与使用分析面板。
- **CLI/包管理器客户端（系统角色）**：mvn / npm / docker / curl 等通过 Bearer Token 或 Basic Auth 鉴权访问格式端点。

## 3. 用户故事

- 作为管理员，我希望创建并配置不同格式、类型（hosted/proxy）与可见性的仓库，以便为团队统一托管多语言制品。
- 作为管理员，我希望管理用户并为每个仓库授予读/写 ACL，以便精确控制谁能读、谁能写。
- 作为注册用户，我希望自助签发与吊销 API Token，以便在 CLI 与包管理器中安全鉴权而不暴露密码。
- 作为 CLI/包管理器客户端，我希望用 Bearer Token 或 Basic Auth 直接 push/pull 制品，以便沿用 mvn/npm/docker/curl 的既有使用习惯。
- 作为匿名访客，我希望无需登录即可读取 public 仓库，以便快速获取公开制品；而对 private 仓库被明确拒绝。
- 作为管理员，我希望部署一个单文件二进制并配一份 TOML 配置即可运行，以便零外部依赖地把制品库立起来。
- 作为部署运维，我希望首次启动时通过环境变量或自动生成的随机口令创建首个管理员账号，以便从空库自举系统、随后登录管理。
- 作为用户，我希望在所有可见仓库中搜索制品、点开制品看详情与“使用方式”片段（如 Maven 依赖、npm/docker 命令），以便快速找到并接入制品。
- 作为用户或管理员，我希望看到制品是否命中已知漏洞（基于本地漏洞库离线匹配），以便规避有风险的制品（P2）。

## 5. 非功能需求（NFR）

- **体积与依赖**：单一二进制体积 < 27MB（release 启用 strip + LTO）；零外部运行时依赖（无需独立数据库/中间件）。
- **启动性能**：典型配置下启动时间 < 2s。
- **大文件处理**：制品上传/下载走流式处理，不整体载入内存。
- **安全**：全局 `forbid(unsafe)`；密码用 argon2 哈希；API Token 以哈希形式存储；密钥/凭据不入库、不进日志、不进 DB 明文。
- **跨平台**：支持 Linux/Windows/macOS，覆盖 x86_64 与 arm64。
- **并发**：多用户并发上传/下载安全；代理缓存并发请求单飞合并。
- **防护边界（P2）**：内置应用层（L7）防护；L3/L4 体积型 DDoS 由前置反向代理 / CDN / WAF 承担，不在二进制内实现。
- **隐私与遥测（P2）**：访问/下载等使用分析数据落本地、不主动外发；任何外部导出默认关闭、需运维显式开启。
- **上传限制**：制品上传大小有可配置上限，超限返回 413；上传/下载流式处理，不整体载入内存。
- **登录防护**：登录失败次数过多触发账户/IP 临时锁定或限流，抵御暴力破解（P2 七层防护提供更强的异常检测与自动封禁）。
- **多校验和**：每个制品提供 sha256 / sha1 / md5 / sha512 摘要供校验（sha1 / md5 主要为兼容 Maven 等客户端 sidecar，安全完整性以 sha256 及以上为准）。
- **漏洞数据不外发（P2）**：漏洞匹配采用本地漏洞库离线镜像 + 制品坐标本地匹配，不把坐标逐包外发到外部漏洞服务（守“数据不外发”红线）。

## 6. 验收标准

- **P1 四种格式实机互通**：Maven/npm/Docker/Raw 各自 hosted+proxy 通过对应官方客户端真机验证（mvn deploy/get、npm publish/install、docker push/pull、curl PUT/GET）——**需由用户实机确认通过**。
- **权限隔离**：私有仓库匿名访问被拒（401/404），公开仓库匿名可读——单测穷举鉴权矩阵 + **需由用户实机确认通过**。
- **打包形态**：release 构建产物 < 27MB，且作为零外部依赖的单文件可正常启动。
- **Web 控制台全流程**：登录 → 建仓 → 授权 → 浏览全流程可用——**需由用户确认通过**。
- **首启自举**：空库首次启动可经环境变量或随机口令创建首个管理员并登录——需实机确认。
- **写入语义**：制品删除（需写权限）与各格式覆盖/不可变策略（如 Maven release 不可覆盖、snapshot 可覆盖）行为正确——单测 + 实机确认。
- **查看与检索**：可在可见仓库范围内搜索制品、点开制品看详情与“使用方式”片段；搜索结果不含调用方无权访问的私有仓库制品——单测 + 实机确认。
- **多校验和**：每个制品的 sha256 / sha1 / md5 / sha512 计算正确，且与各格式 sidecar、客户端校验一致——单测确认。

## 7. 分期（路线）

各期只描述**主题 / 目标**；**具体哪个 FR 属于哪期，以 §4 FR 表的优先级 / 期列为唯一来源**——加需求即在表里标期，本节不重复列编号、不随 FR 增长而改。

- **第一期（MVP）**：把核心制品库立起来——四种高频格式（hosted+proxy）、完整认证与每仓库 ACL、React Web 控制台、SQLite + 文件系统、单一二进制。
- **第二期**：扩格式（Cargo/PyPI/Go/NuGet）、S3 后端、可观测性（审计/指标/限流）、企业认证（OIDC/LDAP）、Nexus OSS 迁移框架；权限增强（用户组/团队、细粒度权限动作）、七层防护（多维限流、并发/连接控制、慢速攻击防护、异常检测与自动封禁、CC 挑战、WAF 规则引擎、监控告警）、使用分析数据面板（访问/下载统计，内部不外发）。
- **第三期**：补齐剩余格式、group/virtual 聚合仓库、GC/保留策略、备份恢复。
- **第四期（Web 控制台 UX 重构）**：把控制台往 Nexus-like 形态升级——折叠图标导航 + 全站信息密度提升、仓库浏览改文件树 + 右侧详情 + HTML View + 多格式依赖坐标、制品搜索移页眉 + 类型 icon + 树形结果、补齐匿名 / 普通用户角色页。属跨页 UI 重构 epic（多为重排已交付 UI + 少量净新），先出 mockup 对齐再逐 FR 开码（FR-92~96）。
- 后续若有新的**大阶段**才在此加一行主题——分期是**少数粗粒度阶段**，不是每个功能 / 版本一期。

> **分期不会堆到上百**：期是粗粒度路线图横轴，数量很少（走到成熟通常 3~6 个），一期含很多 FR、跨很多版本。**产品成熟（1.0 后稳态迭代）就不再加"第 N 期"**，改按版本（CHANGELOG / tag）+ 功能（FR 表 / specs）组织。期数往二十、上百涨，是把"期"误当版本 / 功能单位的滥用信号。
> 某期是否完成，看 §4 表里该期 FR 的"状态"是否都 `已交付`——进度不在本节维护。

## 8. 术语表

- **制品（Artifact）**：被托管的构建产物或包文件本体（如 jar、tarball、镜像层、任意二进制）。blob 本体存于文件系统/对象存储，元数据库仅存其索引与 sha256。
- **仓库（Repository）**：承载某一格式制品的逻辑容器，具有格式、类型（hosted/proxy）与可见性（public/private）属性。
- **hosted（托管型仓库）**：直接接收制品上传并提供下载的仓库类型。
- **proxy（代理型仓库）**：代理上游源、按需拉取并在本地缓存制品的仓库类型；cache-miss 时从上游拉取、校验、落盘、写索引，并发请求单飞合并。
- **ACL（访问控制列表）**：每仓库维度的读/写授权名单，按用户授予 read 或 write 权限。
- **可见性（Visibility）**：仓库的公开级别，取值 public（匿名可读）或 private（对未授权含匿名一律拒绝）。
- **API Token**：供 CLI/包管理器客户端鉴权的访问令牌，以哈希形式存储，可签发、列出与吊销。
- **单飞合并（请求合并）**：proxy 仓库在 cache-miss 并发场景下，将对同一上游资源的多个并发请求合并为一次上游拉取。
- **用户组/团队（P2）**：把多个用户归为一组，可按组授予仓库 ACL，简化批量授权。
- **细粒度权限动作（P2）**：仓库权限从读/写扩展为 read / write / delete / admin 四级动作。
- **七层防护（L7 防护）**：应用层防护手段总称，含多维限流、并发/连接上限、慢速攻击超时、异常检测与自动封禁、IP 黑白名单、CC 挑战与 WAF 规则引擎；不含 L3/L4 体积型 DDoS 缓解（由前置设施承担）。
- **CC 挑战**：针对应用层 CC（HTTP 洪水）攻击的人机校验手段（如验证码 / 工作量证明 PoW）。
- **WAF 规则引擎（P2）**：可配置的请求模式匹配与阻断规则集，用于拦截恶意请求。
- **使用分析（遥测）**：对访问量、下载量、热门制品、仓库用量等使用数据的内部采集与展示；数据落本地、不主动外发。
- **覆盖/不可变策略**：同名版本/路径重复上传时的处理规则，按格式而定（如 Maven release 不可覆盖、snapshot 可覆盖；Docker tag 可覆盖；Raw 可覆盖）。
- **分页响应**：列表接口统一返回 `{ items, total, offset, limit, has_more }` 结构，配合 `offset` / `limit` / 排序 / 过滤参数。
- **首个管理员引导**：空库首次启动时创建第一个管理员的自举流程（环境变量优先，否则随机口令打印到标准输出提示运维；口令不写入运行日志文件，守凭据不进日志红线）。
- **使用方式片段**：制品详情页按格式生成的获取与接入示例（如 Maven `<dependency>`、`npm install`、`docker pull`、Raw URL/curl，及把客户端指向本仓库的配置片段）。
- **多校验和**：每个制品提供的多种摘要（sha256 / sha1 / md5 / sha512）；sha1 / md5 主要为客户端兼容，安全完整性以 sha256 及以上为准。
- **漏洞库离线镜像（P2）**：把公开漏洞库数据（如 OSV）定期下载到本机，按制品坐标本地匹配判断是否命中已知漏洞，不逐包外发查询。
- **坐标级匹配**：按制品的生态坐标（如 Maven groupId:artifactId:version、npm 包名@版本）与漏洞公告比对，判断是否受影响。
