# ADR-0001：技术栈与单一二进制打包

## 状态

已接受

## 背景

JianArtifact 的定位是一个轻量级、零外部运行时依赖的多格式制品库管理器，目标是打包为单一可执行二进制并跨平台分发（Linux/Windows/macOS，x86_64 与 arm64）。这一定位对技术栈提出了硬性约束：发行物要做到单文件启动、无需独立数据库或中间件、release 体积控制在 27MB 以内。同时产品需要一个现代化的 React Web 控制台，但又不希望引入独立的静态资源服务器或额外的运行时。在此约束下需要一次性锁定后端语言、HTTP 框架、前端栈与打包方式。

## 决策

后端采用 Rust + `axum`（HTTP 框架）+ `tokio`（异步运行时），并在整个 crate 上启用 `#![forbid(unsafe_code)]`。前端采用 React + Vite + TypeScript，并以 Mantine 作为 UI 组件库，构建产物在编译期经 `rust-embed` 嵌入二进制，不部署独立静态资源服务器。最终交付单一可执行二进制：release 档启用 strip + LTO + `panic = "abort"`，目标体积 < 27MB。

## 理由

- Rust 产出静态链接的原生二进制，无需随附运行时，天然契合"单文件、零依赖"目标，并通过 `forbid(unsafe)` 在编译期排除内存安全隐患。
- `axum` 基于 `tokio`，异步模型适合制品库高并发上传/下载与代理拉取场景，生态成熟、中间件机制清晰，便于实现认证/鉴权/请求 ID/统一错误处理等横切关注点。
- React + Vite + TypeScript 是产品明确要求的前端形态，并采用 Mantine 组件库以快速构建一致、可访问的控制台界面；经 `rust-embed` 在编译期嵌入，使前端零运行时依赖、随主二进制一并分发，无须额外的静态服务器。
- strip + LTO + `panic = "abort"` 共同压低体积并去除展开开销，以满足 < 27MB 的体积门槛。

## 后果

- 正面：交付物为单一二进制，部署只需该文件加一份配置与一个数据目录；跨平台分发简单；编译期安全约束强。
- 负面/约束：前端改动需重新编译后端二进制才能生效（开发期可用 Vite 独立开发服务器缓解）；技术栈被锁定，后端 Rust+axum+tokio、前端 React+Vite+TS 为架构不变量，更换语言或框架属于架构决策，须另行起草新 ADR；体积门槛要求持续关注依赖引入对二进制大小的影响。

## 备选方案

- Go 或 Java 实现后端：Go 二进制偏大且生态此处优势不明显；Java 需要 JVM 运行时，违背"零外部运行时依赖、单文件"目标。均落选。
- 服务端渲染前端：不满足产品对 React 单页控制台的明确要求，且会让前后端耦合与运行形态复杂化。落选。
