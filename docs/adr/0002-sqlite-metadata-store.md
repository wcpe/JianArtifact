# ADR-0002：嵌入式 SQLite 元数据存储

## 状态

已接受

## 背景

制品库需要持久化一批结构化元数据：用户、API Token、仓库定义、每仓库 ACL、制品索引等。这些数据需要支持多用户并发、关系查询与权限判定。与此同时，产品的核心定位是"单一二进制、零外部运行时依赖"（见 ADR-0001），因此元数据存储不能引入独立的数据库进程或中间件。制品 blob 本体（动辄数百 MB）与轻量结构化元数据的存储诉求截然不同，需要分开承载。

## 决策

元数据采用嵌入式 SQLite 存储，驱动选用 `sqlx`（异步 + 编译期 SQL 校验 + bundled，免外部依赖）。SQLite 作为元数据的唯一真源，承载 users / tokens / repositories / repo_acl / artifacts 等表。制品 blob 本体存放于本地文件系统（P2 增加 S3 兼容对象存储后端），数据库中只保存 blob 的索引与 sha256 校验和，不保存 blob 内容本身。

## 理由

- SQLite 以嵌入方式随二进制运行，无需独立数据库进程，完全契合单文件零依赖目标；运行期只需一个数据目录存放 SQLite 文件与 blob。
- `sqlx` 提供异步访问与编译期 SQL 校验，bundled 模式将 SQLite 一并静态链接，避免对外部库的运行期依赖。
- 元数据与 blob 分离：结构化、需关系查询的部分进 SQLite，体积大的二进制内容进文件系统/对象存储，DB 只存索引 + sha256，既保证一致性校验又避免数据库膨胀。
- 制品写入遵循"先落 blob 并校验 sha256，再写元数据"的顺序，避免产生孤儿索引。

## 后果

- 正面：零外部依赖即可获得事务、关系查询与并发控制；部署与备份简单（一个数据目录即可）。
- 负面/约束：SQLite 的写并发能力弱于服务端数据库，高写入并发需配合事务边界与单飞合并等机制管理；元数据真源唯一性成为架构不变量——blob 在文件系统/对象存储、DB 仅存索引与 sha256；引入任何外部数据库（PostgreSQL/MySQL/Redis 等）均属被禁重型件，需先起草新 ADR。

## 备选方案

- 以纯文件/目录结构充当"数据库"：多用户认证与每仓库 ACL 的关系判定在纯文件方案下实现繁琐、易错且难保证一致性。落选。
- 外部 PostgreSQL/MySQL：功能更强但需要独立数据库进程，直接违背"单一二进制、零外部运行时依赖"的核心定位。落选。
