# ADR-0003：认证机制

## 状态

已接受

## 背景

制品库需要同时服务两类客户端：浏览器中的 Web 控制台用户，以及各类包管理器/命令行客户端（mvn、npm、docker、curl 等）。包管理器客户端有其约定俗成的鉴权习惯——多数走 Bearer Token，部分场景走 Basic Auth。产品要求支持本地用户名/密码登录，并允许用户自助管理供 CLI 使用的 API Token。安全上要求密码与 Token 都不得明文落库。同时产品规划了企业认证（OIDC/LDAP），但这些属于第二期范围，第一期不应为其引入占位字段或半成品实现。

## 决策

第一期实现三种认证方式：本地用户名/密码（密码以 `argon2` 哈希存储）、Bearer Token（供 CLI 使用，以哈希形式存储、不存明文）、Basic Auth（兼容包管理器 CLI 的登录习惯）。Web 端使用会话/JWT 维持登录态。在 `auth` 模块预留认证 provider 抽象作为接口边界，OIDC/LDAP 留待第二期实现，第一期仅保留边界、不落任何占位字段。

## 理由

- 本地用户名/密码 + argon2 哈希满足 Web 控制台登录的基本需求，argon2 是抗暴力破解的现代口令哈希方案。
- Bearer Token 是包管理器与脚本客户端的主流鉴权方式，Token 以哈希存储保证即便数据库泄露也无法还原明文凭据。
- Basic Auth 兼容部分包管理器 CLI 的既有登录习惯，降低客户端接入摩擦。
- provider 抽象只留接口边界、不落占位字段，遵循"简单优先、范围纪律"：避免为尚未实现的 OIDC/LDAP 提前镀金，同时为第二期扩展留出干净的扩展点。

## 后果

- 正面：覆盖 Web 与 CLI 两类客户端的认证诉求；凭据全程不以明文形态存在于 DB；为企业认证预留了清晰的扩展边界。
- 负面/约束：需要同时维护会话/JWT、Bearer、Basic 三条认证路径，认证中间件须统一识别并解析出身份（或匿名）；Token 过期/吊销、Basic/Bearer 边界、密钥脱敏属于高风险验证区，须重点测试；OIDC/LDAP 在第一期严禁出现实现或占位（如 `oidc_issuer`、`ldap_url` 等字段），否则违反范围纪律。

## 备选方案

- 仅支持 Token 认证：包管理器中习惯使用 Basic Auth 的客户端会不便，覆盖面不足。落选。
- 第一期即上 OIDC/LDAP：对 MVP 而言过重，引入外部身份提供方依赖与额外复杂度，与"先把核心制品库立起来"的分期主题冲突。落选（顺延至第二期）。
