# ADR-0004：授权模型

## 状态

已接受

## 背景

制品库需要对"谁能读/写哪个仓库"做出精确判定。访问主体包括匿名访客、注册用户与管理员；客户端既有浏览器也有包管理器 CLI。产品要求支持公开/私有仓库隔离与匿名访客：私有仓库对未授权者（含匿名）一律拒绝，公开仓库允许匿名读取。同时不同用户对不同仓库可能有不同的读写权限，单一的全局角色无法表达这种细粒度授权。需要在"粒度够用"与"避免一次性引入过重 RBAC"之间取得平衡。

## 决策

采用三层综合判定的授权模型：全局角色（Admin / User）+ 每仓库可见性（public / private）+ 每仓库读写 ACL（按用户授权名单，permission 取 read / write）。鉴权中间件按"目标仓库 + 操作（读/写）"综合可见性、角色与 ACL 三者做出判定。匿名访客仅能读取 public 仓库；private 仓库对匿名一律拒绝（401/404）；写操作须校验对应仓库的写权限。

## 理由

- 全局角色解决"谁是管理员"这类系统级权限（用户/仓库/ACL/全局配置管理），每仓库 ACL 解决"某用户对某仓库的读写"这类资源级权限，二者组合既覆盖系统管理也覆盖细粒度授权。
- 每仓库可见性（public/private）天然支持匿名访客模型：公开仓库可匿名读，私有仓库对未授权一律拒绝，满足公开/私有隔离要求。
- 三层综合而非单层，避免了"仅全局角色"粒度不足的问题，也避免了一次性铺开完整 RBAC 的过度设计。

## 后果

- 正面：授权粒度精确到"用户 × 仓库 × 读/写"，同时支持公开仓库的匿名读与私有仓库的严格隔离；模型边界清晰，鉴权判定集中在中间件。
- 负面/约束："私有仓库对匿名/未授权一律拒绝"成为安全不变量，任何放行都是红线；鉴权判定矩阵（public/private × 角色 × ACL 的全组合）是高风险验证区，须穷举测试，尤其"私有对匿名必拒"与"写权限边界"；ACL 数据随仓库与用户增长需要维护。

## 备选方案

- 仅用全局角色：无法表达"某用户只对某些仓库可写"的细粒度需求，粒度不足。落选。
- 一次性引入完整 RBAC（角色-权限-资源全量建模）：对当前需求过重，引入不必要的复杂度，违背简单优先原则。落选。

## 关联

- **P2 扩展**：见 ADR-0007（权限粒度与用户组），在本决策基础上扩展（不取代）——细粒度权限动作（read/write/delete/admin）与用户组/团队批量授权。
