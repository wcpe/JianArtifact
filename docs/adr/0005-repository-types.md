# ADR-0005：仓库类型

## 状态

已接受（其中「group/virtual 顺延第三期、第一期不实现」的部分已被 ADR-0039 取代——group 类型在本期实现）

## 背景

制品库需要为每种支持的包格式提供仓库。仓库在用途上存在两类典型诉求：一类是承载团队自己发布的制品（直传与下载），另一类是代理外部上游并在本地缓存制品以加速与降低对上游的依赖。此外业界还有 group/virtual 这类把多个仓库聚合为统一访问入口的形态，但这会显著增加路由与解析的复杂度。需要明确第一期为每种格式提供哪几类仓库，以及聚合仓库的取舍。

## 决策

每种格式支持两类仓库：hosted（制品直传与下载）与 proxy（代理上游并缓存制品，含缓存）。group/virtual 聚合仓库留待第三期，第一期不实现、不预留占位。proxy 仓库在 cache-miss 时从上游拉取、校验、落盘、写索引，并对并发请求做单飞合并，上游不可用时按策略回退。

## 理由

- hosted + proxy 两类覆盖了制品库最核心的两种用途：内部发布与外部代理缓存，是 MVP 必备能力。
- proxy 的缓存与单飞合并机制能显著降低对上游的重复拉取，并在高并发下避免重复下载，是代理仓库的关键价值。
- group/virtual 聚合涉及多仓库路由、优先级与冲突解析，复杂度高且非核心制品库立项所必需，按分期主题顺延至第三期，避免第一期镀金。

## 后果

- 正面：每格式两类仓库即可满足绝大多数制品库使用场景；proxy 的缓存与回退机制提升可用性与性能。
- 负面/约束：proxy 的代理缓存（cache-miss 拉取、并发单飞合并、上游失败回退、缓存一致性）是高风险验证区，须重点测试；group/virtual 在第一期严禁出现实现或占位字段（如 `upstream_group_members`），一旦出现即应删除或停下来确认；聚合能力顺延意味着需要聚合视图的用户在第三期前需自行组合多个仓库。

## 备选方案

- 第一期即实现 group/virtual 聚合仓库：复杂度高（多仓库路由与冲突解析），对 MVP 而言收益有限且偏离核心目标。落选（顺延至第三期）。
