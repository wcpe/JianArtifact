# ADR-0006：制品库迁移入口

## 状态

已接受

## 背景

许多团队已经在使用既有的开源制品库（典型为某主流开源制品库的 OSS 版本），迁移到 JianArtifact 时需要把存量制品搬运过来：既包括 proxy 仓库的配置与已缓存制品，也包括 hosted 仓库中团队自己发布的完整制品。迁移场景有两种现实形态：源系统仍在线（可经其 REST API 读取），或源系统已下线（只能从其离线 blob store 目录读取）。这部分属于第二期范围——第一期先把核心制品库立起来，迁移能力随已实现格式逐期扩展。

## 决策

提供两个迁移入口：在线 REST API 入口（源系统在线时经其 REST API 读取并搬运）与离线 blob store 入口（源系统已下线时从其离线 blob store 目录读取并搬运）。迁移内容覆盖 proxy 仓库的配置 + 已缓存制品，以及 hosted 仓库的完整制品。迁移能力随 JianArtifact 已实现的格式逐期扩展，第二期先建立迁移框架（独立的 `migrate` 模块）。

## 理由

- 双入口覆盖两种现实迁移场景：源系统在线时 REST API 读取最直接，源系统已下线时离线 blob store 入口仍可完成迁移，二者互补。
- 仅在线方案在源系统已下线时完全不可用；仅离线方案则强依赖源系统的内部 blob store 目录结构，跨版本脆弱。两者并存才能兼顾可用性与稳健性。
- 迁移随已实现格式逐期扩展，避免为尚未支持的格式提前编写迁移逻辑，符合范围纪律。

## 后果

- 正面：为存量用户提供平滑迁移路径，覆盖在线/离线两种现实情形与 proxy/hosted 两类仓库内容。
- 负面/约束：迁移属第二期，第一期严禁出现任何迁移实现或占位字段（如 `migration_source`），否则违反范围纪律；离线入口对源系统 blob store 目录结构的依赖存在跨版本兼容风险，需随源系统版本演进维护；迁移能力的格式覆盖面受限于 JianArtifact 当期已实现的格式。

## 备选方案

- 仅提供在线 REST API 入口：当源系统已下线时无法迁移，覆盖不全。落选。
- 仅提供离线 blob store 入口：强依赖源系统内部目录结构，跨版本脆弱且易失效。落选。
