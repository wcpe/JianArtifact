# ADR-0007：权限粒度与用户组（扩展授权模型）

## 状态

已接受

## 背景

ADR-0004 确立了"全局角色（Admin/User）+ 每仓库读写 ACL"的授权模型，满足第一期的多用户与公开/私有隔离需求。随产品演进，需要更细的权限控制（在读、写之外区分删除与仓库级管理）与批量授权能力（按团队/组授权，而非逐个用户授权）。本决策扩展（不取代）ADR-0004，把增强能力定位在 P2。

## 决策

在现有授权模型基础上扩展两项能力（P2）：

1. 仓库权限动作从 `read` / `write` 细化为 `read` / `write` / `delete` / `admin` 四级动作。
2. 引入用户组/团队（`groups` / `user_groups`），ACL 的授权主体从单一用户扩展为"用户或组"。

仍以全局角色（Admin/User）+ 每仓库 ACL 为基底，**不引入可自定义角色名与权限集的完整 RBAC**。

## 理由

- 四级动作覆盖绝大多数制品库权限场景（读取、发布、删除、仓库管理），用户组显著降低批量授权成本。
- 相较完整自定义 RBAC，该方案保持模型简单、易理解、易穷举测试，符合简单优先。
- 作为对 ADR-0004 的增量扩展，第一期的模型与数据不受影响，演进路径清晰。

## 后果

- 正面：授权更细、批量授权更省心。
- 负面/约束：P2 需扩展 `repo_acl`（授权主体 + 动作枚举）并新增 `groups` / `user_groups` 表；鉴权矩阵的测试组合相应增大（须穷举"用户/组 × 四级动作 × public/private"）；第一期不得提前引入这些字段/表（见 `scope-discipline`）。

## 备选方案

- 完整自定义 RBAC（可自定义角色与权限集 + 继承）：功能最强但复杂度与测试成本高，对当前规模过重。落选。
- 维持仅 `read` / `write`：不满足删除/管理动作与按组批量授权的诉求。落选。
