# ADR-0008：七层（L7）应用层防护

## 状态

已接受

## 背景

制品库对外暴露下载/上传与代理端点，易受应用层刷流与 CC（HTTP 洪水）攻击，需要内置防护与监控。但本产品是单一可执行二进制、定位轻量，既无法也不应承担 L3/L4 体积型 DDoS 缓解（那依赖网络层/带宽层设施）。需明确防护范围与边界，并把增强定位在 P2。

## 决策

在 P2 内置一套应用层（L7）防护中间件链：

- 多维限流（IP / Token / 用户 / 仓库 维度）。
- 并发与连接上限、慢速攻击（slowloris）超时、请求体大小限制。
- 访问异常检测与自动封禁、IP 黑/白名单。
- CC 挑战（验证码 / 工作量证明 PoW）。
- 可配置 WAF 规则引擎（请求模式匹配与阻断）。
- 防护监控与告警。

L3/L4 体积型 DDoS 明确交由前置反向代理 / CDN / WAF 承担，**不在二进制内实现**。限流阈值与 WAF 规则走配置，动态封禁状态落 SQLite。

## 理由

- L7 防护可在应用内按身份/仓库维度精准识别滥用，是单二进制能有效承担的部分。
- 把体积型攻击划到前置设施，既符合分层防御的行业实践，又守住"轻量、零外部依赖"的架构不变量。
- 可配置规则与人机挑战应对 CC，兼顾防护强度与误伤控制。

## 后果

- 正面：具备完整的应用层抗刷与可观测防护。
- 负面/约束：防护中间件位于请求热路径，须严控开销并穷举"误杀合法客户端"与"被绕过"两类边界；CC 挑战与 WAF 引擎增加配置与维护面；运维需在前置设施另行配置 L3/L4 防护（写入 `OPERATIONS` / `SECURITY`）。

## 备选方案

- 仅基础速率限制：不足以应对 CC 与异常封禁需求。落选。
- 在二进制内实现 L3/L4 体积型缓解：单进程无法有效抵御体积型攻击，且违背轻量与单一二进制定位。落选。
- 强依赖外部 WAF 承担全部 L7：降低产品自洽性、对中小部署不友好。落选（最终选择内置 L7 + 前置补强 L3/L4）。
