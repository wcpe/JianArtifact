# ADR-0014：S3 兼容对象存储后端

## 状态

已接受

## 背景

ADR-0002 确立了"嵌入式 SQLite 存元数据、blob 本体落本地文件系统"的存储形态，并预留了"P2 增加 S3 兼容对象存储后端"的演进口子。`storage` 模块当前以 `BlobStore` trait 抽象 blob 读写，`LocalFsStore` 是其唯一实现（先写临时文件、边写边算四种摘要、校验后按 sha256 寻址原子落定）。

PRD FR-30（P2）要求提供 S3 兼容对象存储后端，以支撑制品体量超出单机磁盘、或需要多副本/异地容灾的部署场景。但本产品的核心定位是"单一二进制、零外部运行时依赖、开箱即用"（ADR-0001、架构不变量 §2/§4）。引入对象存储不得侵蚀这一默认形态：本地文件系统必须仍是默认与开箱即用的 blob 后端，S3 只能是运维显式选择的可选第二后端。

同时，对象存储的语义与本地文件系统存在两处关键差异，必须在不破坏既有写入语义（架构不变量 §3：blob 先落盘并校验 sha256 通过、再写元数据索引；失败回滚不留孤儿）的前提下等价处理：

1. **寻址时机**：`LocalFsStore` 把内容写到临时文件后才知道 sha256，再 rename 到最终路径——文件系统允许"先写、后按内容命名"。S3 没有低成本的服务端 rename，对象 key 需在 PUT 时给定。
2. **句柄类型**：`BlobStore::get` 当前返回具体的 `tokio::fs::File`，对象存储无法产出该类型。

## 决策

在既有 `storage::BlobStore` 抽象下新增第二个实现 `S3Store`，作为**可选、opt-in** 的 blob 后端，本地文件系统 `LocalFsStore` 仍是默认。具体落地如下：

### 1. 编译特性 + 配置双开关，默认不编入

- 新增 Cargo 特性 `s3`（默认关闭）。`S3Store` 及其依赖（S3 客户端 crate）全部置于 `#[cfg(feature = "s3")]` 之下；默认 `cargo build` 产物**不含任何 S3 代码与依赖**，单一二进制、零外部运行时依赖、体积（NFR < 27MB）均不受影响。
- 运行期由配置选择后端，新增 `[data.storage]` 配置节：
  - `backend = "fs"`（默认）| `"s3"`。
  - `backend = "s3"` 但二进制未启用 `s3` 特性时，启动**直接报错并退出**（不静默回退本地，避免误以为已用对象存储）。
  - S3 子配置：`endpoint`、`region`、`bucket`、`prefix`（对象 key 前缀，默认空）、`path_style`（兼容 MinIO 等，默认 true）。

### 2. 选用 aws-sdk-s3（rustls）作为 S3 客户端

- 选用官方 `aws-sdk-s3`，`default-features = false` 并显式选用 **rustls** TLS 后端，避免默认拖入 `aws-lc-rs`/OpenSSL 等原生加密，保持与现有 `reqwest = { rustls-tls }`、`jsonwebtoken = { rust_crypto }` 一致的"纯 rustls、零原生 C 依赖"取向。
- 全部置于 `s3` 特性之下，默认产物不引入；体积增量在启用该特性时实测，确保 opt-in 形态可控。

### 3. blob 后端经枚举分发，构造期二选一

在 `storage` 模块以 `enum BlobBackend { Fs(LocalFsStore), S3(S3Store) }` 实现 `BlobStore`，启动时按配置构造其一注入 `AppState`。`format` / `proxy` 等上层只依赖 `BlobStore` trait，**对后端无感**，依赖方向不变（架构不变量 §1）。

### 4. 对象 key 仍以 sha256 内容寻址，写入语义等价保证

- 维持"内容寻址"：对象 key 为 `{prefix}{sha256[0..2]}/{sha256[2..]}`，与本地分桶布局同构。
- 解决"PUT 前未知 sha256"：`S3Store::put` 先把流式入站内容写到**本地临时文件**并边写边算四种摘要（复用既有流式逻辑），校验得到 sha256 后，再把该临时文件以最终 key **流式上传到 S3**；上传成功（HTTP 2xx）后删除临时文件。失败则删临时文件、不写任何对象，等价于"落盘并校验 sha256 通过、再写元数据索引"——`put` 返回成功即代表对象已在 S3 且内容经校验，上层据此再写 SQLite 索引，任一步失败都不留孤儿对象与孤儿索引。
- 幂等：PUT 前先 `HEAD` 该 key，已存在（同 sha256 即同内容）则跳过上传，与 `LocalFsStore` 的 try_exists 复用一致。

### 5. 流式、不整体载入内存

- 上传：大于阈值的对象走 S3 **分段上传（multipart upload）**，按固定分片大小（如 8 MiB）逐段从临时文件读出发送；小对象走单次 PUT。临时文件落本地磁盘而非内存，峰值内存不随 blob 体积线性增长（NFR、架构不变量 §2.4）。
- 下载：`get` 以**流式 GET** 返回响应体字节流，不在内存中聚合整对象。

### 6. `BlobStore::get` 返回类型抽象化

把 `get` 的返回从具体 `tokio::fs::File` 改为统一的 `Box<dyn AsyncRead + Send + Unpin>`（或等价的 `BlobReader` newtype），使两种后端可返回各自的读句柄。这是对既有 trait 的**最小化签名调整**，本地实现把 `File` 装箱即可，上层流式下载逻辑不变。

### 7. 凭据真源在配置/env，绝不入库、不进日志、不进 DB

- S3 access key / secret key 的真源是配置文件与环境变量（沿用 `JIANARTIFACT_` 前缀），**绝不写入 SQLite、绝不出现在日志或错误响应**。沿用 ADR-0002/架构不变量 §3：DB 只存索引 + sha256，不存任何凭据。
- 错误与调试日志中对 endpoint 之外的凭据字段一律脱敏。

### 8. SQLite 仍是元数据唯一真源

切换到 S3 后，SQLite `artifacts` 表语义不变——仍只存索引 + sha256，blob 本体在对象存储。不把对象列表/元数据复制成与 SQLite 互为权威的第二份真源；制品是否存在、归属哪个仓库一律以 SQLite 为准（架构不变量 §3）。

> 本 ADR **扩展**（不取代）ADR-0002：ADR-0002 的"嵌入式 SQLite + 本地文件系统 blob"仍是默认与基线形态，S3 仅为其上的可选 blob 后端。

## 理由

- **守默认形态**：编译特性默认关闭 + 配置默认 `fs`，使"单一二进制、零外部依赖、开箱即用"对绝大多数部署零成本；只有显式需要对象存储的运维才付出其复杂度。
- **复用既有抽象**：`BlobStore` trait 本就是为多后端预留的接缝，新增实现而非改造上层，依赖方向与 `format`/`proxy` 的薄耦合不变，符合开闭原则。
- **写入语义等价可证**：以"本地临时文件算 sha256 → 内容寻址上传 → 成功才返回"复刻本地的"先落盘校验再写索引"，把对象存储的最终一致性差异收敛在 `put` 内部，上层"blob 先于索引"的次序与回滚逻辑无需改动。
- **生态一致**：现有依赖（`reqwest` + `rustls-tls`、`jsonwebtoken` 的 `rust_crypto`、RustCrypto 哈希）已全面采用纯 Rust / rustls、零原生 C 依赖。选用 `aws-sdk-s3` 裁到 rustls，沿用同一取向，避免引入原生后端破坏跨平台静态链接与体积目标。

## 后果

- **正面**：支持超大制品体量与对象存储级别的持久性/容灾；本地与 S3 共用同一套写入语义与多校验和逻辑，行为一致、可穷举测试。
- **负面/约束**：
  - `put` 需一份本地临时文件中转（先算 sha256 再上传），对象写入比本地多一次本地磁盘往返；需保证临时文件在所有失败路径被清理。
  - 引入网络 IO 到 blob 路径：上传/下载受 S3 可用性与延迟影响，错误处理与超时需纳入既有 proxy/IO 超时约定。
  - `BlobStore::get` 返回类型变更会触及上层下载处理代码（最小化、机械式改动），须随本 ADR 落地同 PR 更新。
  - S3 后端为 P2，启用它的部署不再是"零外部运行时依赖"——这是运维的显式选择，须在 `OPERATIONS` 注明 S3 凭据与 endpoint 的配置方式及"启用即引入外部依赖"。
  - 必测高风险区（testing-and-quality §2.4/§2.5）须对 S3 后端同样覆盖：流式不整体载入内存、传输中断清理、sha256 校验失败回滚不留孤儿对象、blob 先于索引的次序与并发一致性。

## 备选方案

- **不抽象、直接在 `format`/`proxy` 里按后端分支**：会把存储细节泄漏到上层、违反依赖方向与开闭原则，且制造跨格式的 `if-else` 堆叠。落选。
- **S3 作为默认或唯一后端**：直接破坏"单一二进制、零外部依赖、开箱即用"的核心定位，违反架构不变量 §2/§4。落选。
- **运行期动态加载 S3 后端（插件/dylib）**：与单一静态二进制定位冲突，引入动态链接与分发复杂度。落选（改用编译特性 + 配置开关）。
- **把 sha256 改为上传后由 S3 校验和回读**：依赖各家对象存储的校验和实现差异大、且无法在 PUT 前内容寻址；不如本地临时文件算 sha256 稳定可控。落选。
- **rust-s3（非官方轻量客户端）**：依赖更少、体积更友好，但 multipart/签名与各家 S3 兼容实现的成熟度与维护活跃度弱于官方 SDK；协议正确性优先，落选，改用 `aws-sdk-s3` 裁 rustls。
