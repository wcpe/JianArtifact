# ADR-0015：可观测性——审计日志与 Prometheus 指标端点

## 状态

已接受

## 背景

制品库在多用户、多仓库场景下需要可追溯的安全审计能力（谁在何时对哪个仓库做了什么、结果如何），以及供运维抓取的运行指标（请求量、错误率、延迟、上传/下载字节、代理缓存命中等）。对应 PRD 的 FR-31（审计日志）与 FR-32（Prometheus 指标端点），均定位 P2，属"可观测性"主题（见 PRD §7 第二期）。

需要在以下约束内确定方案：

- 隐私红线（沿用 ADR-0009 基调）：数据本机内部，**默认不主动外发、不向外部遥测平台 phone-home**；任何外部导出须运维显式开启、默认关闭。
- 架构不变量：单一二进制、零外部运行时依赖（不引外部时序数据库 / 日志系统）；元数据只用嵌入式 SQLite，`meta` 是其唯一访问入口；`api` 层保持轻薄、中间件职责单一；不阻塞主请求路径（testing-and-quality §2.8）。
- 与既有能力区分：本 ADR 是面向**安全审计 + 运维监控**的可观测性；ADR-0009 的"使用分析"是面向运营的业务数据面板（访问量/下载量/热门制品）。三者数据去向一致（本地、默认不外发），但模型与受众不同，不复用同一张表、不互相替代。

## 决策

在 P2 提供两项相互独立的本机可观测能力，统一遵循"本地落地、默认不外发"：

### 1. 审计日志（FR-31）——元数据级安全事件，落 SQLite

- **事件模型**：只记元数据级安全/管理事件，不记请求体与制品内容。字段固定为
  `id`（自增）、`ts`（UTC 时间戳）、`actor`（用户名或 `anonymous`；不记密码/Token）、`actor_kind`（`session` / `token` / `basic` / `anonymous`）、`request_id`（关联 `api` 既有请求 ID）、`source_ip`、`action`（枚举：`login` / `token.issue` / `token.revoke` / `user.create` / `user.update` / `repo.create` / `repo.update` / `repo.delete` / `acl.update` / `artifact.upload` / `artifact.delete` 等）、`target_repo`（可空）、`target`（受影响对象坐标/路径，可空）、`result`（`success` / `denied` / `error`）、`detail`（结构化补充，JSON 文本，禁含凭据/隐私）。
- **采集范围（精选而非全量）**：认证结果（登录成功/失败/锁定）、授权拒绝（私有仓库未授权访问，记 `denied` 但不泄露资源存在性细节）、Token 与用户管理、仓库与 ACL 变更、制品上传/删除等**写与管理类**事件。普通匿名 public 读取不逐条入审计（交指标计数），避免撑爆 SQLite。
- **存储与体量治理**：经 `meta` 写入新增 `audit_log` 表（`meta` 仍是元数据唯一入口，不绕过它直接读写 SQLite）。配置 `observability.audit.retention_days`（默认 90 天）做基于时间的轮转：后台任务按保留期定期删除过期行；并提供 `observability.audit.max_rows` 行数硬上限作兜底（超限删最旧）。审计表只读查询经管理 API 暴露（仅 Admin），分页复用统一 `offset/limit` 结构。
- **异步不阻塞**：审计事件经进程内有界 channel 投递给单独的写入任务批量入库；主请求路径只做一次非阻塞 enqueue，**采集/写入失败只记 WARN 日志、不影响业务响应**；channel 满时按"丢弃并计数 + WARN"降级，不反压主路径。
- **脱敏**：密码、Token 明文、上游凭据、JWT 等一律不入审计；`actor` 只记用户名，`detail` 写入前过滤敏感键。

### 2. Prometheus 指标端点（FR-32）——进程内注册表，本机 `GET /metrics`

- **指标库**：采用进程内指标注册表（`metrics` facade + `metrics-exporter-prometheus` 进程内 recorder），不引外部时序数据库、不主动向任何外部端点推送。
- **端点**：在管理 API 下新增 `GET /metrics`，由 exporter 在请求时渲染当前注册表为 Prometheus 文本格式，**由运维自行用 Prometheus 抓取（pull 模型）**，二进制不做任何 push / remote-write。
- **指标集（初版，低基数）**：HTTP 请求总数 / 错误数（按 `method`、`status_class`、`format` 维度，**不以仓库名/路径做无界标签**避免高基数）、请求延迟直方图、上传/下载字节计数、代理缓存命中/未命中、上游拉取耗时与失败数、当前并发上传数、审计 channel 丢弃计数。指标值无凭据、无个人数据。
- **端点鉴权策略**：`/metrics` **默认要求认证、且仅 Admin 可访问**（复用现有认证中间件 + 全局角色判定）；考虑到 Prometheus 抓取常用静态凭据，提供配置 `observability.metrics.enabled`（默认开）与 `observability.metrics.allow_anonymous`（默认**关**）——仅当运维显式置真时才允许免认证抓取（用于把 `/metrics` 限定在内网/反代后的部署）。指标端点不暴露任何样本级敏感信息。
- **异步/低开销**：指标在请求热路径上只做原子计数/直方图观测（`metrics` 宏，无锁或细粒度原子），渲染只发生在 `/metrics` 被抓取时；不在热路径做字符串格式化或锁外重 IO。

两项能力的实现位置：审计经 `api` 中间件链中一个**单一职责的审计中间件**（在鉴权判定之后捕获 actor/result）+ 各管理 service 在领域动作处发审计事件；指标经 `api` 入口一个**单一职责的指标中间件**采集 HTTP 维度，其余指标由对应模块（proxy / storage）在自身边界埋点。`api` handler 保持轻薄，不写审计/指标聚合逻辑。

外部对接（如将指标接入外部 Prometheus 之外的 OTLP、或把审计转发到外部 SIEM）一律不在本期实现，且未来若做也须默认关闭、运维显式开启（与 ADR-0009 一致）。

## 理由

- **审计落 SQLite** 而非落文本日志文件：审计是可查询、需按仓库/动作/时间检索的结构化安全事实，与"SQLite 是元数据真源"一致；经 `meta` 入口统一治理体量与轮转，避免又引入一套日志文件管理。普通分级运行日志仍走既有 `tracing`，与审计各司其职。
- **进程内 Prometheus exporter（pull）** 守住"零外部运行时依赖、不主动外发"：指标只在本机端点被动暴露，是否抓取、抓到哪由运维决定，phone-home 风险为零；Prometheus 文本格式是运维生态事实标准，对接成本低。
- **异步有界 channel + 失败降级**：满足 testing-and-quality §2.8"采集异步不阻塞、采集失败不影响业务"；有界 + 丢弃计数避免审计积压拖垮内存或反压主路径。
- **精选事件 + 保留期轮转**：审计只记安全/管理类元数据事件、读流量交指标计数，从源头控制写入量，配合时间/行数双重轮转避免撑爆 SQLite。
- **`/metrics` 默认需 Admin、匿名显式开**：在"开箱安全"与"Prometheus 抓取便利"之间取平衡，默认不泄露运行画像，运维可在受控网络内显式放开。
- **审计 vs 使用分析分表**：受众与模型不同（安全取证 vs 运营统计），强行合并会污染语义、增大单表与查询复杂度；分表各自治理更清晰，且都守同一隐私红线。

## 后果

- 正面：具备安全可追溯（谁做了什么）与运维可观测（指标抓取）两项能力，全部本地化、隐私可控、零新增外部运行时依赖。
- 负面/约束：
  - 新增 `audit_log` 表与一支后台轮转任务，需在 ARCHITECTURE 数据模型、API.md（审计查询端点 + `/metrics`）、OPERATIONS（保留期、抓取与鉴权配置、`allow_anonymous` 风险提示）、SECURITY（审计脱敏与 `/metrics` 暴露面）同步说明。
  - 审计/指标中间件位于请求热路径，须穷举测试"不阻塞主路径""采集失败不影响业务""高基数标签不爆表""凭据/隐私不入审计与指标"等边界。
  - 指标标签必须严控基数（禁用仓库名/路径/用户作为无界标签），否则注册表与抓取代价随基数膨胀。
  - 新增 `metrics` / `metrics-exporter-prometheus` 依赖（已按 §15 经用户确认）。
  - `/metrics` 的 `allow_anonymous=true` 是运维显式承担的暴露面，须在文档明确其前提（限内网/反代后）。

## 备选方案

- **审计写入文本/JSON 行日志文件（不落 SQLite）**：实现简单，但无法结构化查询、需另建文件轮转与检索，且与"SQLite 是元数据真源"不一致，控制台也难直接展示。落选（运行态分级日志仍用 `tracing`，但审计走 DB）。
- **指标用 `prometheus` crate（含默认全局注册表）替代 `metrics` + exporter**：可行且纯 Rust，但 `metrics` facade 与项目既有 `tracing` 生态、tokio 异步栈衔接更自然，宏式埋点更轻、可换 recorder，耦合更低。列为次选。
- **复用 ADR-0009 的 analytics 表承载审计**：语义混淆（业务统计 vs 安全取证）、单表膨胀、查询与保留策略互相牵制。落选——两者独立建模、各自治理。
- **指标主动推送（push gateway / OTLP push / remote-write）**：违背"默认不外发、不 phone-home"。落选——只做被动 pull 端点；任何外部对接默认关闭、留待后续 ADR 评估。
- **不做审计 / 不做指标**：不满足 FR-31 / FR-32 的安全可追溯与运维可观测诉求。落选。
