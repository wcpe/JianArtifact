# ADR-0017：防护监控与告警

## 状态

已接受

## 背景

ADR-0008 已落地七层（L7）应用层防护——多维限流 / 并发控制（FR-33、FR-51）、慢速攻击防护
（FR-52）、访问异常检测与自动封禁 + IP 黑白名单（FR-53）、CC 挑战（FR-54）、可配置 WAF 规则引擎
（FR-55）。这些防护已能拦截，但**缺少可观测性**：运维既无法量化各防护被触发的规模与趋势，也无法
在被攻击 / 误杀时及时知晓。FR-56「防护监控与告警」要补齐这一环。

约束（来自架构不变量）：进程内、无外部 DB / MQ / Redis；`#![forbid(unsafe_code)]`；分层单向
`api → (auth/authz/repo/format) → (proxy/storage/meta) → config`；**数据不外发、不向外部遥测
phone-home**（ADR-0009）；只做应用层（L7）观测，不在二进制内做体积型缓解（ADR-0008）。已有指标
设施为进程内 `metrics` + Prometheus exporter、被动 pull（ADR-0015），标签**严格低基数**。

## 决策

防护监控与告警由三部分组成，全部进程内、零外发：

1. **防护计数接入既有 `/metrics`（ADR-0015）**：为五类防护新增低基数 `jianartifact_` 前缀指标，
   在各防护中间件命中点用 `metrics` 宏原子累加（未启用 recorder 时为 no-op）。指标名：
   `jianartifact_rate_limit_rejected_total`（标签 `dimension=ip|token|repo|concurrency`）、
   `jianartifact_ban_triggered_total`、`jianartifact_ban_active_ips`（gauge）、
   `jianartifact_cc_challenge_issued_total`、`jianartifact_cc_challenge_failed_total`、
   `jianartifact_waf_blocked_total`、`jianartifact_slowloris_timeout_total`。**严禁**以 IP / 仓库名 /
   规则模式串作标签（守 ADR-0015 基数纪律）。

2. **进程内阈值告警引擎 + 中文分级日志 + 落库**：进程内评估器在固定时间窗内按维度累加防护事件
   计数，单维度窗内计数达配置阈值即产生一条告警——按严重度记**中文分级日志**（WARN/ERROR）并经
   有界 channel **异步不阻塞落 SQLite**（`protection_alerts` 表，迁移 `0010`）。同一维度窗内**去抖**
   （一窗只告警一次，不刷屏）；评估纯在内存态完成，热路径只做原子累加 + 一次内存计数，**锁内无 IO**；
   告警入库与审计 / 使用分析同款范式（写任务批量落库、行数兜底裁剪、失败仅 WARN 不阻塞）。
   **默认关闭**、阈值默认保守宽放，避免无人值守刷告警与正常高频访问误报。

3. **管理员只读状态端点**：`GET /api/v1/protection/status`（仅 Admin），返回各维度窗内计数、当前
   封禁 IP 数、最近告警；`GET /api/v1/protection/alerts`（仅 Admin）分页查询告警历史（统一
   offset/limit）。**纯本机聚合、零外发**；axum handler 保持薄，逻辑下沉 service / meta。

4. **坚持数据不外发**：告警是本机内部数据，只落本地 SQLite、经 `/metrics` 被动 pull 或状态端点本地
   查询；**不内置任何外发型通知（Webhook / 邮件 / IM 等）**。若未来确需外发型通知，须另写 ADR。

本 ADR 扩展 ADR-0008（七层防护）的「监控告警」部分，并复用 ADR-0015（可观测性：指标 + 异步落库
范式）的设施；不改动二者正文。

## 理由

- **复用既有设施**：指标走已有 `/metrics`，告警落库走审计 / 使用分析已验证的异步有界 channel 范式，
  无需引入新机制，符合「简单优先」与架构不变量。
- **进程内 + 默认关闭 + 保守阈值**：满足无外部依赖约束，又把误报 / 噪声降到最低；与 ban / cc 默认
  关闭风格一致，启用与阈值由运维显式承担。
- **指标与告警分工**：指标给连续量化与趋势（外部 Prometheus 可抓取告警规则自定义），进程内告警给
  开箱即用的阈值提醒与本地落库审阅；两者互补。
- **只读状态端点**：让运维无需接外部监控也能本地巡检防护健康，且严守「数据不外发」。

## 后果

- 正面：防护从「能拦」升级到「可观测 + 可告警」；运维可量化攻击规模、发现误杀、本地审阅告警历史。
- 负面 / 约束：告警阈值需运维按自身流量基线调优（默认偏保守，可能漏报低烈度异常）；进程内计数与
  封禁状态重启即清（与 ADR-0008 一致，非持久统计真源——长期趋势看 `/metrics` 抓取侧或告警落库）。
- 约束固化：新增的防护指标必须保持低基数；告警表不存任何凭据；不得引入外发型通知（须另走 ADR）。

## 备选方案

- **接外部告警系统（Alertmanager / Webhook / 邮件）**：违反「数据不外发」默认且引入外部依赖，
  落选；外部告警可由运维在 Prometheus 侧基于本服务暴露的指标自行配置（pull 模型，不由本服务外发）。
- **告警状态也持久化为统计真源**：与 ADR-0008「封禁 / 计数进程内、重启即清」不一致且无必要；
  落库仅作告警事件审阅，连续量化交指标。
- **把告警评估放在独立后台 tick 扫描**：需另维护跨维度时序缓冲，复杂度更高；选择在命中点即时累加
  判定（去抖防刷屏），热路径开销仅一次原子累加 + 一次内存计数，更简单。
