# ADR-0019：迁移执行异步化为进程内任务（进度轮询 + 客户端重连）

## 状态

已接受

> 扩展 ADR-0006（Nexus 迁移）。**保留** ADR-0006「无须持久化迁移任务表」的决策——本 ADR 只引入**进程内（内存态）**任务与进度，不落库、不推翻该决策。

## 背景

FR-82 的在线拉取迁移是**同步**的：一次 HTTP 请求阻塞到整仓搬完才返回（实测某仓库 240 个 asset 约 4 分钟）。这带来两个问题：① 长时间阻塞请求、无任何进度反馈，运维无法判断卡住还是在跑；② 浏览器 / 网络一断，运维就失去了对迁移的可见性。迁移是低频、管理员手动触发的运维操作，但单次可能很久，需要可观测与可重连。

## 决策

把在线拉取迁移的**执行**从同步阻塞改为**进程内异步任务**：

- 触发端点（`POST .../online/migrate`）**立即返回 `job_id`**（`202`），实际搬运在后台 `tokio` 任务中跑。
- 任务进度（阶段 / 总数 / 已完成 / 已跳过 / 当前仓库与 asset / 各仓库结果 / 错误）保存在**进程内任务注册表**（`AppState` 持有的 `Arc<RwLock<HashMap<job_id, Arc<Mutex<进度>>>>>`），**不落库**。
- 新增只读查询端点：`GET .../migrate/jobs/{id}`（单任务进度快照）与 `GET .../migrate/jobs`（活动 / 近期任务列表，供**重连**找回）。
- 前端执行后**轮询**进度、展示导入队列；浏览器 / 网络断开重开后经任务列表 / 本地记的 `job_id` 重连续看。
- 任务注册表**有界**：保留最近 N 个任务、超出按时序淘汰，避免内存无界增长。
- **断点重连的范围**：仅客户端重连（任务在服务端后台继续）。**服务器重启任务即丢失**——靠迁移本身的幂等（同坐标已存在即跳过）重跑恢复，不持久化任务状态。
- 仅迁移**执行**异步化；发现 / 预览（FR-36/37）仍同步返回。

## 理由

- 内存态任务满足「长任务可观测 + 客户端可重连」的核心诉求，且**不违背** ADR-0006「无须持久化迁移任务表」——迁移幂等可重入，服务器重启后重跑即恢复，无须把任务状态做成第二份真源。
- 守架构不变量：进程内状态、不引外部队列 / 协调组件（见 architecture-invariants）；任务注册表只存内存态进度，SQLite 仍只是制品元数据真源。
- 轮询（而非 websocket / SSE）与既有防护监控页同款，前端实现一致、无需新长连接机制。
- 有界注册表避免内存泄漏；淘汰已完成的旧任务对运维无损（迁移结果以实际落库的制品为准）。

## 后果

- 正面：长迁移可观测（队列进度）、客户端可断可重连；触发请求不再长阻塞。
- 负面 / 约束：服务器重启丢失进行中任务的进度与「正在跑」状态（须重跑，靠幂等恢复）；进度为内存态、多副本部署下不共享（本项目为单二进制单实例，不涉及）；任务并发由管理员手动触发约束，不内置任务排队 / 限流（超出范围）。

## 备选方案

- **持久化任务表到 SQLite**（服务器重启续跑）：推翻 ADR-0006「无须持久化迁移任务表」，引入第二份真源与一致性维护成本；迁移幂等已能靠重跑恢复，收益不抵复杂度。落选。
- **SSE / websocket 推送进度**：需新长连接机制，与既有轮询范式不一致；迁移低频，轮询足矣。落选。
- **维持同步阻塞 + 仅加超时**：无法解决可观测与重连，长任务体验差。落选。
