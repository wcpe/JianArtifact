# ADR-0021：在线更新（管理员手动触发的自更新）机制

## 状态

已接受

## 背景

当前升级形态是运维手动「停进程 → 替换二进制 → 重启」（OPERATIONS §2），零自动化。FR-85 要在此之上提供管理员手动触发的完整自更新：程序自查 GitHub 最新稳定 Release、与当前版本比对，管理员确认后自行下载本机平台资产、校验完整性、替换运行中的二进制并自动重启。

这引入两个对架构有影响的新行为，须经 ADR 裁定：① 程序主动出站到 GitHub（与「数据不外发 / 不 phone-home」红线的边界）；② 程序替换并重启**自身运行中的二进制**（跨平台、与既有运维模型的关系）。

约束：技术栈锁定 rustls、单一二进制零原生运行时依赖、`#![forbid(unsafe_code)]`（ADR-0001）；密钥/凭据不入库、不进日志、不进 DB 明文（架构不变量 §3）；数据不外发、不向外部遥测 phone-home（架构不变量 §3、ADR-0009）；简单优先、不预留空壳。数据来源是 FR-86 发布流水线产出、命名契约固定的 Release 资产；出站复用 FR-84 / ADR-0020 的统一出站客户端。

## 决策

新增顶层模块 `update`（编排，依赖 `config`，不碰 `meta`/DB），经 `api::update` 薄 handler 暴露两个**仅 Admin** 端点：

1. **更新检查** `GET /api/v1/update/check`：查配置仓库 `{api_base_url}/repos/{repo}/releases/latest`，比对版本，返回是否有更新。
2. **应用更新** `POST /api/v1/update/apply`：按本机 target 下载对应资产 → 边下边算 sha256、与同名 `.sha256` 资产比对 → 一致则**原子替换**二进制 → 触发**自动重启**。

关键裁定：

- **出站默认关闭**：`[update] enabled=false` 为默认；未显式开启时检查/应用端点一律拒绝、绝不联网。仓库源 `repo` 与 `api_base_url` 可配（默认 `wcpe/JianArtifact` + 公共 GitHub API）。
- **仅版本检查 + 资产下载，不外发使用数据**：出站只做「读公开 Release 元数据 + 下载发布资产」，**不携带任何使用数据 / 制品坐标 / 遥测**；守「数据不外发」红线（与 ADR-0012 漏洞库「只取公开数据、不外发坐标」同基调）。
- **仅 Admin 手动触发**：不做无人值守定时自动替换（避免坏版本无人值守上线）；不开放匿名/普通用户。
- **完整性校验强制**：sha256 不一致即拒绝替换、删临时文件、保留旧二进制、进程续以旧版运行。仅做 sha256 完整性校验，**不做签名验签**（签名需另起基建 + 新 ADR）。
- **原子替换（跨平台）**：仅在校验通过后替换、且替换是最后一步。Unix `rename` 同卷原子覆盖（运行中进程保旧 inode），替换前留 `{exe}.bak` 单步回退兜底；Windows 先把运行中 `.exe` 改名 `{exe}.old` 再落新文件、下次启动清理 `.old`，失败尽力还原。
- **自动重启经 graceful-shutdown 排空后拉起**：替换成功 → 置进程级重启请求 + 触发优雅停机 → axum drain 在途请求（含应答）→ `serve` 返回后在 `main` 拉起新进程（`restart_mode=self`）或直接退出交外部进程管理器（`restart_mode=exit`，适配 systemd/docker，避免与管理器重启叠加成双实例）。先排空再拉起以规避端口争用。
- **可选凭据**：公开仓库免凭据；私有仓库经可选 token，真源 env `JIANARTIFACT_UPDATE_TOKEN`，不入库、不进日志、不回显。
- **平台边界**：仅 FR-86 三发布目标（linux-x86_64 / windows-x86_64 / macos-arm64）可自更新；其余平台明确报错、不静默乱下。

## 理由

- **出站默认关闭 + 仅拉取公开数据**：把自更新这一主动出站行为收敛为运维显式开启的 opt-in，且只读公开 Release、不外发任何本机数据，与「数据不外发 / 不 phone-home」红线一致。
- **仅 Admin 手动**：自替换生产二进制风险高，手动 + 仅 Admin 是把控制权与审计点留给运维；定时无人值守留待后续按需另议。
- **sha256 强校验、校验后才替换**：保证「下载损坏 / 中间人篡改字节」不会落地为可执行；校验失败永不触碰二进制，满足「保留旧二进制」验收。
- **graceful-shutdown 先于拉起**：排空在途请求并释放监听端口后再起新进程，规避新旧进程端口争用，是跨平台最稳的重启序列。
- **restart_mode 双模**：`self` 适配裸进程部署（自拉起）、`exit` 适配进程管理器部署（退出即被重启），避免「自拉起 + 管理器重启」叠加成双实例的footgun。
- **不引新依赖**：版本比较 / target 推导 / JSON 解析 / 哈希用现有 `serde_json` / 标准库 / `sha2`，守单一二进制零原生依赖与简单优先。
- **复用 ADR-0020 出站 helper**：自更新出站经 `build_outbound_client` 注入 `[network.proxy]`，受限网络下与其余出站点一致经代理。

## 后果

- 正面：运维可一键升级；与既有「替换二进制」运维模型衔接（`exit` 模式即「自动停、由管理器替你重启」）；出站可控、默认关。
- 负面 / 约束：
  - 自替换运行中二进制 + 自重启的**端到端正确性依赖真机**（尤其 Windows 运行中 `.exe` 改名、重启端口序列），单元测试只能覆盖纯逻辑与替换规划，端到端须 Linux + Windows 各真机验。
  - 仅 sha256 完整性、无来源签名：信任链止于「GitHub Release + HTTPS + sha256」；更强的来源真实性（签名）需另写 ADR。
  - 出站代理 / 仓库源 / token 在启动期装载，运行时不热替换（与其余出站客户端生命周期一致）。
  - 自更新只认 FR-86 命名契约的资产；契约变更须 FR-85 / FR-86 同步。
- OPERATIONS 升级章节补「在线自更新」路径；ARCHITECTURE 增 `update` 模块与重启机制；API.md 增两端点。

## 备选方案

- **暂存、下次启动生效（不自重启）**：更稳但不满足用户明确要求的「自动重启」；落选（但 `restart_mode=exit` 在有进程管理器时近似此体验）。
- **无人值守定时自动更新**：坏版本可能自动上线、无审计点，风险高；落选，留待后续按需另议。
- **签名验签 / SBOM / 更新通道（stable/beta）**：当前无明确需求、需额外基建，属镀金；按需再走新 ADR。
- **引入 `self_update` / `self-replace` 等第三方 crate**：会引入新依赖与其传递依赖（部分拖入 native-tls / 额外 TLS 栈），与零原生依赖、rustls-only、简单优先相悖；自实现替换/重启仅用标准库 + 现有依赖，落选第三方 crate。
