# ADR-0023：主机 / 系统监控采集（经 sysinfo，仅 Admin、本机不外发）

## 状态

已接受（「不后台轮询、不落库、不留历史时序」一条已被 ADR-0027 取代；其余决策——经 sysinfo 采集、仅 Admin、本机不外发、按请求实时快照端点——仍有效）

## 背景

运维需要在控制台看到运行制品库的**这台主机**当前的基础资源状况（CPU 使用率、内存与磁盘的已用 / 总量、进程已运行多久），用于容量与健康判断（对应 PRD 的 FR-98，P2）。这与既有可观测能力互补但不重叠：

- ADR-0015 的 **Prometheus 指标端点**面向运维抓取**应用自身**的运行指标（请求量、错误率、延迟、缓存命中等），是 pull 模型的应用级埋点；它不采集**宿主机**的 CPU / 内存 / 磁盘资源画像。
- ADR-0015 的**审计日志**与 ADR-0009 的**使用分析**分别面向安全取证与运营统计，均与主机资源无关。

需在以下约束内确定方案：

- 隐私红线（沿用 ADR-0009 / ADR-0015 基调）：主机指标是本机内部运行数据，**默认不主动外发、不向外部遥测平台 phone-home**；任何外部导出须运维显式开启、默认关闭。
- 架构不变量：单一二进制、零外部运行时依赖；`api` 层保持轻薄、handler 不写业务；分层依赖单向无环；高开销 IO 置于锁外。
- 简单优先：本期只需「按请求采样 + 单个查询端点」，不做后台持续轮询、不留历史时序、不引外部 agent。
- 跨平台：需在 Linux / Windows / macOS 上统一读取 CPU / 内存 / 磁盘 / uptime。

## 决策

在 P2 新增**主机 / 系统监控采集**能力，作为 ADR-0015 可观测性主题的扩展（不取代它，二者数据去向一致：本地、默认不外发）：

### 1. 经 `sysinfo` 跨平台采集基础主机指标

- 新增第三方依赖 **`sysinfo`**（已按治理 §15 经用户确认）：钉版本、`default-features = false`，仅启用 `system`（CPU / 内存 / 交换 / uptime）与 `disk`（磁盘）两个 feature，剔除 `component`（温度传感器）/ `network`（网卡）/ `user`（账户）以控二进制体积与依赖面。
- 采集范围：CPU 全局使用率百分比 + 逻辑核数；内存与交换分区的总量 / 已用（字节）；每块磁盘的挂载点 / 总量 / 可用（字节）及汇总；系统 uptime（秒）。均为本机非敏感运行画像，不含任何凭据 / 个人数据。

### 2. 按请求采样，不做后台持续轮询

- 采样在请求到达时进行：单进程共享一份 `sysinfo::System`（`refresh` 需 `&mut`，经 `Mutex` 串行化），handler 在锁内 `refresh` + 读数后立即组装 DTO 返回。磁盘列表按请求新建刷新。
- **CPU 使用率首样取舍**：sysinfo 的 CPU 使用率需两次采样间隔才有非零值；本期单次 refresh，首次 / 间隔过近的采样 CPU 使用率可能为 `0`（合法值），不为它引入后台轮询。该取舍在 spec 与本 ADR 显式记录。
- 不留历史时序、不落库：主机指标是「当下快照」，按请求实时读即可，避免又引一套时序存储与轮转。

### 3. 仅 Admin、本机内部不外发

- 经 `api` 层一个**薄 handler** 暴露 `GET /api/v1/monitor/host`，复用现有认证中间件 + 全局角色判定：未认证 401、非管理员 403、Admin 200。
- 采集逻辑下沉到独立 `monitor` 模块；「sysinfo 读数 → DTO」的映射尽量做成无副作用纯函数，便于穷举单测。handler 不写采集 / 汇总逻辑。
- 纯本地采样查询，**不接任何外部上报 / 导出 / phone-home**；GET 读取类不入审计（与 FR-97「读取类不留审计」一致）。

外部对接（如把主机指标接入外部监控平台）一律不在本期实现，未来若做须默认关闭、运维显式开启（与 ADR-0009 / ADR-0015 一致）。

## 理由

- **引 `sysinfo` 而非自实现**：跨平台读取 CPU / 内存 / 磁盘 / uptime 在三大 OS 上各有专有系统调用（Linux `/proc`、Windows WinAPI、macOS Mach / IOKit），自实现成本高、易错且需大量平台分支与 `unsafe`；`sysinfo` 是成熟的纯抽象库，裁掉非必需 feature 后体积可控，且其内部 `unsafe` 不影响本仓库的 `#![forbid(unsafe_code)]`（仅约束本仓库代码）。
- **按请求采样、不轮询**：满足「当下资源画像」诉求的最简实现；无后台任务、无时序存储，零额外常驻开销，符合简单优先。
- **仅 Admin + 不外发**：主机资源画像属管理视图，默认仅管理员可见；纯本地查询守住「数据不外发」红线。
- **扩展而非复用指标端点**：`/metrics` 是应用级 pull 埋点、低基数标签纪律严格，把宿主机资源塞进去会混淆受众与语义；独立端点 + 独立 DTO 更清晰，且控制台可直接展示结构化 JSON。

## 后果

- 正面：开箱即得本机资源画像（CPU / 内存 / 磁盘 / uptime），跨平台统一、隐私可控、无新增常驻任务。
- 负面 / 约束：
  - 新增 `sysinfo` 依赖（已按 §15 经用户确认）；`Cargo.lock` 随之变更。
  - 新增 `monitor` 模块与 `GET /api/v1/monitor/host` 端点，须在 ARCHITECTURE（模块 + 机制）、API.md（端点 + 鉴权）、CHANGELOG 同步说明。
  - CPU 使用率首样可能为 0 是已知取舍，需在 API.md / spec 提示调用方。
  - 采样在 `Mutex` 锁内做（refresh + 读数，纯内存 + 系统调用、无网络 / blob IO），开销小；不得在该锁内做重 IO（守「锁外做 IO」）。

## 备选方案

- **自实现跨平台系统信息读取**：需逐平台写专有系统调用与大量 `unsafe`，维护成本高、易错。落选——引成熟 `sysinfo` 更简单可靠。
- **复用 ADR-0015 的 `/metrics` 端点承载主机指标**：受众与语义不同（应用埋点 vs 宿主机资源），且会破坏低基数标签纪律。落选——独立端点独立建模。
- **后台持续轮询 + 落库历史时序**：超出本期诉求（只需当下快照），引入常驻任务与时序存储与轮转，违背简单优先。落选——按请求采样即可，未来确有历史需求再走新 ADR。
- **不做主机监控**：不满足 FR-98 的运维容量 / 健康可视化诉求。落选。
