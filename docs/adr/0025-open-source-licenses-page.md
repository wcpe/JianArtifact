# ADR-0025：开源许可清单构建期扫描 + 数据嵌入二进制 + 公开页

## 状态

已接受

## 背景

本产品把众多 Rust crate 与前端 npm 包打进单一二进制分发。多数开源许可（MIT / Apache-2.0 / BSD 等）要求**分发时附带归因（作者）与许可证信息**，当前控制台无任何开源许可展示，存在合规缺口（FR-102，P2）。

约束与驱动因素：

- 运行形态是**单一二进制 + 零外部运行时依赖**（ADR-0001）：许可数据须随二进制一起分发，不能依赖运行时外部服务或联网拉取。
- 数据不外发红线（ADR-0009 基调）：许可归因是公开信息，但本产品**不主动外发、不向外部平台 phone-home**——许可数据的采集必须在本地 / 构建期完成，运行时只读不外联。
- 依赖清单须覆盖 Rust + 前端两套生态、且区分运行时与开发依赖（PRD FR-102）。
- 简单优先（架构不变量 §2）：不为此引入运行时新依赖、不引前端图表库。

## 决策

许可清单在**构建期由独立工具自动扫描生成**，产物**嵌入二进制**，运行时经一个**公开（匿名可读）端点 `GET /api/v1/licenses`** 与公开页 `/licenses` 只读展示；扫描与生成绝不在运行时发生、不外发。

- **Rust 侧**：引入 `cargo-about` 作为**构建期 CLI 工具**（非项目运行时依赖，不进 `Cargo.toml`），按仓库内 `about.toml` 的 accepted 许可清单扫描运行时（normal + build）crate 的名 / 版本 / 作者 / 许可；开发依赖（dev-only crate）经 `cargo metadata` 全图与运行时集合的差集补齐。
- **前端侧**：用 pnpm 内置 `pnpm licenses list --json`（`--prod` 区分运行时 / 开发）提取，不引入新的 license 提取依赖。
- **合并与嵌入**：生成脚本 `scripts/gen-licenses.mjs`（Node）把两套生态合并为一份结构化 JSON（`{name, version, license, author, kind, source}` 列表 + 汇总），写入 `src/licenses/data.generated.json`；后端 `licenses` 薄模块经 `include_str!` 在编译期嵌入。仓库内提交一份 `generated=false` 的占位 JSON（空清单），生成脚本覆盖之，使 `include_str!` 恒可编译（与 `frontend/dist/.gitkeep` 同属「保证可编译的占位」）。
- **本地优雅降级**：本地开发未跑生成脚本时，端点 / 页面据占位的 `generated=false` 返回空清单 + 「未生成」标记，不阻断编译 / 启动、不崩溃。CI 在编译二进制**前**跑生成脚本（`release.yml`），正式产物含真实清单。

## 理由

- **构建期生成 + 嵌入**完美契合单一二进制 / 零运行时外部依赖：许可数据随二进制走，运行时不联网、不读外部服务，天然守「数据不外发」。
- **cargo-about** 是 Rust 生态成熟的许可扫描工具，能按 accepted 清单做合规门禁（引入未授权许可时生成报错）；作为构建期 CLI 不污染运行时依赖树、不增二进制体积与攻击面。
- **pnpm 内置 `licenses list`** 免引新前端依赖即可提取归因，`--prod` 一刀切出运行时 / 开发，简单可靠。
- **占位 + 降级**让缺数据不阻断本地编译与启动，符合「缺数据优雅降级」与可独立编译的工程纪律。
- 公开端点：开源归因本应公开可查，匿名可读符合合规透明诉求，且不涉及任何私有 / 敏感数据。

## 后果

- 正面：合规归因可在控制台公开自查；许可数据本地生成嵌入、运行时零外联，守隐私红线；不增运行时依赖与体积。
- 负面 / 约束：
  - 多一个**构建期前置步骤**（生成脚本），CI 须在编译前跑、且依赖 `cargo-about` 可安装；本地开发者若要看真实清单需手动跑一次。
  - `about.toml` 的 accepted 许可清单需随依赖演进维护：引入带新许可的依赖时生成会报错，须显式补 accepted（这正是合规门禁的预期行为）。
  - Rust 运行时 / 开发切分依据 cargo-about 默认依赖图（排除 dev-dependencies）与 cargo metadata 全图的差集；dev-only crate 的许可取自 metadata 的 `license` 字段（SPDX 表达式原样透传，不做本地许可证文件确认）。
  - 占位 `data.generated.json` 入库——它是「保证可编译的占位」而非生成产物语义，与 `.gitkeep` 同类，不违反「生成产物不入库」。

## 备选方案

- **运行时扫描 / 联网拉取许可**：违反单一二进制 / 零运行时依赖与数据不外发红线。落选。
- **手工维护许可清单**：易腐化、随依赖漂移失准，维护成本高。落选。
- **把许可证全文打进二进制并展示**：体积膨胀且超出 FR-102 范围（仅需归因元数据：名 / 版本 / 许可证 / 作者）。落选。
- **引入前端专用 license 提取依赖（如 license-checker）**：pnpm 内置能力已够，避免镀金式新依赖。落选。
