# ADR-0026：自更新回滚（增强 ADR-0021）

## 状态

已接受

## 背景

在线更新（FR-85，ADR-0021）已支持管理员一键升级：查 GitHub 最新 Release → 下载 → 校验 sha256 → 原子替换二进制 → graceful-shutdown 后自动重启。但 ADR-0021 只保证「升级前留单步回退兜底」（Unix `{exe}.bak`、Windows 运行中 `.exe` 改名 `{exe}.old`），**没有暴露给运维的回滚操作**：升级后若新版有问题，运维只能手工换回旧二进制。

且现有两条备份路径都不适合作「持久回滚源」：

- Windows `{exe}.old` 是「下次启动清理」的临时档——`cleanup_stale_old` 在 main 早期就会删它，留不到运维想回滚的时刻。
- Unix `{exe}.bak` 虽未被清理，但 ADR-0021 把它定义为「单步回退兜底」，并非稳定的、跨平台一致的回滚契约，跨平台行为不对称（Windows 侧根本没有等价持久档）。

FR-104 要在 FR-85 之上提供管理员一键回滚到上一版本。需经 ADR 裁定的新行为：① 引入一条**持久、跨平台一致**的回滚源备份路径（与 ADR-0021 的临时备份语义并存、不冲突）；② 新增一个会替换并重启运行中二进制的端点（回滚方向，与 apply 同样高风险）。

约束沿用 ADR-0021：rustls-only、单一二进制零原生依赖、`#![forbid(unsafe_code)]`；不引新第三方 crate；分层 `api → update → config`，`update` 不碰 `meta`/DB；简单优先、不预留空壳（单备份，不做多版本历史）。

## 决策

在 `update` 模块与 `api::update` 薄 handler 上增强自更新，新增**仅 Admin** 端点 `POST /api/v1/update/rollback`，并约定单一持久回滚备份：

1. **单备份语义**：只保留**上一版本**一个回滚源，不做多版本历史、不支持回滚到任意指定版本。每次升级覆盖该单一备份。
2. **持久回滚备份路径**：在 exe 同目录（同卷以保 rename 原子）落 `{exe}.rollback.bak`，作为跨平台一致的回滚源。`apply_update` 在原子替换前，把当前运行的二进制复制为该路径（落盘失败即报错回滚、不触碰二进制）。该路径**独立于** ADR-0021 的 Unix `.bak` / Windows `.old`，且**不被启动清理**（`cleanup_stale_old` 不动它）——ADR-0021 的临时备份语义保持不变。
3. **回滚 = 还原上一版 + 重启**：`POST /update/rollback` 校验 `{exe}.rollback.bak` 存在 → 把备份原子换回当前二进制（复用 ADR-0021 的 `execute_replace` 跨平台原子替换：Unix `rename` 覆盖、Windows 先把运行中 exe 改名 `.old` 再落回）→ 置位重启请求，按 `restart_mode`（`self`/`exit`）走 ADR-0021 既有重启链路。无备份返 `409`「无可回滚的备份版本」，不静默。回滚原子、失败尽力回退、不留半截。
4. **与升级共用单飞互斥**：回滚复用 apply 单飞 guard（ADR-0021 M2），同一时刻只允许一个二进制变更在途（升级或回滚），避免并发互踩 `.new`/`.old`/备份中间态；在途时再触发返回 `409`「更新进行中」。
5. **回滚可用性暴露**：设置聚合视图（FR-87）的 `update` 增 `rollback_available: bool`（备份文件是否存在），供控制台启用 / 禁用回滚按钮；不回显任何路径细节。

## 理由

- **持久独立备份**：把回滚源与 ADR-0021 的临时兜底档解耦，避免「Windows `.old` 被启动清理后无可回滚」的缺口，且跨平台一致（两平台都有 `{exe}.rollback.bak`），回滚逻辑无需按平台分叉备份来源。
- **单备份、不做多版本**：守简单优先与 YAGNI——回滚的真实诉求是「坏版本退回上一个能跑的」，多版本历史 / 任意版本回滚需额外存储与 UI，属镀金，按需再走新 ADR。
- **复用 execute_replace 与重启链路**：回滚本质是「再做一次原子替换，只是新内容是旧二进制」，复用 ADR-0021 已验证的跨平台原子替换与 graceful-shutdown 重启序列，不另造一套替换 / 重启机制。
- **共用单飞 guard**：升级与回滚都改同一个运行中二进制，逻辑上互斥；复用既有 guard 比新增第二把锁更简单，也杜绝「升级与回滚并发」这一最危险的竞态。
- **不引新依赖**：复制 / 替换 / 路径推导用标准库与现有 `update` 机制，守零原生依赖、rustls-only、单一二进制。

## 后果

- 正面：运维升级后发现问题可一键回滚到上一版；跨平台一致、持久可用；与 ADR-0021 升级链路同构、复用其原子替换与重启序列。
- 负面 / 约束：
  - 与升级同样，**自替换运行中二进制 + 自重启的端到端正确性依赖真机**（尤其 Windows 运行中 `.exe` 改名、重启端口序列），单元测试只覆盖纯逻辑与回滚 / 替换规划；端到端真机回滚须 Linux + Windows 各真机验。
  - 仅单备份：连续两次升级后只能回滚到「最近一次升级前」的版本，更早版本不可回滚（符合单备份语义，非缺陷）。
  - 备份占额外一份二进制磁盘空间（与运行中二进制同量级）。
  - ARCHITECTURE 自更新章节补「回滚」路径；API.md 增 rollback 端点与 settings 视图 `rollback_available` 字段；CHANGELOG 记本次新增。
- 对 ADR-0021 的关系：**增强**——ADR-0021 的升级流程、临时备份（`.bak`/`.old`）与重启机制**保持不变**；本 ADR 仅在其上增加持久回滚备份与回滚端点，不推翻任一既有决策。

## 备选方案

- **直接复用 Windows `.old` / Unix `.bak` 作回滚源**：`.old` 被启动清理留不住、跨平台不对称，无法作稳定回滚契约；落选。
- **多版本回滚历史 / 回滚到任意版本**：需额外存储与版本索引、UI 更复杂，当前无明确需求，属镀金；落选，按需另走新 ADR。
- **自动回滚（健康探测失败自动回退）**：需健康判定 + 重启编排，复杂度高、误判风险大，超出 FR-104「管理员手动一键回滚」范围；落选。
