# ADR-0027：统一指标时序采集与查询（后台采样落 SQLite + 保留期滚动 + 查询 API，取代 ADR-0023「不留时序」）

## 状态

已接受（取代 ADR-0023「不后台轮询、不落库、不留历史时序」一条；ADR-0023 其余决策——经 sysinfo 采集、仅 Admin、本机不外发——仍有效）

## 背景

ADR-0023 为 FR-98 主机监控定下「**按请求采样，不做后台持续轮询、不留历史时序、不落库**」，理由是当时诉求只是
「当下资源画像」，最简实现即可。监控页重设计（FR-99）改变了诉求：控制台需要**多指标时序网格**——展示一段时间内
各域指标（主机 / 存储仓库 / 防护 / 使用分析等）的变化曲线、悬停看某时间点取值。当下快照无法回答「过去一小时
CPU 怎么变」这类问题，需要历史时序。

需在以下约束内确定方案：

- 隐私红线（沿用 ADR-0009 / ADR-0015 / ADR-0023 基调）：指标是本机内部运行数据，**默认不主动外发、不向外部
  遥测平台 phone-home**；本期不做任何外部导出。
- 架构不变量：单一二进制、零外部运行时依赖；`meta` 是元数据唯一 DB 入口，不绕过它直连 SQLite；高开销 IO 置于锁外；
  `api` 层薄、handler 不写业务。
- 简单优先 + 量级可控：引入时序存储必然增加 DB 写入与存量，须用采样间隔 + 保留期 + 行数兜底把量级压住，不撑爆 SQLite。
- 与 FR-98 实时快照的关系：实时快照仍有其用（要「此刻」精确值时按请求采），不应被时序取代。

ADR-0023 的「不留时序」是这次决策要推翻的点；按「ADR 不可变、只取代」治理，写本 ADR 取代该条，不改 ADR-0023 正文。

## 决策

在 P2 新增**统一指标时序采集与查询**能力（对应 FR-105），取代 ADR-0023「不留时序」决策：

### 1. 统一时序存储（经 meta 落库）

- 新增一张**通用扁平时序表** `metric_samples(id, metric_key TEXT, ts INTEGER, value REAL)`（`ts` 为 Unix 毫秒 UTC），
  配 `(metric_key, ts)` 复合索引。一行 = 一个指标键在一个时刻的标量取值。
- 经 `meta` 模块（新增 `metrics.rs`，与 `usage.rs` 同属元数据访问层）读写，**不绕过 meta 直连 SQLite**；
  SQLite 仍是该数据的唯一真源。
- 本期只做「单指标键 → 标量值」的扁平时序，**不做高基数标签 / per-repo 多维**（避免量级失控），未来确需再走新 ADR。

### 2. 后台定期采样 + 保留期滚动清理

- 后台 `tokio` 定时任务按**可配采样间隔**（`sample_interval_secs`，默认 60s）每拍采样一组 gauge、同一 ts 批量落库
  （沿用 audit / usage 的有界批量落库范式）；采集失败只 WARN、不影响业务。**间隔走配置、不硬编码**。
- 另一后台任务按**可配保留期**（`retention_days`，默认 7 天）周期删除更早样本，并设**行数硬上限**（`max_rows`，默认
  1_000_000）兜底（沿用 `spawn_audit_retention` 范式）。
- **采哪些（本期做能低成本拿到的）**：
  - 主机 CPU% / 内存% / 磁盘%（复用 ADR-0023 的 sysinfo 采集逻辑 `monitor::collect`，新增「字节 → 百分比」纯函数，改为后台采样落库）。
  - 存储 / 仓库：仓库总数、blob（去重 sha256）总数、存储总字节（经 meta 新增轻量 `COUNT` / `SUM` 查询）。
  - 防护：当前活跃封禁 IP 数、限流累计被拒数（复用既有进程内计数器 getter）。
  - 使用分析：累计访问量 / 下载量（复用既有 `usage_stats` 聚合查询）。
- **缓存命中率本期降级不采**：proxy 模块当前无 hit/miss 计数器（仅 Prometheus 占位键未实现），新埋点代价显著且属另一条
  改造线；按 YAGNI 本期不为它撑大范围，待后续单独埋点后再纳入。

### 3. 查询 API（仅 Admin）+ 保留 FR-98 实时快照

- 新增 `GET /api/v1/monitor/metrics?metric=<key>&from=<ms>&to=<ms>&step=<ms>`（属监控 Admin 域，未认证 401、非管理员 403）：
  经 `meta` 取原始样本 → **降采样 / 聚合纯函数** `downsample` 按 step 分桶聚合 → 返回时序点。handler 薄，聚合逻辑下沉纯函数可穷举测。
- **FR-98 实时快照端点 `GET /api/v1/monitor/host` 保留不变**：要「此刻」精确值时仍按请求实时采。时序是其**补充**而非替代——
  二者数据去向一致（本机内部、不外发），各司其职（实时点查 vs 历史曲线）。

### 4. 本机内部、不外发

- 时序数据落本地 SQLite，**绝不主动外发、不向外部遥测 phone-home**（守 ADR-0009 / 0015 / 0023 一贯红线）；本期不做任何
  外部导出 / 上报开关。未来若做外部对接须默认关闭、运维显式开启。

## 理由

- **通用扁平表 vs 每指标一表**：一张 `(metric_key, ts, value)` 表即可承载所有标量 gauge，新增指标只是多一个键、无需改 schema；
  扁平结构查询简单、量级好估。比按域建多表更简单。
- **后台采样 vs 复用按请求快照**：时序需要均匀时间网格的历史点，只有后台定时采样能产生；按请求快照不落库、点位随机，无法成曲线。
- **采样间隔 + 保留期 + 行数兜底**：三道闸把量级压住——10 个键 × 60s 间隔 × 7 天 ≈ 每键约 1 万、合计约 10 万行，`max_rows` 兜底，
  SQLite 完全可承受。
- **复用既有数据源、缓存命中率降级**：主机 / 存储 / 防护 / 使用分析的读数都已有低成本来源，直接复用；唯独缓存命中率需新埋一套
  热路径计数，代价不对称，本期降级不采（守简单优先 + YAGNI），不为单个指标撑大范围。
- **保留实时 + 新增时序**：两类诉求（此刻精确点查 vs 历史趋势）语义不同，分别由 FR-98 端点与本时序端点承载，互不取代、各自清晰。
- **经 meta 不绕过 + 锁外 IO**：守架构不变量——DB 只经 meta，主机 refresh 在其自身锁内（纯内存 + 系统调用），DB 写在 meta、不在持锁期间做重 IO。

## 后果

- 正面：控制台获得跨域历史时序能力（支撑 FR-99 监控页时序网格）；量级可控、隐私可控、无新增外部依赖；复用既有采集 / 计数，改动收敛。
- 负面 / 约束：
  - 新增常驻后台任务（采样 + 清理）与时序存储，带来少量周期 DB 写入与存量（已用间隔 / 保留期 / 行数兜底压住）。
  - 新增 `metric_samples` 表与迁移、`meta/metrics.rs`、采样 / 清理任务、查询端点，须在 ARCHITECTURE（模块 + 机制）、API.md（端点）、
    CHANGELOG 同步。
  - 缓存命中率本期缺数据源（监控页该项暂缺），属已知降级，待后续埋点纳入。
  - counter 类指标（限流被拒 / 访问下载）存为累计值，曲线为单调累计；需要每段增量时由查询侧 / 前端差分（采样侧保持无状态）。

## 备选方案

- **维持 ADR-0023「不留时序」、监控页只展示当下快照**：无法满足 FR-99 的历史趋势诉求。落选——这正是本 ADR 要推翻的点。
- **按域建多张时序表（host_samples / storage_samples …）**：schema 随指标膨胀、新增指标要改 DDL，复杂度高于收益。落选——通用扁平表更简单。
- **引入外部时序数据库（Prometheus TSDB / InfluxDB 等）**：违背「单一二进制、零外部运行时依赖、不引重型件」架构不变量。落选——嵌入式 SQLite 足够。
- **把时序塞进 `/metrics`（ADR-0015 Prometheus 端点）让外部抓取存历史**：那是 pull 给外部的应用埋点、且会把存储 / 受众外移，与「本机内部、
  控制台自给」诉求不符，也触碰数据外发面。落选——内部时序自建、独立查询端点。
- **采样侧对 counter 做差分存增量**：采样任务需维护上次值状态、重启 / 漏采会失真。落选——存累计值、差分留查询 / 前端，采样无状态更稳。
