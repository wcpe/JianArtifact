# ADR-0028：动态配置持久化（文件默认 + DB 覆盖 + 内存缓存）

## 状态

已接受　·　扩展 ADR-0022、ADR-0018；不触动 ADR-0001 技术栈

## 背景

当前配置真源是 **TOML 文件 + 环境变量**，启动期一次性装载到 `Config`；少数节（出站代理 / 在线更新 ADR-0022、防护 ADR-0008/FR-79）建了**进程内热替换槽**支持运行时 PATCH 即时生效，但**不落库**——重启即回落文件 / env。痛点：运维拿到单二进制后，想在面板上改限流阈值 / 防护策略 / 采样间隔等，改完**重启就丢**，只能去编辑 `config.toml`。

用户诉求："除数据目录外的配置都能在面板在线配置并持久化"，按现代 Web 应用的**黄金组合**：文件兜底（保证起得来）+ DB 覆盖（运行时动态）+ 缓存加速。

约束（红线，不可破）：
- **凭据绝不入库明文**（架构不变量 §3、ADR-0018）：密码 / Token / 代理账密 / 上游凭据 / OIDC·LDAP 密钥 / JWT 密钥的真源是文件 + env，DB 只存引用。
- **依赖方向单向无环**（架构不变量 §1）：`api → … → meta → config`，`config` 是最底层，**不得反向依赖 `meta` / DB**。
- `meta` 是元数据唯一 DB 入口，其他模块不绕过它读写 SQLite。
- 使用 / 配置数据是本机内部数据，不外发。

## 决策

引入**配置覆盖层**，三段式（黄金组合），严格隔离凭据、不破依赖方向：

### 1. 持久化落在 `meta`（不让 config 依赖 DB）
- `meta` 新增 `app_settings(key TEXT PRIMARY KEY, value_json TEXT NOT NULL, updated_at INTEGER)` 表（键为 `section.field` 点分路径，值为 JSON 标量 / 片段）。`meta` 暴露 `load_settings() / upsert_setting() / delete_setting()`。
- **`config` 模块保持纯文件 / env、零 DB 依赖**。覆盖发生在**应用装配层**（`main`）与既有**进程内热替换槽**：启动序列 `加载文件配置（默认） → 打开 meta → 从 app_settings 读覆盖 → 合并到生效快照 → 填充热替换槽`。config 不反向依赖 meta，依赖方向不变。

### 2. 三段式生效（黄金组合）
- **启动（文件兜底）**：照旧 `Config::load`（默认 → TOML → env）得到**默认值**，保证 DB 为空 / 不可用时应用仍能起来。
- **运行时（DB 覆盖）**：meta 就绪后读 `app_settings`，把已持久化的覆盖项 merge 到生效配置之上，填入热替换槽。
- **变更（写 DB + 换槽）**：面板 PATCH 校验通过 → 写 `app_settings`（meta）→ 重建 / 原子换内存槽即时生效。**DB 是动态配置的持久真源**；内存槽是其缓存（读路径只读槽、不每请求查库）。

### 3. 凭据严格隔离（守红线）
- **只有"非密钥"配置项进 `app_settings`**。**含凭据的配置项不入库**——继续走"文件 / env 真源 + 进程内内存槽"（即 ADR-0022 现状不变）：
  - 出站代理（`[network.proxy]`，URL 可内嵌 `user:pass@`）、在线更新 `token`、`[auth.oidc] client_secret`、`[auth.ldap] bind 口令`、JWT 密钥 —— **一律不写 `app_settings`**。
- PATCH 落库前经**白名单 + 脱敏过滤**：凭据字段只更新内存槽、绝不写 DB；非密钥字段才 upsert 到 `app_settings`。

### 4. 三类配置项的归属（边界）
| 类别 | 例 | 真源 / 可在线改 |
|---|---|---|
| **Bootstrap（启动必需）** | `server.bind/port`、`data.data_dir`、DB 路径、`data.storage` 后端 | **文件 / env only**，不入库、不在线改（先有鸡先有蛋：DB 配置不能存在 DB 里） |
| **Secret（含凭据）** | 代理账密、update token、OIDC/LDAP 密钥、JWT 密钥 | **文件 / env 真源 + 内存槽**（ADR-0022 不变），**不入库** |
| **Dynamic（非密钥动态项）** | 限流 / 封禁 / 慢速 / CC / WAF / 告警阈值、上传上限、审计保留期、使用分析开关、指标采样间隔 / 保留期、在线更新非密钥字段、漏洞库开关 / 源 / 周期、会话 TTL / 登录锁定阈值 | **DB 覆盖（app_settings）+ 文件兜底 + 内存槽缓存**，**可在线改、持久化** |

> 具体纳入哪些 Dynamic 节，spec §2 列清；**首批可只覆盖高频节**（限流 / 防护 / 上传上限 / 采样 / 在线更新非密钥），其余按需增量纳入（YAGNI），不一次铺满全部节。

## 理由

- **落 meta、不让 config 依赖 DB**：守依赖方向红线；config 仍是可被任何层依赖的纯叶子。覆盖在装配层做，符合现有 `main` 编排职责。
- **文件兜底**：DB 空 / 损坏 / 迁移失败时仍能用文件默认起来，韧性优先；不引入"配置必须先写库才能跑"的脆弱性。
- **复用既有热替换槽作缓存**：读路径零额外 DB 开销，与 ADR-0022/0008 的"槽即当前真源"一致；只是多了"启动从 DB 预填 + PATCH 同时写库"。
- **凭据隔离**：在"配置可在线持久化"与"凭据绝不入库"间取真正的边界——非密钥落库、凭据永不落库，红线不破。
- **键值 + JSON 而非每节建表**：配置项稀疏、节多且会增量纳入，KV 表零 schema 漂移、加新项不迁移；强类型校验在反序列化 + 既有各节 `validate()` 完成。

## 后果

- 正面：非密钥配置面板可改、重启不丢；运维不再被迫编辑 TOML；凭据 / bootstrap 边界清晰。
- 负面 / 约束：多一层"文件 默认 ⊕ DB 覆盖"心智，须明确**优先级**（env > DB > 文件默认？还是 DB > env > 文件？见下）；`app_settings` 为弱类型 KV，靠反序列化 + `validate()` 兜类型；含凭据节仍不持久化（重启回落文件 / env，与 ADR-0022 同）。
- **优先级待定（spec 须定）**：建议 **文件/env = 默认基线，DB 覆盖之；但 env 显式给值时 env 最高**（部署环境强约束 > 面板改动），即 `生效 = env 显式 ⊕ DB 覆盖 ⊕ 文件默认`。spec §3 钉死并测试。

## 备选方案

- **config 直接读写 DB**：最直观，但 `config` 反向依赖 `meta` / DB，破依赖方向红线、且启动 DB 未就绪时无配置可用，落选。
- **全部配置（含凭据）加密入库**：需引入主密钥管理 + 加解密层，复杂度与泄露面陡增，与"凭据不入库"基调冲突，落选（用户已选"凭据仍走文件/env"）。
- **每节一张强类型表**：节多、会增量纳入，每加项就迁移、schema 漂移大，落选（选 KV + JSON）。
- **维持现状（仅内存槽不落库）**：解决不了"重启即丢"的核心诉求，落选。
