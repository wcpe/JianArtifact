# ADR-0029：运行时日志文件 sink + 读取 API

## 状态

已接受　·　扩展 ADR-0015 可观测性；不触动 ADR-0001 技术栈、不改 ADR-0002 真源

## 背景

应用运行时技术日志走 `tracing`，当前 `init_tracing` 只把日志格式化输出到 **stdout**。痛点：

- 进程重启 / stdout 未被外部捕获时，**早期与历史运行日志即丢失**，排障无据。
- 运维要看运行日志只能登录主机翻 stdout / 进程管理器，**无法在控制台查看**。

诉求："应用运行日志也写入数据目录下的日志文件 + 提供读取 API + 控制台「系统日志」页"（FR-107）。

与**审计日志**（ADR-0015 / FR-31 / FR-77）的边界必须清楚：审计是**业务安全事件留痕**（写 / 管理 / 授权拒绝），经 `meta` 异步落 **SQLite**、脱敏、有保留期；本项是**运行时技术日志**（tracing 的 ERROR/WARN/INFO/DEBUG），载体是**文件**、不落库、不计安全事件。两者读者与用途不同，互不替代。

约束（红线，不可破）：

- **不引重型件 / 优先不引新依赖**（架构不变量 §2、§4）：用既有 `tracing-subscriber` 实现，不为日志滚动强引 `tracing-appender` 等新 crate。
- **SQLite 是元数据唯一真源**（架构不变量 §3）：运行日志不写 DB，不与元数据混为一谈。
- `#![forbid(unsafe_code)]`、中文分级日志、凭据不进日志、数据不外发（既有基调，不变）。

## 决策

给运行日志**增设一个文件 sink**，并提供**仅 Admin** 的只读 API；载体是文件、不落库。

### 1. 文件 sink（保留 stdout，追加文件层）

`init_tracing` 在保留 stdout 输出的同时，追加一个文件输出层，把运行日志写到 `{data_dir}/logs/app.log`；级别过滤沿用既有 `RUST_LOG` / 默认 `info`。

**时序问题**：`init_tracing` 在 `main` 第一行调用，但 `data_dir` 要等配置加载 + CLI `--data-dir` 覆盖后才确定。解法用 **tracing-subscriber 的 `reload` 层**（`std` 默认特性，**不引新依赖、不开新特性**）：

- `init_tracing()` 用 `registry()` 装配：EnvFilter + stdout fmt 层 + 一个**可重载**的 `Option<文件 fmt 层>`（初始 `None`），返回 reload **句柄**。
- `main` 拿到 `data_dir` 后创建 `{data_dir}/logs/`、构造文件 writer，`句柄.reload(Some(文件层))` 换入——此后日志同时进 stdout 与文件。data_dir 确定前的少量配置加载日志只进 stdout（已知取舍，不为此硬编码默认路径）。

### 2. 简单滚动（单文件 + 单次大小滚动，std 实现）

文件 writer 自实现轻量 `Write`（`RollingFileWriter`）：追加写 `app.log`，当文件超过大小上限时滚动一次（`app.log` → `app.log.1`，旧 `.1` 被覆盖），用 `std::fs` 即可。**不引 `tracing-appender`**——单文件 + 单次大小滚动够 P2 用。不做按日期归档 / 压缩 / 保留期清理（YAGNI，后续若需更强滚动再走新 ADR）。写与滚动判断在写路径内经 `Mutex` 串行化（本地小 IO，符合既有写串行化基调）。

### 3. 读取 API（仅 Admin，薄 handler，纯函数解析）

- `GET /api/v1/system-logs`，**仅 Admin**（未认证 401 / 非管理员 403 / Admin 200）：读 `{data_dir}/logs/app.log` 全部行（受滚动上限约束的小文件）→ 解析 → 可选级别**精确**过滤（`level=ERROR|WARN|INFO|DEBUG`）→ **tail**（最新在前）+ `offset`/`limit` 分页（默认 200 / 上限 1000）→ 统一分页响应 `{ items, total, offset, limit, has_more }`（对齐 API.md §1 与 audit 端点）。
- 文件不存在 / 为空 → 返回空列表、HTTP 200，不报错。GET 读取类**不入审计**（符合 FR-97）。
- "日志行 → 结构化条目"的解析（取时间戳 / 级别 / 消息）、级别解析、tail / 过滤 / 切片做成 `logs` 模块的**纯函数**，无 IO、可穷举单测；读文件 IO 留 handler 侧薄封装。无法识别级别的行不丢、归无级别、原文进 message。

## 理由

- **复用 tracing-subscriber 的 reload**：在"早期日志不丢"与"文件路径须等 data_dir"间取干净解，零新依赖、零新特性，守"优先不引依赖"。
- **载体文件、不落库**：运行日志高频、体量大、是技术流水，写 SQLite 会污染元数据真源、撑大库、与审计混淆；文件 + 大小滚动是日志的惯常载体，简单且与既有 stdout 输出同源。
- **自实现单文件滚动**：单文件 + 单次大小滚动用 std 几十行即可，引 `tracing-appender` 是为不需要的按时间归档付复杂度，不划算。
- **纯函数解析 + 薄 handler**：解析 / tail / 过滤无副作用、可穷举测；handler 只做鉴权 + 读文件 + 调纯函数 + 封装，守"api 薄"与"锁外 / 纯函数"基调。
- **与审计严格分线**：明确"审计=业务留痕落库 / 系统日志=技术日志落文件"，避免两套日志互相侵蚀职责。

## 后果

- 正面：运行日志持久可查、控制台可视；排障不再依赖外部捕获 stdout；与审计边界清晰。
- 负面 / 约束：
  - data_dir 确定前的早期日志只进 stdout、不进文件（取舍）。
  - 滚动简单（单文件 + 1 个 `.1`），不归档 / 不压缩 / 无保留期；高写入量下历史窗口有限——P2 可接受，更强滚动后续走新 ADR。
  - 读取端点读全文件再切片（非流式倒读），靠滚动上限约束文件大小；超大日志场景非本期目标。
  - 多一个文件句柄与写锁（本地小 IO，串行化开销可忽略）。

## 备选方案

- **引 `tracing-appender` 做按日滚动**：得到成熟的时间 / 大小滚动与非阻塞写，但为不需要的归档能力引新依赖，与"优先不引依赖 / 简单优先"冲突，落选。
- **运行日志落 SQLite（复用审计基建）**：可统一查询，但污染元数据真源、撑大库、与审计语义混淆、高频写压力大，破"DB 仅存元数据"基调，落选。
- **不落盘、仅靠外部捕获 stdout**：零改动，但解决不了"重启即丢 / 控制台不可见"的核心诉求，落选。
- **deferred 全量延后 init 到 data_dir 已知**：免 reload，但会丢失配置加载阶段的早期日志（连 stdout 也丢），不如 reload 干净，落选。
