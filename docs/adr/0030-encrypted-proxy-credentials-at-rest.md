# ADR-0030：网络代理凭据加密落库持久化

## 状态

已接受　·　扩展 ADR-0028（动态配置入库）、ADR-0024 / ADR-0022（代理）、细化 ADR-0018（凭据真源）；不触动 ADR-0001 纯 Rust 零原生依赖

## 背景

网络代理（`[network.proxy]`）经网页 PATCH 配置后只入**进程内热替换槽**（ADR-0022），**不落库**；FR-106（ADR-0028）把其它动态配置落库持久化，却把代理整体作为"含凭据项"排除在外（守"凭据不入库"红线）。后果（真机暴露）：**网页配的代理重启即丢**——尤其在线更新自更新会**重启**应用，重启后代理没了、出站全断，自我拆台。用户要求代理（含密码）**持久化、重启不丢**。

约束（红线，不可破）：
- **凭据绝不写 DB 明文**（架构不变量 §3、ADR-0018）：密码 / token / 密钥的真源在文件 + env，DB 不存明文凭据。
- 纯 Rust 零原生依赖（ADR-0001）：加密用 RustCrypto 生态（与既有 `argon2` / `sha2` / `jsonwebtoken(rust_crypto)` 一致），不引 ring 作直接加密 API。
- `config` 不反向依赖 DB（ADR-0028）；DB 只经 `meta`。

## 决策

把网络代理**持久化到 `app_settings`**，**非密钥部分明文、密码加密落库**——"明文密码绝不入库"红线不破：

1. **拆分持久化**：每代理（http/https/all）落库为 `{ url（脱敏 host，无凭据）, username（明文，标识非密钥）, password_enc（密文，可空）}`；`no_proxy` 明文。URL 与用户名属非密钥（沿用 ADR-0024「用户名是标识、非密钥」），明文落库；**密码仅以密文落库**。
2. **加密落库（AEAD）**：密码用 **XChaCha20-Poly1305**（RustCrypto `chacha20poly1305` crate，纯 Rust）加密——每次加密用 **192 位随机 nonce**（`rand`），存 `nonce ‖ ciphertext`（base64）。AEAD 提供机密性 + 完整性（防篡改 / 截断）。
3. **主密钥真源在文件**：加密子密钥经既有 `JwtSigner::derive_key(b"proxy-credential-encryption-v1")`（`HMAC-SHA256(jwt_secret, domain)` → 32 字节）派生——真源是数据目录的 `.jwt_secret` 文件（ADR-0011），**密钥绝不入库**，与会话 JWT 域分隔、互不串味。
4. **落库与恢复**：PATCH 改代理 → 更新内存槽（即时生效，ADR-0022 不变）**同时**把上述拆分形态写 `app_settings`（密码加密）；启动时从 `app_settings` 读回 + 用文件密钥解密密码 → 重建代理 URL 填入内存槽。代理（含密码）**完全持久、重启不丢**。
5. **边界**：`network.proxy` 经**专用持久化路径**处理（不并入 FR-106 的 `DYNAMIC_KEYS` 明文白名单——那条默认拒绝一切凭据键，代理密码走独立加密通道）。`config` 仍不依赖 DB：解密/恢复在装配层（`main`）+ `meta`，config 只读文件默认。

## 理由

- **明文密码绝不入库**：DB 里密码是**密文**（非明文），解密密钥真源在 `.jwt_secret` 文件——与 ADR-0018「绝不写 DB 明文、真源在文件」一致，红线是"明文"，密文 + 文件密钥不破。
- **复用既有密钥派生**：`derive_key` 已被 CC 挑战用于域分隔子密钥，复用同一真源、同一模式，不新增密钥文件、不泄露 JWT 密钥本体。
- **XChaCha20-Poly1305 而非 AES-GCM**：192 位随机 nonce 无 AES-GCM 96 位随机 nonce 的生日界顾虑，随机 nonce 用着更稳；纯 Rust RustCrypto，与项目加密生态一致。
- **拆分而非整体加密**：URL / 用户名明文落库便于网页回显（沿用脱敏口径），只加密真正的秘密（密码），最小化加密面。
- **专用路径而非白名单**：代理含凭据、加解密逻辑特殊，独立通道比塞进通用 KV 白名单更清晰、更难误把明文凭据写库。

## 后果

- 正面：代理（含密码）重启不丢；自更新重启后出站仍走代理，解决自我拆台；密码以密文落库、红线不破。
- 负面 / 约束：新增 `chacha20poly1305`（纯 Rust RustCrypto，纳入 `cargo audit`）；解密依赖 `.jwt_secret`——轮换该文件会使已落库代理密码无法解密（解密失败按"无密码"降级、WARN，不阻断启动，用户重填即可）；加密只覆盖代理密码（update token / OIDC·LDAP 密钥仍按 ADR-0028 走文件 / env，未纳入本次）。
- update token、auth 密钥等其它凭据是否同样加密落库，按需另行评估（本 ADR 只解代理持久化的当前痛点，YAGNI）。

## 备选方案

- **明文落库**：直接破 ADR-0018 红线，落选（用户要密码持久 ≠ 要明文）。
- **密码仍只走文件/env（ADR-0028 现状）**：解决不了"网页配的代理重启即丢"，用户明确要密码持久，落选。
- **用 ring::aead 加密**：项目刻意用 RustCrypto 纯 Rust、避开 ring 作直接加密 API（见 Cargo 注释），落选。
- **专设独立加密密钥文件**：多一个密钥文件与轮换面，复用 `.jwt_secret` 派生更省，落选。
