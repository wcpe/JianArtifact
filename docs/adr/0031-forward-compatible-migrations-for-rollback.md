# ADR-0031：向前兼容迁移 + 容忍未知已应用迁移（保自更新回滚安全）

## 状态

已接受　·　扩展 ADR-0002（SQLite 迁移）、ADR-0026（自更新回滚）

## 背景

自更新回滚（ADR-0026 / FR-104）只还原**二进制**、不还原 **DB schema**。真机暴露的严重 bug：

- 用户更新到新版二进制，新版应用了它新增的迁移（如 `0011_metric_samples`、`0012_app_settings`）到 DB。
- 随后**回滚到旧二进制**（如 dev.8，内嵌迁移只到 `0010`）。
- 旧二进制启动跑 `sqlx::migrate!().run()`，**默认会校验「DB 里所有已应用迁移都必须在二进制内嵌集里」**——发现 DB 有它不认识的 `0011` → 报 `migration 11 was previously applied but is missing in the resolved migrations` → **DB 打不开 → 回滚后起不来**（回滚等于把实例锁死）。

根因：sqlx 默认迁移校验对「DB 比二进制更新（有更高已应用迁移）」零容忍，而回滚天然制造这种「DB 领先二进制」的状态。

## 决策

两条，合起来让回滚跨迁移安全：

1. **迁移只增不改不删（向前兼容，硬约束）**：所有迁移只能**新增表 / 新增列 / 新增索引**，**禁止**重命名 / 删除 / 改类型 / 加破坏性约束。如此新版加的 schema 旧版**不用、不冲突**，旧二进制在新版迁移过的 DB 上仍能正常工作。确需破坏性变更 → 先写新 ADR 评估回滚影响，不静默引入。
2. **迁移器 `set_ignore_missing(true)`**：`meta::MetaStore::open` 跑迁移时设 `set_ignore_missing(true)`——旧二进制遇到 DB 里它**不认识的更高已应用迁移**时**忽略、不报错**，只跑自己内嵌集里尚未应用的待应用项。配合约束 1，旧二进制能在「被新版增量迁移过」的 DB 上正常打开、运行。

## 理由

- **回滚只换二进制不改 DB（沿用 ADR-0026）**：不维护 down 迁移（破坏性、易错）；靠「向前兼容 + 容忍未知迁移」让旧二进制直接吃新 DB，最简、最稳。
- **约束 1 是前提**：`set_ignore_missing` 只忽略未知迁移、不撤销其 schema 变更；只有迁移是增量式（旧版不依赖被删/改的东西），忽略才安全。两条缺一不可。
- **影响面小**：仅 `MetaStore::open` 两行（生产 + 测试内存库），不改迁移内容、不改回滚链路。

## 后果

- 正面：自更新回滚（含跨迁移）后旧二进制能正常打开 DB，回滚真正可用；DB schema 单向前进、不因回滚倒退。
- 负面 / 约束：迁移从此**只能增量**（破坏性变更须先走 ADR 评估回滚代价）；`set_ignore_missing` 会容忍 DB 里的"未来"迁移——这是回滚场景的预期行为，正常升级路径不受影响（待应用项照跑）。
- **遗留**：本修复只让**带本修复的二进制**容忍更新的 DB；已发布的旧二进制（如 dev.8，无本修复）仍会在更新的 DB 上失败——这类已发布版本无法追溯修复，运维恢复办法：跑任一**含相应迁移**（≥ 引入该迁移的版本）或**带本修复**的二进制即可打开 DB。

## 备选方案

- **维护 down 迁移、回滚时倒迁 DB**：down 迁移破坏性强、难维护、易丢数据，落选。
- **回滚前检查迁移边界并拒绝跨迁移回滚**：限制回滚能力、且解决不了"已发生"的锁死，落选（选向前兼容 + 容忍）。
- **保持 sqlx 默认严格校验**：回滚跨迁移必锁死，正是本 bug，落选。
