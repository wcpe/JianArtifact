# ADR-0038：统一进程内异步任务注册表（多类任务收口 + 迁移单飞 + 有界历史保留）

> 本 ADR **修订 ADR-0019**（迁移执行异步化为进程内任务）：把其「单一迁移 job 注册表」扩为「kind-agnostic 统一任务模型」，并补「迁移单飞」与「有界历史保留供找回」两项约束。**ADR-0019 旧文不改正文**，主控落地时把其状态标「已被 ADR-0038 修订」。

## 状态

已接受（修订 ADR-0019）

## 背景

ADR-0019 把在线拉取迁移执行改为**进程内有界任务注册表 + 轮询 + 客户端重连**，并明确「不落库、重启即丢、靠迁移幂等重跑恢复」（保留 ADR-0006「无须持久化迁移任务表」）。其后：

- FR-124/125 把离线预览 / 搬运并入同一 `MigrationJobs`。
- FR-126 在线更新另立一套 `UpdateJobs`（结构 `UpdateProgress`，并对「不落库」开了**有意例外**——更新替换二进制不可重跑，故终态留存到 `{data_dir}/update-state.json`）。
- 漏洞库刷新（FR-70）是 `main.rs` 的后台周期循环，无 job 概念、不在任何注册表。

结果：长耗时任务散落多套注册表 + 一条无注册的循环，运维**无统一入口**看「全部在途 / 近期任务」；迁移**无单飞**约束（理论上可并发触发多个搬运、互相踩）；离页 / 重连只能各端点各查。FR-131 要把进程内任务收口为统一模型。

ADR-0019 的进程内、不落库、有界、轮询重连的范式仍然适用且正确；缺的是「多类任务统一收口 + 单飞 + 历史保留供找回」。这是对 ADR-0019 的**扩充与修订**，而非推翻——故写新 ADR 取代其范围，不改其正文。

## 决策

引入 **kind-agnostic 统一进程内任务注册表**（`api::TaskRegistry`），把迁移 / 在线更新 / 漏洞库刷新三类长耗时任务统一登记为带 `kind` + 统一状态（running/paused/succeeded/failed/cancelled）+ 起 / 止 / 更新时间 + 可选 error 的轻量 `TaskRecord`：

- **统一收口**：三类任务进同一注册表。各 kind 的**进度明细**仍归各自专表（`MigrationJobs` 的 `OnlinePullProgress`、`UpdateJobs` 的 `UpdateProgress`）——统一表只存轻量记录，**进度单一真源仍在专表**，统一表与专表用同一 `job_id` 关联；不把进度复制成第二份真源。
- **迁移单飞**：迁移**搬运**任务（在线拉取 / 离线 proxy / 离线 hosted）同时只允许一个在途；第二个触发在「检查无在途迁移 → 登记」的同一把写锁内被拒（`409`），杜绝竞态双开。离线**预览**（枚举、非搬运）不参与单飞门。
- **有界历史保留**：注册表有界（保留近期 N 个，超出按时序淘汰），**保留已完成 / 失败任务**供「找回」（不只活跃任务）。
- **新增只读端点**（仅 Admin）：`GET /api/v1/tasks`（活跃 + 近期跨 kind 列表）、`GET /api/v1/tasks/{id}`（统一记录 + 按 kind 附进度明细）。既有 `GET /migrate/jobs(/{id})`、迁移控制端点、`GET /update/jobs(/{id})` **契约不变**（前端轮询不破）。
- **仍不跨进程重启续跑**：统一表进程内、**不落库、重启即清**——与 ADR-0019「不落库」一致。本 ADR**不**引入任务持久化；FR-126 的更新终态状态文件留存是 FR-126 自身的既有有意例外，本 ADR 不扩大其范围、不把它推广到迁移 / vuln。
- **分层守恒接入 vuln**：`vuln` 模块（低于 `api`）不得反向依赖 `api`；故在 `vuln` 内定义轻量 `RefreshObserver` 回调 trait，`spawn_refresh_loop` 增可选 observer 参数（`None` 时行为同旧、向后兼容），`main.rs` 注入由 `TaskRegistry` 支撑的 adapter 把每轮刷新登记进统一表。

## 理由

- 统一模型满足「一处看全部长任务 + 单飞 + 找回」核心诉求，且**不违背** ADR-0019 / ADR-0006「不落库」——进程内有界、重启即清，迁移幂等仍可重跑恢复。
- 守架构不变量：进程内状态、不引外部队列 / 协调组件（architecture-invariants）；SQLite 仍只是制品元数据真源；统一表不复制进度、不造第二份真源。
- 单飞在写锁临界区内原子「判定 + 登记」，避免并发触发竞态；只对搬运设门、预览放行，避免误伤。
- vuln 经 observer trait 解耦接入，守 `vuln` 不反向依赖 `api` 的分层方向。
- 既有端点契约保持不变，FR-124/125/126 与前端轮询零破坏。

## 后果

- 正面：多类长任务统一可观测（活跃 + 近期）、迁移单飞防并发踩、离页 / 重连一处找回；为 FR-132（任务中心 / 通知中心）提供统一数据源。
- 负面 / 约束：统一表与各 kind 专表为两份（轻量记录 vs 进度明细）须保持 id 一致与状态同步（在后台任务收尾处统一置终态收敛）；重启仍丢进行中任务的「正在跑」态（迁移靠幂等、更新靠状态文件 / 各自机制）；不内置任务排队 / 限流（超范围）；`spawn_refresh_loop` 多一个可选参数。

## 备选方案

- **直接合并三套进度结构为单一进度类型**：迁移与更新进度语义差异大（asset 计数 vs 阶段枚举），强并会造成上帝结构 + 大量可选字段。改为「统一轻量记录 + 各 kind 专表存进度」，职责清晰。落选。
- **持久化统一任务表到 SQLite**（重启续跑）：推翻 ADR-0019/0006「不落库」，引入第二份真源与一致性成本；迁移幂等可重跑、更新已有状态文件，收益不抵复杂度。落选。
- **vuln 循环直接持有 `TaskRegistry`**：造成 `vuln → api` 反向跨层依赖，违反架构不变量。改用 observer trait 解耦。落选。
- **统一单飞接管 apply 互斥**：apply 已有 `restart.try_begin_apply` 单飞且与系统重启 / 关闭 / 回滚共用，语义独立且更强（进程级二进制变更互斥）；统一表只记录、不接管。落选。
